# FP16 — Library shipping blockers + INI visibility (closed 2026-05-04)

## What changed

User reported four issues mid-flight on the v1.0.0 cut, all
surfaced from real-data UAT against the running app:

1. "Nothing happens when I click on a game."
2. "The search filter isn't working."
3. "How do I tell if the INIs have been downloaded?" (followed by
   "There is currently no ini folder in data" — meaning
   `refresh-inis` had never run, but the UI gave no signal of that
   state).
4. "Failed to fetch dynamically imported module: SessionsPage-…js"
   — stale `index.html` cached in the browser referenced bundle
   hashes from a prior commit.

Investigation found four real bugs, two of them from FP11
placeholders that were never wired:

- **A — Backend↔frontend GET /api/games parameter mismatch.**
  Frontend `useGames.toQueryString` sent `search`, `year_from`,
  `year_to`. Backend `api/routes/games.py:60` accepts `q`,
  `year_min`, `year_max`. Result: search and year-range filtering
  silently no-op'd in production. FP15 didn't catch this because
  no test exercises the URL-construction at the integration
  boundary.

- **B — `LibraryPage.onOpen` was a no-op stub.** Same FP11 § B6
  placeholder pattern as FP15 § A: a comment claiming
  "AlternativesDrawer wiring lands in FP11 § B6 follow-up" that
  never got filled. Clicking a game card silently did nothing.

- **C — `SettingsPage.setupInfo` prop was never passed.** The
  banner had been wired in FP11 § B3 to consume
  `useSetupCheck()` data, but `SettingsRoute` in App.tsx never
  called the hook nor threaded the prop. The banner code lived
  but never rendered. So users had no UI signal of INI presence.

- **D — Stale `index.html` cached by browser.** SPA shell from
  the FP15 commit referenced bundle hashes that no longer existed
  after rebuilds. Browser kept the old shell, hit 404s on the
  deleted hashes. Backend `_SPAStaticFiles` shipped without
  cache-control headers, so the browser had no signal to
  revalidate.

Four clusters in one commit:

- **A — useGames param names.** `search` → `q`, `year_from` →
  `year_min`, `year_to` → `year_max`. Inline comment names the
  failure mode so future contributors see the trap.

- **B — LibraryPage AlternativesDrawer wiring.** Added
  `useAlternatives` + `useOverride` hooks under
  `frontend/src/hooks/useAlternatives.ts`. LibraryPage now tracks
  `openedShortName: string | null`; clicking a card sets it; the
  drawer mounts when non-null; override mutation invalidates the
  games + alternatives queries; success toast + drawer close.

- **C — INI status surfacing.** Added `useSetupCheck()` hook
  (GET `/api/setup/check`). Wired into SettingsRoute; passed as
  `setupInfo` prop to SettingsPage. Setup banner now renders a
  second line listing the per-INI status — "Reference INIs: 4 / 4
  present." or "Missing: catver.ini, ... — run `uv run mame-curator
  refresh-inis --dest data/ini` to download." Inline runtime
  guidance closes the discoverability gap.

- **D — Backend SPA cache-control headers.**
  `_SPAStaticFiles.get_response` now sets headers post-response:
  `assets/*` get `public, max-age=31536000, immutable` (Vite's
  `[name]-[hash].js` pattern is content-addressed, safe forever);
  `index.html` and SPA-fallback paths get
  `no-cache, must-revalidate` so a fresh deploy is never masked
  by a stale shell.

Bundled with the v1.0.0 version bump (pyproject.toml + __init__.py
0.0.1 → 1.0.0; classifier "3 - Alpha" → "5 - Production/Stable"
— see `chore(release)` rationale in the parent commit body).
This same commit is tagged both `FP16-complete` and `v1.0.0`
(same-SHA, per the FP12+FP13 precedent for fix-passes that close
on the same push as a release boundary).

## What was learned

- **The "FP11 placeholder" pattern is now a known bug class.**
  FP14 (GameCard layout overflow) was a P06 surface gap caught
  by closing review. FP15 (Sessions Save no-op) and FP16 § B/C
  are different — they're FP11 placeholder comments
  (`// Wired to ... in follow-up`) that were never followed up
  on. Three instances in two days. **Lesson:** when closing a
  fix-pass with placeholder comments, grep for the pattern before
  the close commit and either complete the wiring or remove the
  comment. Adding to the workflow: a `git grep -E 'follow.up|TODO'
  src/ frontend/src/` scan as part of `/close-phase`'s pre-commit
  gate would have caught all three.

- **Frontend↔backend parameter contracts need mock-driven tests
  at the call site.** Cluster A could have been caught by a
  useGames test that asserts the URL it constructs. The existing
  test suite covers component behaviour but not the URL-building
  contract. Adding URL-shape assertions to integration-style
  tests is the cheapest forward defense.

- **SPA cache headers are a release-mechanics blocker.** v1.0.0
  shipped without them; first user with a cached shell hit a
  404'd module. The `index.html` no-cache + `assets/*`
  immutable convention is well-known but the project's
  `_SPAStaticFiles` shipped without it. Worth a one-pager in
  `docs/standards/coding-standards.md` post-v1.

## What was deferred

- **Per-game review state (P14 candidate).** User mental model
  for "sessions" was "track which games I've reviewed already"
  — that's a different feature class from the v1 Sessions
  (filter-set bookmarks). Filed as P14 post-v1. The FP15 § C
  explainer text was tightened in this commit to clarify the
  v1 feature does NOT track per-game review progress.

- **Loud-error access logging.** User noted uvicorn logs 404s as
  INFO. That's uvicorn's default access-log level — the 404 is
  a regular HTTP response, not a system error. Distinguishing
  4xx/5xx in the access log requires a custom logging filter.
  Skipped for v1; can be added post-v1 if loud error visibility
  matters.

- **CONTRIBUTING.md, screenshots, hero-shot.** P09 already
  deferred these; FP16 doesn't change that.

## False positives confirmed

- None. All four reports were real bugs. The cache issue (#4)
  is borderline — the user's symptom was self-recoverable via
  hard-refresh, but the lack of cache-control headers IS a
  bug worth fixing for the v1 release.
