## What was learned

- **e2e walkthroughs catch host-conditional unmount bugs that unit
  tests miss.** The canonical example is FP26-V. The FP25-H unit test
  rendered `<LibraryErrorPanel isFetching />` directly and asserted
  `disabled` was true. Production-correct. But the host LibraryPage's
  `{games.isError ? <Panel/> : <Grid/>}` conditional unmounted the
  panel during the very state (refetch from error) the affordance
  was meant to serve. Unit tests can't catch this without rendering
  the host AND injecting react-query state, which is rarely worth
  the test setup. Playwright drives the real production bundle and
  catches the host's conditional naturally. Pattern saved: any
  feature whose user contract is "X becomes visible while Y is
  pending" needs an e2e walkthrough exercising the host's render
  condition, not just the child component in isolation.

- **The "recursive audit returns zero" heuristic (FP10) holds at the
  static layer; semantic review still finds drift at scale.** FP25's
  closing static-tool audit (ruff/mypy/bandit/semgrep/gitleaks/eslint/
  tsc) returned 0 findings. The 4-lane semantic indie-review surfaced
  5 Tier 1. Pattern reaffirmed: scale closing review depth with
  surface area; trust the recursion-converges-at-zero claim only when
  surface area is narrow.

- **User mid-session scope-add patterns through cleanly.** The user's
  Playwright direction came AFTER FP26 was scoped from the
  /indie-review findings. Adding 5 new sub-bullets (Q/R/S/T/V — V was
  surfaced as a CONSEQUENCE of the Playwright work) into the same
  fix-pass kept the diff cohesive. Alternative would have been to
  spawn FP27 for the walkthroughs and FP28 for FP26-V — that would
  have fragmented the close path. Pattern: in-scope additions that
  share verification artifacts (the e2e spec) belong in the same
  fix-pass; spin out only when the additions touch independent
  subsystems.

- **The "Chrome Claude extension" was not loaded in this session.**
  User's first mention of "Chrome Claude extension and Playwright"
  initially confused me — I searched for an MCP browser tool and
  found none. User clarified mid-session that another Claude Code
  session (RetroDB) had successfully used Playwright; what that
  session actually used was the project's local `npx playwright
  test` runner via Bash. Same surface, no MCP required. Pattern
  saved: when the user references "Playwright", check the project's
  `frontend/node_modules/@playwright` + `playwright.config.ts` first
  before searching for a browser MCP tool.
