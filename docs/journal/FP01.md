## What shipped

### Tier 1 fixes (6/6)

- **`copy_one` signature drift.** Spec § "Atomic copy primitive"
  declared `copy_one(src, dst, progress=None)`; shipped code
  required `*, short_name, role, progress=None`. Updated spec
  to match (the metadata is needed for `CopyOutcome`
  construction; carrying it via the call is cleaner than
  re-deriving in `run_copy`).
- **`KeyboardInterrupt` mid-copy leaks `.tmp`.** `executor.py`
  caught only `OSError`; spec step 6 mandated cleanup on any
  exception. Refactored to `try/finally` so any `BaseException`
  triggers tmp cleanup before re-raising.
- **`OverwriteRecord` allocated but never appended.** Runner
  declared `overwritten: list[OverwriteRecord]` and persisted
  it on `CopyReport.overwritten` but never wrote to it in the
  REPLACE / REPLACE_AND_RECYCLE branches. Fixed: append one
  record per replaced winner, identifying the displaced short
  name from the existing playlist.
- **`PlaylistError` not raised on missing append decision.**
  Spec mandated raise; code silently defaulted to KEEP_EXISTING.
  Resolved by changing the design (not the code): the runner
  doesn't carry `cloneof_map` and so cannot detect
  "conflicting winner" — that detection is caller-side. Spec
  updated to make presence-in-`append_decisions` the conflict
  signal; runner trusts the caller. CLI/API will pre-detect
  conflicts using their `cloneof_map`.
- **`recycle_file` collision logic broken.** The
  `while ... and any(... for _ in [None])` was a 1-iteration
  generator that never advanced; same-second same-name recycle
  would clobber. Rewrote with a clean counter loop:
  `while (target_dir / path.name).exists(): counter += 1; ...`.
- **`read_lpl` no legacy 6-line format support.** Spec
  promised tolerance; code only handled v1.5+ JSON. Narrowed
  spec to "v1.5+ JSON only — legacy is post-v1." Pragmatic
  given Pleasuredome users are typically on modern RetroArch.

## What was learned

- **`/audit` and `/indie-review` are complementary, not
  redundant.** `/audit` (ruff + mypy + bandit + pytest-cov +
  grep) catches grammar — what a tool can mechanically detect.
  `/indie-review` catches semantics — what a fresh human read
  spots: "spec says X, code does Y," "this allocates a list and
  never writes to it," "this catches OSError but not
  KeyboardInterrupt." Both are necessary; neither is
  sufficient.
- **Round-2 indie-review on patched code is worth running.**
  After FP01 round-1 closed all 6 Tier-1, round-2 returned 5
  fresh Tier-2 — fresh-eyes findings on the surrounding code
  that round-1 didn't surface (it was busy verifying round-1
  fixes). One was a real user-data risk (B-T2-3). Convention:
  always run at least round-2 for high-stakes fix-passes.
- **The convergence-checkpoint matters more than the fix
  count.** App-Build's threshold is "5 fix-passes in a row
  before pause." We did 2 (DOC01, FP01) — well below. But
  pragmatically, deferring 15 round-2 items to FP02 was the
  right call: P03 was already large; piling more in would
  have made the close uninspectable. FP02 captures them; the
  loop closes when each round returns clean.
- **Caller-side conflict detection is the right architecture.**
  The original spec asked the runner to verify a property
  (cross-version conflict) it couldn't compute (no
  `cloneof_map`). The fix wasn't to push `cloneof_map` down to
  the runner — it was to acknowledge that conflict detection
  belongs at the layer that already has the data, and have
  the runner trust the caller. Looser coupling, simpler API.
