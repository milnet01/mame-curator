## What was learned

- **Audit-agent claims need spot-verification before spec authoring.**
  The 5-lane audit flagged 51 sub-fixes; pre-spec verification (global
  rule 13) caught 2 fabricated/mistyped citations during spec drafting
  (T1.4 filename) and a further 4 during implementation: T2.18 (CI
  matrix is multi-OS, not Linux-only — agent was wrong), T1.15
  (legacy-mtime path still live, agent wanted to migrate), T1.13b
  (test_fp09_fixes.py "second TestClient" was a misread — test
  doesn't use the fixture client), T2.6 (claimed `config` fixture
  inlined 30× — actually it's a single module-level definition with 30
  reference uses). Cumulative: ~10% of the audit's findings turned
  out to be wrong on closer inspection. The pre-spec grep pass IS
  worth the time.
- **Closing review on test-cleanup phases finds drift, not bugs.**
  Cluster R1's three findings were all comment-drift (stale
  cross-references, wrong row-pitch number, imprecise HTML
  rationale). No semantic test bugs. The closing-review value still
  positive — comment-drift compounds across releases — but the value
  curve is shallower for test-only sweeps than for production-code
  fix-passes.
- **Helper hoisting alone doesn't move file caps.** Three over-cap
  test files (`SettingsPage.test.tsx` 686, `test_runner.py` 528,
  `test_dat.py` 453) all stay over cap post-DS04 even with the
  Tier 2 cohort applied. The hoist-vs-split decision matters: hoisting
  saves ~20-40 LoC per file via DRY; structural splits along seams
  are needed to drop below the 500-line hard cap. T2.1–T2.4 + T2.7
  (helper hoist + YAML extract) were deferred for this reason — they
  trade complexity for marginal LoC reduction without solving the
  cap problem.
- **vitest `globals: true` auto-cleanup makes most `afterEach(cleanup)`
  blocks redundant.** RTL auto-cleanup is enabled when vitest globals
  are on — 11 redundant `afterEach(() => cleanup())` calls were
  removed in T3.1 with zero test failures. The same is likely true
  for other React+vitest projects; worth checking before adding
  cleanup boilerplate to new tests.
- **`@pytest.mark.parametrize` with explicit `ids=` paramset names is
  the cheapest deduplication win.** Three sites (T1.5 iterparse,
  T1.13 retroarch, T2.12 retroarch) collapsed 6+ near-identical tests
  to 3 parametrized blocks. Pattern: when tests differ ONLY in a
  named-argument permutation, collapse to `it.each` / `parametrize`
  with explicit IDs. Reads cleaner, fewer LoC, semantically equivalent.
