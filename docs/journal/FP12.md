# FP12 — Settings page list editors + path picker (closed 2026-05-04)

## What changed

P06's spec § "Out of scope" line 511 declared "all Settings-page
controls are built here," but FP11 only closed the ~11 binary
toggles + theme/layout pickers. Every list-shaped or free-text
field still needed hand-editing of `config.yaml`. FP12 closed
that gap with two reusable UI primitives (`<ChipListEditor>`,
`<DragReorderList>`) plus the long-deferred `<FsBrowser>` path
picker (R29-R34 backend already shipped in P04). Ten clusters
A-J shipped across ~13 commits between 2026-05-02 and
2026-05-04:

- **A — `<ChipListEditor>`** primitive used by 7 fields
  (`drop_categories` / `drop_genres` / `drop_publishers` /
  `drop_developers` + `preferred_genres` / `preferred_publishers`
  / `preferred_developers`). Free-text add via Enter; chip click
  removes; paste-comma-separated for bulk import.
- **B — `<DragReorderList>`** for `region_priority`. User elected
  simpler arrow-button + ArrowUp/Down keyboard reorder over
  dnd-kit pointer-drag (Karpathy "push back on complexity"
  override).
- **C — `<YearRangeEditor>`** for `drop_year_before` /
  `drop_year_after` — paired Switch + `<Input type="number">`,
  null state via switch, min/max 1971..currentYear.
- **D — `default_sort` `<Select>`** dropdown (4 literal values)
  + new shadcn-vendored `select` primitive at
  `components/ui/select.tsx`.
- **E — `updates.channel` `<Select>`** (`stable` | `dev`).
- **F — Editable `media.cache_dir`** — `<Input>` (controlled
  draft, patches on blur) + Browse → `<FsBrowser>` directory
  mode.
- **G — `<FsBrowser>` modal path picker.** Self-contained,
  owns its own `useFs*` hooks (no pure-prop seam — single
  import surface). Quick-jump for Home + drive roots + allowed
  roots; file vs directory mode; `fs_sandboxed` 403 surfaces a
  grant prompt that POSTs R33. First MSW-backed component test
  in the suite.
- **H — Settings → Paths in-place editable.** `<PathRow>`
  helper renders 4 rows: `source_roms` / `dest_roms` /
  `source_dat` (file mode) / `retroarch_playlist` (file mode).
  DAT swap is destructive (`replace_world`) — surfaces
  `ConfirmationDialog` with concrete action label `"Swap DAT
  to <path>"` before patching.
- **I — Settings → Snapshots tab** (R16 list + R17 restore via
  concrete-label `ConfirmationDialog`; new `SettingsRoute`
  container per FP11 § B8 pattern).
- **J — Settings → Backup tab** (R18 export + R19 import via
  `ConfigExportBundle` JSON; `BackupTab` primitive +
  `useConfigExport` / `useConfigImport` hooks; export does
  Blob → `<a download>`, import does `File.text()` →
  `JSON.parse` → mutate; `ConfirmationDialog` labels the file
  by name).

## What was learned

- **"Wide phase + closing fix-pass" shape works at FP12 scale.**
  FP12 was new feature work in a fix-pass slot (filling P06's
  Settings-page gap, not closing-review of an existing phase);
  the closing audit then surfaced legitimate review findings
  for the new surface. Folding into FP13 instead of fixing
  in-place during FP12 kept the FP12 contract boundary clean
  and let FP13 absorb the messy real-world residue across 5
  thematic clusters. Same pattern as P04+FP09 / P05+FP10 /
  P06+FP11.

- **Cluster-by-cluster TDD is the right cadence for 10-bullet
  fix-passes.** FP12 shipped one commit per A-J cluster, FP13
  shipped one commit per A-E cluster (with E split into E1 +
  E2-E6 because the SettingsPage extract was substantial).
  Each commit was reviewable on its own; tests stayed green
  throughout. Compare to a single mega-commit: harder to
  review, harder to bisect if a regression surfaces later.

- **Conditionally-mounted `<FsBrowser>` keeps existing
  pure-prop tests free of MSW.** F + H both use
  `{open && <FsBrowser />}` so the underlying `useFs*` hooks
  don't fire until the user clicks Browse. Existing
  SettingsPage tests render without QueryClient + MSW handlers
  because the FsBrowser tree never mounts in the closed state.
  Pattern saved for any future "hook-owning component used
  rarely from a pure-prop parent."

- **Per-tab extraction at the §2 hard cap is mechanical.** The
  9-tab SettingsPage decomposed cleanly via the FP13 § E1 cut
  into FiltersTab / PickerTab / UpdatesTab / MediaTab + shared
  PrefSwitch + PathRow helpers. Each tab's concerns are
  independent (no cross-tab shared state); the typed-key
  updaters pass through cleanly as props. SettingsPage went
  551 → 305 lines; every existing integration test still
  passed.

  not in FP12 scope. Mitigation lives at the FP11 § H4

  requirement; resolution goes to the next debt-sweep
  (DOMPurify or markdown/HTML through a safe pipeline).

- **Per-tab unit tests** for the FP13 § E1 extracted
  components. Each new tab is covered indirectly by the
  existing SettingsPage integration tests; direct per-tab
  tests would be added if a future refactor narrows a tab's
  prop contract.
