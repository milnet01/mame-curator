# FP14 — GameCard layout overflow (closed 2026-05-04)

## What changed

User screenshot 2026-05-04 showed 26,539 games in Grouped layout
with every tile blank — no labels, no titles, no shortnames.
With 26.5k games and most without box art (the libretro-thumbnails
set covers maybe 40-60% of MAME's catalog), the library was
unusable.

Root cause: `GameCard.tsx:64` used `aspect-[3/4]` for the image
container. At ~360px grid-column width that forces 480px image
height. Combined with CardContent (~80px), the card grew to ~540px
total. Virtualizer row was 280px and Card had `overflow-hidden`,
so the bottom ~260px (including the entire `<h3>{description}</h3>`
heading) was clipped.

Two fixes:

1. **Layout fits the row.** Image area changed from `aspect-[3/4]`
   + `object-cover` to `flex-1 min-h-0` + `object-contain`. The
   image now fills whatever space remains after CardContent claims
   its natural height (`flex-shrink-0`). Card gets `h-full` to
   fill the row exactly. Works at any rowHeight (covers = 360,
   masonry / grouped = 280).

2. **Identifier always visible.** When the image fails, the
   placeholder now renders `card.description` (the actual game
   name) in the image area instead of the generic "No artwork
   available" string. Plus: shortname `<p>` (font-mono) added
   below the heading — what `mame-curator copy` consumes; also
   disambiguates same-name re-releases (1942 Capcom vs Williams).

## What was learned

- **Production UX bugs surface from real usage, not unit tests.**
  GameCard had passing unit tests for description rendering, but
  the bug was in card-vs-row sizing — only visible in the running
  app with the virtualizer math against ~26k cards. Tests were
  correct in isolation; the integration broke.

- **Closing reviews catch design-doc drift but miss layout drift.**
  FP11's 3-lane indie-review on P06 surfaced ~40 issues but no
  reviewer had a real 26.5k-card mock to render against — so the
  CSS overflow gap stayed hidden until first real-data use.
