## What was learned

- **Cold-eyes review on the spec catches the highest-leverage drift.** Five loops on `docs/specs/FP27.md` surfaced 29 findings on loop 1, converging to 0 on loop 5. The most impactful corrections happened on loops 1–3 and saved real implementation rework — the spec's first draft cited `api/activity.py:append_event` (non-existent; the real writer is at `copy/activity.py:append_activity`), invented a `dest_path` parameter for `download()` that didn't exist, and used the wrong class names (`*Event` instead of `*Details`) for `copy/types.py`'s discriminated union. Each would have been an implementation false-start. Investing in cold-eyes-on-spec is cheap (per-loop cost is bounded by document length, not test count) and dwarfs the per-loop cost of cold-eyes-on-code.
- **xfail markers + pytest-fast hook compose cleanly for tier-batched TDD.** The project's pre-commit `pytest-fast` hook runs the whole suite — RED-only commits would fail it. Adding `@pytest.mark.xfail(strict=True, reason="FP27 T<batch> not yet implemented")` to each not-yet-fixed test let each tier-batch commit ship its tests + implementation atomically while keeping the not-yet-landed batches in-tree as RED but non-blocking. Each subsequent batch drops its own xfail markers; the final state has zero xfails.
- **respx-mocked `tracemalloc` thresholds need 2.5× body-size headroom.** Pre-fix-vs-post-fix RAM distinguishability for B3 + B4 was the trickiest test-design challenge. The naïve "peak < body_size / 10" threshold flakes because respx pre-allocates the body in test scope AND httpx holds a second copy on the response object — baseline is ~2× body before the test code even runs. The 2.5× body_size threshold cleanly distinguishes pre-fix (which adds a third 1× copy via `chunks: list[bytes]` + `b"".join(...)`) from post-fix (slice-only). Pattern logged for future streaming-vs-buffer tests.
- **Closing-review value compounds.** Cluster R1's four findings are all the kind that compound — R1a (downloads.py OSError gap) would have caused real on-disk orphans in production; R1b (Firefox quick-find) would have caused a UX regression on a fast-typing user the first time they hit `/` off-route. Both were missed in initial implementation review (mine + tests) but caught by independent cold-eyes on the diff. The hour spent on the closing review paid for itself.
- **Spec → tests → implementation → closing-review is a single coherent loop.** FP27 ran the full 9-step App-Build loop end-to-end with no shortcut: Step 1 (spec, 5 cold-eyes loops), Step 2 (plan implicit in spec), Step 3 (17 RED tests with xfail markers), Step 4 (5 implementation commits), Step 5/6 (closing `/audit` already green from pre-commit + closing `/indie-review` 2 lanes), Step 7 (Cluster R1 fold-in), Step 8 (final five-gate green), Step 9 (close + tag). Total wall time: ~3 hours of automated work.
