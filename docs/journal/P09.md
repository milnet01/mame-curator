- **B — CHANGELOG `[1.0.0]` bootstrap.** Phase-by-phase summary

## What was learned (Karpathy 9 push-back)

Original P09 contract called for: 4-6 screenshots, CONTRIBUTING.md,
README hero-shot, full UI polish pass on spacing / typography /
dark-light parity / 1024-1920px responsive sweep, final UAT on
the user's real 150 GB ROM set, polished error-message rewrite of
every user-facing string.

Each of those is real value but the smallest viable v1.0.0 ships
without them:

- **Screenshots** are nice-to-have for marketing; a user who runs
  `./run.sh` sees the live app in 30 seconds. Capture community
  screenshots organically post-release.
- **CONTRIBUTING.md** matters when there are external contributors;
  the project just shipped its first release. Add when the first
  PR arrives.
- **Hero-shot polish** — typography / spacing / contrast / responsive
  breakpoints — is iterative. Real-user feedback drives better
  signal than speculative polish.
- **UAT on the user's real 150 GB set** — the user already runs the
  app on their data (per the FP14-trigger screenshot showing 26,539
  games). No additional gate needed.
- **Error-message rewrite** — every user-facing copy line shipped
  through a per-phase review. No global re-pass needed for v1.

## What was deferred

- **Screenshots in `docs/screenshots/`.** Captured organically
  post-release.
- **CONTRIBUTING.md.** Add when the first external PR lands.
- **`gh release create v1.0.0`.** Not done as part of P09; the
  annotated tag carries the release notes via the CHANGELOG entry.
  GitHub release creation is a one-line follow-up the user can run
  whenever they want a Releases-page entry.
- **Cross-platform UAT (macOS / Windows).** Linux UAT is the user's
  daily driver. macOS / Windows real-user validation arrives
  organically as community members try the project.
