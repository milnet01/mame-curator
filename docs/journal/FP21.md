## What was learned

- **Per-opponent first-decisive semantics is observable, not just
  spec-internal.** The snapshot drift on `filter_smoke.json` was wide:
  every contested group lost its over-reported tail. The frontend's
  WhyPickedPanel doesn't assert specific counts so no UI change was
  needed, but the spec clarification means a future TUI / CLI consumer
  of `Explanation.hits` won't see misleading "decisive" claims.

- **Write-first beats move-with-rollback for recycle atomicity.**
  FP25-C added a rollback envelope on the move-then-write order — sound,
  but with a worst-case orphan-file state on rollback failure. FP21-D's
  write-first ordering removes the worst case entirely: a write failure
  means the source never moved, so there's nothing to rollback. The
  vestigial `RecycleError.recycled_orphan` field is preserved for
  backward compat but stays `None` under the new ordering.

- **TOCTOU in copy_one was FAILED, not SKIPPED_MISSING_SOURCE.** The
  fix is one `try/except FileNotFoundError` block around the initial
  `src.stat()`. The wider lesson: any place that does `stat()` on a
  user-supplied path needs a TOCTOU guard, even if there's an upstream
  existence check — the SKIPPED variant exists for exactly this case.

- **`call_soon_threadsafe` is FIFO from the same thread.** This
  ruled out FP21-L (late-progress-after-terminal) as a reachable
  bug. The worker queues progress callbacks before queueing
  `_on_worker_done`; both go into the same loop's ready queue;
  Python's docs guarantee FIFO from a single producing thread. The
  `if self._current is None: return` guard in `_emit` is still
  defensive in case a future refactor introduces another dispatch
  path — pinned with a test so the guard doesn't regress.

- **Composite ARIA grid is more than `role="grid"`.** The roving
  tabindex + arrow-key handler had to live on the OUTER container
  (which receives focus when no card is focused) AND coexist with
  GameCard's own `role="button"` keyboard handler (FP24-E/Q).
  `Enter` and `o` work at both levels; the grid handler runs first
  (event bubbling on the parent), so the active-cell logic fires
  before GameCard's button activation. Future feature-conformance
  test should pin both the grid-level and card-level keyboard paths.

- **`useKeyboard` ref-based + `useLayoutEffect` for staleness.** The
  ref pattern is well-known, but the choice between `useEffect`
  (after paint) and `useLayoutEffect` (before paint) for the ref-update
  matters when a same-frame keystroke arrives after a re-render.
  `useLayoutEffect` ensures the listener sees the latest bindings for
  any keystroke arriving on the next event-loop tick.

## What was deferred

- **`tools/check_error_codes_sync.py`** referenced in `strings.ts`
  comments doesn't actually exist. The byCode no-dead-entry contract
  is informal today. Adding the tool is a debt-sweep candidate (DS-
  level) — not blocking on FP21 close.

- **AppConfigPatch's nested `*Patch` types** (PathsConfigPatch,
  ServerConfigPatch, …) per spec line 647 weren't fully materialised;
  the implementation accepts `dict[str, Any] | None` per section,
  rejects unknown top-level keys, and relies on the merged AppConfig's
  own validation. Fully typed nested patches are ~200 LOC of
  field-mirroring that buys little above the top-level whitelist +
  AppConfig roundtrip. Deferred unless the spec requires
  field-level patch validation downstream.

- **L's deeper investigation** of any non-FIFO dispatch path stayed
  at the analysis level. If a future refactor introduces a
  non-worker-thread dispatch (e.g. UI-initiated controller events that
  emit progress), the analysis should be revisited.
