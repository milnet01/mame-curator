## What changed

The user asked for review state after observing that "sessions" didn't
capture what they meant: *"I went through all games A to C."* — i.e. a
per-game "I've handled this one" mark that survives across catalogue
views and sessions, not a saved filter. P14 delivers that: every
machine carries one of four states, the default (`pending`) is stored
as **absence** in a sparse `data/state.yaml`, and the mark is global —
a game marked Reviewed from the full catalogue is still Reviewed when
seen through an "80s shooters" session.

Two architectural decisions shape the whole phase:

- **Passive field swap (INV-4).** `WorldState.review_state` is a new
  frozen field, and `replace_world(review_state=...)` swaps it
  **without** re-running the filter pipeline. Review state does **not**
  gate machine eligibility — every machine in the DAT is eligible
  regardless of whether it has been reviewed — so `world.filter_result`
  (the materialised eligible set from `run_filter()`) stays
  `is`-identical to the base across a review-state-only swap. A test
  pins that identity.
- **Per-request visibility filter, not a pipeline stage.** The
  `?review_state=` narrowing on `GET /api/games` is applied in the
  route handler *after* the cached `filter_result` slice — never inside
  `run_filter()`. The world's eligible set is cached at world-build
  cost and serves `?review_state=pending` to one request and
  `?review_state=reviewed` to the next without a rebuild. Folding the
  filter into `run_filter()` would force a full re-filter per keypress.

The two filters compose: eligibility (from `run_filter`) then
visibility (from review-state), and the per-request stage can only ever
remove games already in `filter_result`, never reintroduce one a
session filter dropped.

**Backend (chunks 1–6):**

  `load_overrides` exactly (missing file → empty model, empty/`null`

  `previous` are plain `str` (the log records the sparse-store sentinel

- **Chunk 3** `api/state.py` — `WorldState.review_state` field +

  persistence via the existing `write_yaml_atomic`; `state.yaml` is
  deliberately **not** snapshotted (recovery is `activity.jsonl` replay).

## Mid-phase decisions

- **Two planned chunks folded, not dropped.** Chunk 8 (a dedicated
  activity-renderer switch case) folded into the generic-summary path —
  `ActivityPage` already renders each event's human-readable `summary`,
  and the emitter's `"marked sf2 as reviewed"` / `"cleared sf2"` lines
  surface without a switch arm. Chunk 15 (Playwright e2e) was deferred
  to a follow-up; per-test coverage in vitest + pytest pins behaviour
  against drift. So 16 planned chunks shipped as 11 commits.
- **Sparse store, absence = pending.** Only the three non-default
  states are written to `state.yaml`; `pending` is the absence of an
  entry. This keeps the file small and makes "clear back to pending" a
  delete, not a write — and it is enforced at the wire layer so the
  invariant can't be violated by a malformed POST.
- **`session_id = ""` for review-state events.** Per-game state is a
  global annotation, not scoped to any session or copy job. No existing
  activity emitter was in production yet, so there was no convention to
  anchor against; P14 set the precedent that non-job-scoped events use
  the empty string rather than inventing a placeholder.

## Workflow lessons

- **Caching boundary is the design lever.** The "per-request filter vs
  pipeline stage" call is the whole phase in miniature: put the
  fast-mutating, per-request concern (visibility) outside the
  expensive-to-build, cached concern (eligibility). The passive-swap
  `replace_world` and the route-level `?review_state=` filter are two
  faces of the same decision — and the INV-4 identity test (`filter_result
  is base.filter_result`) is what keeps a future contributor from
  "helpfully" wiring review state into `run_filter()`.
- **Optimistic UI forces a frontend-only type.** The review-state badge
  could not go on the backend `Badge` enum without either a backend
  emitter (defeating the same-frame requirement) or failing the
  type-sync gate. The `ReviewBadgeKind` frontend-only type is the
  honest expression of "this is computed client-side from cache."
- **Parameterise the skip, don't special-case the propagation.**
  `focusNextPending(activeIndex + 1)` sidesteps the race where the
  just-marked card hasn't yet left the pending set when auto-advance
  runs. The caller, which knows which card it just marked, owns the
  skip — cheaper and more robust than having the hook reason about
  in-flight mutation state.
