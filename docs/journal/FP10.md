## What changed

5 actionable findings from P05's closing `/indie-review` (1 Tier-1, 1 Tier-2, 3 Tier-3) closed in a single fix-pass. **All five gates remain green throughout: 423 tests pass, coverage 89.12%, mypy strict / ruff / format / bandit all clean. `media/cache.py` at 100% line coverage.**

- **A1** (Tier 1) — `api/app.py:45` `httpx.AsyncClient(timeout=10.0, follow_redirects=True)` (was `timeout=10.0` only). Without `follow_redirects=True`, httpx defaults to `False`, returning a 301/302 directly to the route which surfaces it as `MediaFetchError("upstream 301 ...")` → spurious 502 to the client. Libretro CDN renames or migrations are rare today but the failure mode is visible-to-user when they happen.
- **A2** (Tier 2) — `media/cache.py:73-77` raises `MediaFetchError(f"empty body for {url!r}")` when `resp.status_code == 200 and not resp.content`. `raw.githubusercontent.com` rate-limit interstitials and CDN edge cases occasionally return `200 + Content-Length: 0`. Without the guard, a zero-byte file would land in the cache slot and every subsequent request would serve nothing forever (`path.exists()` short-circuits before any upstream re-check).
- **A3** (Tier 3) — `routes/media.py:53` `raise MediaUpstreamError(str(exc)) from exc` (was `f"upstream error: {exc!r}"`). The pre-fix path produced a user-facing `detail` like `upstream error: MediaFetchError("upstream 500 for 'https://...'")` — class name leaks into the wire body and the prefix is redundant with the inner cause. Chained `__cause__` continues to carry the typed exception for logs.
- **A4** (Tier 3) — `media/cache.py:60-62` one-line invariant comment naming the append-only-via-`atomic_write_bytes` assumption. The `path.exists()` → eventual `read_bytes()` (in the route) race is benign today because nothing in `media/` deletes; a future "verify checksum" or "delete corrupt entry" path would reintroduce TOCTOU. The comment makes the load-bearing fact visible to future readers.
- **A5** (Tier 3) — `media/cache.py:68` `MediaFetchError(f"network error for {url!r}: {exc}") from exc` (dropped second `!r`). The chained `__cause__` already carries the typed exception (e.g. `httpx.ConnectTimeout("connect timeout")`); repeating the class name in the user-facing message was noise. The first `!r` (URL) stays — FP08-era policy mandates repr-quoting user-controlled paths/URLs in warning emits.

## What was learned

- **One-round fix-pass converge — fastest yet.** FP10's own closing `/audit` (ruff / bandit / mypy / gitleaks-full / trivy / semgrep p/security-audit + p/python + p/fastapi 200 rules) returned 0 findings; `/indie-review --fix HEAD` returned **Accept** with no Cluster-R items. Compare: FP02 closed with R1-R4 inline drift, FP05 / FP06 / FP07 / FP08 / FP09 each closed with 1-5 Cluster-R items. FP10's small surface (113-line patch across 6 files, all narrowly scoped to the 5 numbered findings) didn't generate patch-introduced drift the way larger fix-passes do. Pattern: **smaller fix-passes converge faster** because the patch surface is too narrow to introduce new contract-level drift.
- **`/indie-review --fix HEAD` is the right shape for narrow fix-passes.** The full-sweep `/indie-review` is calibrated for ≥5 commits across ≥3 subsystems; FP10's 5 findings on 4 files don't earn that cost. Dispatching one focused subagent against the FP10 commit diff with the ROADMAP entry as the contract takes ~60s vs the multi-lane sweep's ~5 min, and produces tighter signal because the agent's mandate is "judge whether each numbered finding is closed" rather than "scan everything for blind spots." Saved as the standard fix-pass close shape.
- **Spec-as-contract continues to pay rent.** A1 and A2 modified the contract (`follow_redirects=True` is now part of the lifespan-client invariant; empty-body responses are now treated as fetch errors). Updating `docs/specs/P05.md` inline kept the spec faithful — without that, future readers comparing spec-vs-code would catch a "drift" that was actually intentional. The "specs are for features, not fixes" rule (no `docs/specs/FP10.md`) plus the "fix-pass amends parent spec when contract-touching" pattern is the right combination: fix-passes don't get their own contract document, but they update the parent's where the fix tightens or extends it.
- **The FP02 / DS01 / FP05 / FP06 / FP08 / FP09 closing-review-of-closing-review precedent held.** Even though FP10's closing review found nothing, running it was the load-bearing step that proved the patches don't introduce drift. Skipping it on the basis of "this is a small fix-pass" would have lost that signal. Keep doing it.
