across the FP12 surface; 3-lane `/indie-review` returned the

FP11 cadence:

## What was learned

- **Fix-pass on a fix-pass-of-indie-review converges on local
  gates.** FP13 closed without re-running the multi-agent
  `/indie-review` — the ROADMAP fold-in pattern (each finding
  becomes a tracked cluster sub-bullet → tested → closed)
  combined with the 8 local gates (ruff / format / mypy / bandit
  / eslint / build / vitest / pytest) provided the audit signal.
  This is the FP11 "user-elected close-on-CI-green" precedent
  applied at the local-gate level, since the project doesn't
  run CI on every push (no PR workflow).

- **Per-tab extraction is the right shape for a 500+-line page
  component.** SettingsPage's 9 tabs each had distinct concerns
  (chip-list editing, drag reorder, year-range pair, dropdowns,
  destructive DAT swap, snapshot list, backup, etc). Each tab
  extraction was a clean cut at a natural seam — no shared
  state across tabs, only the typed-key updaters which pass
  through cleanly as props. Total: 6 new files, 246 lines
  removed from SettingsPage, every existing test still passed
  unchanged. Pattern saved for any future page-level component
  approaching the §2 hard cap.

- **The `apiErrorToast` helper extracted at exactly 3 call
  sites.** Not the first or second mutation — the third was
  the trigger per the project's Rule of Three. Result: a
  small, focused helper (~25 lines) used in three places, each
  with a one-line `onError: toastApiError` wiring. This is the
  inverse-of-premature-DRY pattern saved as a project precedent.

- **FsBrowser's `code in error` duck-typing is the right
  pattern for vitest module-identity drift.** Cluster C kept
  the existing duck-type check — alternative was instanceof
  on ApiError, which fails when vitest re-evaluates the api/client
  module across test files. Kept the comment justifying the
  pattern.

## What was deferred

- **`HelpPage.tsx:67 dangerouslySetInnerHTML`** — out of FP12
  surface (P06/FP11 territory), deferred to the next
  debt-sweep. The mitigation is the existing FP11 § H4
  `FIXME(security)` comment in HelpPage that names the
  trust-boundary requirement for upstream-sourced help; the
  debt-sweep should resolve it via either DOMPurify or
  rendering markdown/HTML through a safe pipeline.

- **Per-tab tests for the new extracted components.** Each new
  tab component (FiltersTab / PickerTab / UpdatesTab /
  MediaTab) is currently covered indirectly by the existing
  SettingsPage integration tests. Direct per-tab unit tests
  weren't added; the integration coverage held. If a future
  refactor narrows a tab's prop contract, that's the moment
  to add direct tests at that boundary.

- **DragReorderList runtime-uniqueness guard.** Cluster E4 went
  with documented-precondition over runtime check — the only
  consumer (region_priority) has server-validated config that
  enforces uniqueness upstream. If a future caller bypasses the
  contract, React's key-collision warning will surface; that's
  acceptable signal for a precondition violation.

## False positives confirmed

- None. Every actionable finding was a real bug, regression,
  zombie code path, or coding-standards violation. The one
  semgrep hit (`HelpPage.tsx:67 dangerouslySetInnerHTML`) was
  real but out of FP12 scope; it goes to known-issues / next
  debt-sweep, not the allowlist.
