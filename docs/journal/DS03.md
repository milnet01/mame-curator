## What was learned

### 1. `>=X` constraints with no upper bound silently accept major-version bumps

The pyproject `mypy >=1.13` constraint was meant to defer mypy 2.x to a follow-up phase. Cluster F's `uv lock --upgrade` blew through that intent because nothing forbade the major. Two lessons:

- For a dep where the project genuinely wants to gate a major bump, the constraint needs an upper bound (`>=1.13,<2`). A docstring comment is not load-bearing — uv resolves against numbers, not narrative.
- Conversely, "non-breaking only" sweeps that rely on test-suite-as-arbiter can accept opportunistic majors when the tests genuinely pass clean. The spec was amended in R1 to record this carve-out rule explicitly.

The mypy 2.0 default-shifts (`--strict-bytes` PEP 688, `--local-partial-types`) are now documented inline in `pyproject.toml [tool.mypy]` so future contributors know which strictness changes the `strict = true` config inherits.

### 2. CI gates don't cover frontend until you wire them

Step 1 inventory surfaced that `ci.yml` had no `actions/setup-node` step at all — the frontend was gated locally only. Every prior release shipped trusting per-developer pre-commit. Cluster G closed this gap; user-2026-05-16 explicitly redirected the spec to fold the new lane in rather than defer.

The new job reads `node-version-file: frontend/package.json` so a future `engines.node` bump propagates to CI automatically — no second config drift surface to forget about. Same pattern Cluster D adopted for the cross-pin coupling test.

### 3. Pre-commit hook `additional_dependencies` is a third drift surface

The cross-pin coupling test covers `rev:` lines against `uv.lock`, but `additional_dependencies:` constraint floors under the `mirrors-mypy` hook also drift independently and weren't caught by the test. R1b folded the drifted floors back into pyproject lockstep. A follow-up DS or FP should either (a) extend `test_dep_pin_coupling.py` to assert hook `additional_dependencies` ≥ pyproject floors, or (b) remove the floor strings entirely and rely on `pre-commit autoupdate` to resolve. Logged as a follow-up below.

### 4. Spec-text drift is real even between adjacent phases

Cluster H corrected `pnpm` → `npm` references in DS02 + DS05 specs. The references slipped past two prior cold-eyes spec reviews because they were inside acceptance-criteria bullets, not architecture text. Two takeaways:

- Verification grep patterns (Cluster H acceptance) need to be narrow enough that the spec documenting the rule doesn't self-trip. R1g amended the recipe to exclude DS03.md itself.
- Source-of-truth references (P06.md:122 "npm not pnpm/yarn/bun") need to be respected explicitly — narrowing the verb-pattern grep preserved the prose mention.
