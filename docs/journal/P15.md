## What changed

A user-feedback-driven redesign of the Library page: opening the
v1.2.0 build showed `21,049 games · 0.0 GB · 0 BIOS deps` in the
bottom bar with no clear path to "select games I want and copy
them to my console". P15 replaces the dead bottom-bar with a
cart-first selection model and surfaces the parent/clone-collapse
state explicitly when listxml is missing or empty.

Three new backend models, five new backend tests, ~22 frontend
files net (4 new components, 3 new hooks, 6 surgical edits, 14
test files, 1 e2e spec). Picker fix shipped early in FP23
(2026-05-07) once the cold-eyes review caught the misdiagnosis
that the picker was unwired (it was wired correctly; the user's
`config.yaml` had `paths.listxml: null` which let the runtime
collapse fail silently).

## What was learned

- **Cold-eyes spec review at Step 1 caught a misdiagnosis that
  would have shipped a no-op fix.** P15's spec round 1 had me
  asserting "the picker isn't wired" — wrong. The reviewer
  cross-checked `state.py` / `runner.py` / `picker.py` and
  pointed at `cloneof_map={}` from a missing `paths.listxml` as
  the actual cause. That reframe spawned FP23 (which shipped
  the fix immediately rather than waiting for P15 implementation
  to start), kept P15's scope honest (no spec-mandated picker
  rewrite), and left a clean regression test as the durable
  artefact. **Pattern reaffirmed: the cold-eyes round catches
  bias-driven misreads, not just typos.** Saved as a feedback
  memory under `feedback_cold_eyes_spec_review_at_step_1.md`.
- **Mid-implementation scope-reset honesty is cheaper than
  over-promising.** During FP23, initial sign-off was "wire
  onCopy/onDryRun to the filter result"; on inspection, full
  Copy needed SSE + conflict resolution (~500 lines + tests)
  which is genuinely P15-scale. Surfaced the revised three-
  option scope question to the user mid-session; they picked
  the middle option (DryRun lands now, Copy ships with P15).
  Saves both author and user from a mid-implementation
  discovery cascade.
- **CartBar's GB figure is the kind of thing only an
  independent reviewer catches.** I rendered `total_bytes` from
  the filtered library on the cart bar — the prop was named
  `totalBytes`, the data was there, the wiring "looked right".
  Indie-review Lane 8 caught it: the cart's bytes are not the
  filtered library's bytes; passing the filtered figure is
  actively misleading. Spec § 4.4 explicitly said "per-cart GB
  is v1-deferred" — I read past it. **The takeaway is not
  "read the spec more carefully" but "the cold-eyes round on
  the implementation is the load-bearing audit signal for
  invisible-on-first-glance UX bugs."**
- **Tier-3 debt is worth folding in the same fix-pass.** FP24
  closed all three tiers in 13 commits: Tier 1 (user-visible
  blockers), Tier 2 (hardening), Tier 3 (debt). The Tier 3
  surface was 8 frontend + 5 backend findings — a separate
  DS## sweep would have added a context-load + dispatch + close
  cycle for findings already enumerated in the indie-review.
  Same author + same week + same context = one fix-pass.
  Pattern: only split into a dedicated DS## when the indie-
  review pre-dates the implementation by enough commits that
  the author has lost context.
- **Per-finding TDD pays off in fix-passes too.** Every cluster
  (or pair of clusters) had a failing test landing first;
  `pytest -xvs` + `vitest run` confirmed red, then green, then
  committed. No surprise regressions across 13 commits.
- **The picker fix's regression test belongs at two levels.**
  B1's runner-level test
  (`test_cloneof_map_empty_self_parents`) is the load-bearing
  invariant; the API-level test
  (`test_cloneof_map_collapses_winners`) is a smoke that the
  invariant survives the runner→API plumbing. The skip on the
  API-level no-listxml regression is intentional — that case
  is covered transitively by the runner-level test, and
  duplicating it at the API layer would just couple the test
  to the route's response shape without buying additional
  coverage.

## What was deferred

- **Per-cart GB total.** Spec § 4.4 marks this v1-deferred:
  `GameCard` has no `roms`/`size` field (it's a thin display
  projection). FP24-B confirmed by dropping the misleading
  display. Surfacing it would require either adding a
  `byteEstimate` field to `GameCard` (per-row response model
  inflation) or a per-cart sum endpoint. Re-evaluate post-v1
  when usage shows whether users actually want the figure.
- **Cart sync (multi-device).** Spec § 7.1 — opt-in cloud
  sync of the in-progress copy cart, server-side state. Not in
  v1 budget.
- **Featured-tile editor in Settings.** Tiles are code-defined
  in `strings.ts` for v1; growing the catalogue is a one-PR-
  per-tile chore. A Settings-tab editor is a v1.X enhancement.
- **Mobile / responsive layout pass.** Cart-bar + sidebar +
  featured-tiles row need separate breakpoints before phones /
  tablets work. Desktop-first today; § 7.1 mobile non-goal.
- **Cart UI polish — drag-to-reorder, cart export/import,
  in-cart-panel search.** Spec § 7.1 forward-look.
- **Game-selection wizard** — spec § 7.1; rejected for v1 but
  kept as a Help-page-launchable surface for new users post-v1.
- **Five additional E2E cases** (FP24-KK partial). Vitest unit
  coverage pins the contracts; the finding explicitly accepted
  Vitest-only when an equivalent assertion exists.
