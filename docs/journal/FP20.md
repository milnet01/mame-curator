## What was learned

- **DOMPurify v3's _isValidAttribute filter trumps ADD_ATTR.** Setting `ADD_ATTR: ['target']` alone is insufficient ‚Äî the per-attribute validator strips `target` AFTER the `uponSanitizeAttribute` hook runs. Only `data.forceKeepAttr = true` survives that pass. This was discovered during FP20-L by inspecting `node_modules/dompurify/dist/purify.cjs.js:1206` after the test failed with target stripped despite ADD_ATTR being honoured.
- **DOMPurify's `DATA_URI_TAGS` special-case branch overrides `ALLOWED_URI_REGEXP`.** Even with `ALLOWED_URI_REGEXP = /^(?:https?|mailto):/i`, an `<img src="data:...">` survives because img/source/audio/video/track sit in `DATA_URI_TAGS`. The spec-prescribed config alone is insufficient ‚Äî needs an explicit `uponSanitizeAttribute` strip per tag.
- **`uv run pytest` resolves to the system pytest if `~/.local/bin/pytest` predates the project venv's stub shebang.** Encountered this session: the venv's `.venv/bin/pytest` shebang pointed at the pre-storage-migration path `/mnt/Emulators/...` (since dead); shell fall-through hit `/home/ants/.local/bin/pytest` which lacks pytest-cov. Fix: `uv venv --clear && uv sync --extra dev` regenerates shebangs against the current venv path.
- **Spec-mandated route coverage drift is easy to miss.** FP20-C's "deferred" comment for `curate.py` and `games.py` reads as a deliberate scope narrowing, but the P04 spec table is normative ‚Äî every row must be lock-guarded. The closing `/indie-review` Lane 3 caught the gap; without that pass, the half-delivery would have shipped as "complete". Standing recommendation: every P##/FP## that touches a spec table should grep the spec for surfaces it's narrowing.
