# P08 — Setup wizard (closed 2026-05-04, slim)

## What changed

Slim P08 — **bootstrap scripts only**, no in-browser wizard.

- **A — `run.sh` (Linux/macOS).** Five steps: python3 detection
  (3.12+, with platform-specific install hints if missing) → uv
  auto-install via the official installer (https://astral.sh/uv) →
  `uv sync` → interactive `mame-curator setup` if config.yaml
  absent → `mame-curator serve` + best-effort browser open
  (`xdg-open` / `open`). Idempotent. ~95 lines of bash.

- **B — `run.bat` (Windows).** Same flow translated for cmd.exe;
  PowerShell `irm | iex` for the uv installer path. ~75 lines.

## What was learned (and Karpathy 9 push-back)

The original P08 contract called for a full Stage 2 in-browser
wizard: multi-step UI (Welcome → Source paths with `<FsBrowser>`
→ Destination → Reference data download → Filter preview → Filter
customization → Done) + serializable state persistence across
reboots + cross-platform UAT on three clean VMs.

Karpathy 9 push-back at session start: that's substantial scope
for a feature whose v1 user value can be served by:

1. The existing **`mame-curator setup` CLI** (terminal-prompted,
   ships since P00) — writes a starter config.yaml.
2. The existing **Settings → Setup banner** (FP11 § B3) — surfaces
   missing-paths state once the user is in the SPA.
3. The new **`run.sh` / `run.bat`** scripts that wire those two
   together for a "clone, double-click, follow the prompts" UX.

The deferred surface — in-browser wizard, FsBrowser-driven path
picking, filter preview during onboarding, state persistence —
is real value but **not v1-blocking**. Captured as a post-v1
candidate in ROADMAP § P08's scope-refinement note.

## What was deferred

- **Stage 2 in-browser wizard.** Multi-step `<WizardLayout>` with
  Welcome / Paths / Destination / Reference data / Filter preview
  / Done steps. Reuses `<FsBrowser>` from FP12 § G when revived.
  Captured as a post-v1 P13 candidate.

- **Resumable wizard state.** `data/.wizard-state.json` round-trip
  + restart-mid-flow recovery. Lands with the in-browser wizard.

- **Cross-platform UAT on clean VMs.** Original contract required
  Linux / macOS / Windows clean-machine validation. For v1, the
  user (project author) does Linux UAT; macOS + Windows real-user
  validation happens organically post-release. Smoke testing the
  bash script's syntax (`bash -n run.sh`) is the close gate.

- **Two-flow `-listxml` acquisition** (per ADR-0003). The user
  supplies `-listxml` themselves for v1 (already documented in the
  README / setup prompts). Tiered acquisition (local mame binary
  → user-supplied path → community mirror) is post-v1.

## False positives confirmed

- None — no audits run on P08 close per session-wide direction.
  Local gate: `bash -n run.sh` syntax-check passes; the existing
  `mame-curator setup --help` smoke confirms the CLI dependency
  surface is intact.
