## Contract

Given a parsed DAT and the existing P05 libretro-thumbnails baseline, this phase adds four fallback art / metadata sources so games the upstream MAME repo doesn't cover still surface a face ‚Äî and so the Alternatives drawer can show a sentence of flavor text. Specifically:

1. **Add four image sources** (`progettoSnaps`, `ArcadeDB`, `Wikipedia` *image*, `MobyGames`) that produce candidate URLs for `boxart` / `title` / `snap` kinds. Each source is opt-in/opt-out via a new `media.sources` config array.
2. **Walk a fallback chain** ‚Äî for one `(machine, kind)` request, try sources in user-configured order; first source that returns a real image wins. 404 from one source falls through to the next; rate-limit / network errors fall through too, with structured logging.
3. **Add a Wikipedia flavor-text endpoint** (`GET /media/{name}/wiki`) returning ~1‚Äì2 sentences (the page's `extract` field) for the Alternatives drawer. Separate route because it returns JSON, not an image.
4. **Cache like P05** ‚Äî every source's URL feeds the same `fetch_with_cache` (image) or a parallel `fetch_text_with_cache` (Wikipedia extract). SHA-256-keyed flat directory; `.tmp + os.replace` atomic write; 404 returns `None`; non-200 raises `MediaFetchError`.
5. **License gate** ‚Äî each source carries a `license_compatible: bool` flag. Sources whose images aren't CC-BY-compatible (today: MobyGames) are flagged in the source metadata so a future P11 "contribute back to libretro-thumbnails" flow can short-circuit on them. P10 itself only reads images for display; the gate is data that P11 will consume.
6. **Rate-limit handling** ‚Äî per-source token-bucket limiter inside the process; on exhaustion the source signals "soft-skip" so the chain moves on. No retries. No persistent state (token bucket lives in-memory; resets on `mame-curator serve` restart).

The module depends on `parser/` (for `Machine`) and `_atomic.atomic_write_bytes` only ‚Äî same shape as P05. It MUST NOT import from `api/`, `filter/`, `copy/`, or `updates/` (the four sibling top-level packages under `src/mame_curator/`). The route wiring lives in `api/routes/media.py` (caller side); `media/` itself stays HTTP-agnostic.

EmuMovies is explicitly **out of scope** (paid account required ‚Äî `ROADMAP.md` ¬ß `mame-curator-1005`).

## Public API

### `Kind` (new typed alias)

```python
Kind = Literal["boxart", "title", "snap"]
```

P10 **introduces** this typed alias in `media/sources.py` (it does not pre-exist ‚Äî `api/routes/media.py:27` currently uses `_VALID_KINDS = {"boxart", "title", "snap", "video"}` as an untyped set, with `video` short-circuited inside the handler per P05). The new alias *excludes* `video` deliberately: `MediaUrls` has no `video` field (P05 spec ¬ß "class MediaUrls") so no `MediaSource` can ever cover that kind. The route's `_VALID_KINDS` set stays unchanged (it gates user input including video, which still 404s); the new `Kind` literal is the *source-chain* vocabulary.

Wikipedia *flavor text* is not a `Kind`; it has its own endpoint (¬ß "Wikipedia flavor-text surface ‚Äî new").

### `class MediaSource(Protocol)`

Every source implements this interface. The `@runtime_checkable` decorator lets the registry call `isinstance(x, MediaSource)` at registration time ‚Äî per PEP 544 this verifies *attribute presence* only (the four names below exist on the object), not method signatures or `ClassVar` types. Signature correctness is enforced by `mypy` at build time, not by the runtime check; the runtime check is a defence-in-depth gate against truly-malformed extensions, not a typed contract.

```python
@runtime_checkable
class MediaSource(Protocol):
    name: ClassVar[str]                    # config key ‚Äî "libretro", "progettoSnaps", ...
    license_compatible: ClassVar[bool]     # future contribute-back gate
    kinds: ClassVar[frozenset[Kind]]       # which kinds this source covers

    # NEW (readiness surface). Sources that decide they can't serve any
    # request set this to a human-readable reason. The registry filters
    # them out of the fallback chain; the readiness endpoint surfaces the
    # string to the UI. ``None`` means "ready". Instance-level (not
    # ``ClassVar``) because the same Source class can be ready on one
    # process and disabled on another (different env, missing dotfile).
    disabled_reason: str | None

    async def prepare(self, machine: Machine, *, client: httpx.AsyncClient) -> None:
        """Populate any per-machine lookup state this source needs.

        Two-step sources (ArcadeDBSource, MobyGamesSource) hit their JSON
        endpoint here, cache the result via ``fetch_text_with_cache``, parse
        the body, and populate ``self._url_cache``. Single-shot sources
        (LibretroSource, ProgettoSnapsSource, WikipediaImageSource) still
        implement ``prepare`` explicitly ‚Äî see "Single-shot vs two-step
        sources" below ‚Äî but the body is just ``return``. The Protocol's
        ``...`` is a typing stub, not an inheritable default; every
        implementer must provide the method.

        ``MediaRateLimited`` may be raised here ‚Äî that's the canonical signal
        the orchestrator catches to fall through to the next source. Network
        / parse errors raise ``MediaFetchError``.
        """
        ...

    def url_for(self, machine: Machine, kind: Kind) -> str | None:
        """Return URL for (machine, kind), or None if this source has no candidate.

        ``None`` is the deterministic "skip this source for this machine" signal ‚Äî
        e.g. a source that only has flyers returns None for ``kind="snap"``,
        or ArcadeDBSource returns None when ``prepare`` hasn't been awaited yet
        (the lookup-cache miss). Distinct from a 404 at fetch time (network
        outcome, not lookup outcome).

        SYNC by design ‚Äî the protocol's deterministic shape lets the orchestrator
        iterate sources in a tight inner loop without per-source ``await``.
        """
        ...
```

**Single-shot vs two-step sources.** Single-shot sources implement `prepare` as `async def prepare(self, machine, *, client): return` ‚Äî explicit, one-line body. Two-step sources do the lookup in `prepare` and stash the URL on `self._url_cache[machine.name][kind]`; their `url_for` reads from that dict. The orchestrator awaits `prepare` once per `(source, machine)` before iterating kinds.

**Disabled-source mechanism.** A source that can't serve any request sets its instance-level `disabled_reason: str | None` to a human-readable string at construction (or mid-process for runtime-disabled flows). The registry's `chain_for(kind)` filters out sources with non-None `disabled_reason` *before* the orchestrator iterates, so disabled sources never get a `prepare` call or a `url_for` call. The reason string is what the readiness endpoint (¬ß "Readiness surface ‚Äî UI + API for config-bearing sources") returns to the UI.

For MobyGames specifically (the only P10 source that can become disabled ‚Äî others have no config to be missing):

- **Construction-time disable.** `MobyGamesSource.__init__` checks the API key sources in order: `os.environ["MOBYGAMES_API_KEY"]`, then `CWD/data/secrets/mobygames.key` (mode-0600 enforced). On missing / mode-wrong / empty key ‚Üí emits one WARNING line and sets:
  ```python
  self.disabled_reason = (
      "No API key configured. Set MOBYGAMES_API_KEY env var, or paste a "
      "key into Settings ‚Üí Media (writes data/secrets/mobygames.key, mode 0600)."
  )
  ```
- **Runtime disable** (after 401/403 from a *configured* key). The disabled state must persist across the per-request source instances ‚Äî process-wide, not per-request. **As shipped (chunk 6), the reason rides on an injected `SourceDisabledFlag` holder, not a bare `app.state` string** (`media/` is HTTP-agnostic ‚Äî it can't reach back into `app.state`; see ¬ß "4. MobyGames ‚Üí Chunk 6 implementation notes"). The lifespan owns the instance as `app.state.mobygames_disabled` and injects it at construction (`MobyGamesSource(..., disabled_flag=...)`). On the first 401/403, `prepare` writes the failure reason to `disabled_flag.reason` and emits one WARNING. Subsequent `MobyGamesSource.__init__` calls read the flag; if non-None, `self.disabled_reason = disabled_flag.reason` regardless of whether the key now resolves (a process restart is required to clear the flag ‚Äî matches the "key is wrong; retrying won't help" semantics).
- **Generalises naturally.** Any future source that needs config follows the same pattern: set `disabled_reason` at construction; one-time WARNING; registry filters; readiness endpoint surfaces the string. No special-cases.

### `class MediaSourceRegistry`

```python
class MediaSourceRegistry:
    """Resolves a list of source names + a kind ‚Üí an ordered list of MediaSource."""

    def __init__(
        self,
        configured: tuple[str, ...],
        available: Mapping[str, MediaSource],  # chunk-7 realisation ‚Äî see note below
    ) -> None: ...
    def chain_for(self, kind: Kind) -> tuple[MediaSource, ...]:
        """Sources that cover ``kind`` AND are ready, in the user's configured order.

        Filtering rules, in order:

        1. Unknown source names are silently dropped with a one-time WARNING.
        2. "libretro" is appended if absent from the configured tuple (it's the baseline; `build_registry` always includes it in `available`).
        3. Sources that don't cover ``kind`` are filtered out (kind mismatch).
        4. Sources with non-None ``disabled_reason`` are filtered out (config
           missing / runtime-disabled). The readiness endpoint surfaces the
           reason to the UI separately ‚Äî the orchestrator just skips them.
        """
```

#### Chunk 7 implementation notes (shipped 2026-07-01 ‚Äî registry + orchestrator)

Three realisation pins differ from / extend the contract above; all are cold-eyes-reviewed (5 loops to a clean contract) and code-and-test-backed.

1. **The registry takes an `available: Mapping[str, MediaSource]`, not just `configured`.** The original signature (`__init__(self, configured)`) never said where source *instances* come from ‚Äî but concrete sources need runtime deps (`ArcadeDBSource(limiter=‚Ä¶, cache_dir=‚Ä¶)`, `MobyGamesSource(‚Ä¶, disabled_flag=‚Ä¶)`, `ProgettoSnapsSource(snap_dir=‚Ä¶)`). Per the spec's own "constructor-injection lets tests substitute a fake limiter without monkey-patching app state" principle, the composition root builds the instances (injecting the app-state limiters + the `SourceDisabledFlag`) and hands the registry a `name ‚Üí instance` map plus the order tuple. The registry stays a pure filter/orderer; tests build one from fake sources directly ‚Äî no `app.state`, no monkey-patching. `chain_for` maps each configured name through `available`; a name absent from `available` is the "unknown source" case (one-time WARNING, deduped process-wide via a module-level `set`, since the registry itself is per-request). The MobyGames *missing-key* WARNING gets the **same** process-wide dedup treatment (a module-level guard in `mobygames.py`, added by chunk 7 per the chunk-6 note's "the registry of chunk 7 owns any cross-request warning dedup") ‚Äî without it, per-request source reconstruction would log a keyless-MobyGames WARNING on every thumbnail request.
2. **A `build_registry(...)` factory in `media/resolve.py` is the composition root.** It constructs only the sources named in `configured` (‚à™ the `libretro` baseline) ‚Äî so removing `mobyGames` from `media.sources` also suppresses its keyless-startup WARNING. The route calls it with `world.config.media.sources` + the `app.state` limiters/flag. Keeping the factory in `media/` (deps passed explicitly) honours the anti-jump rule ‚Äî no `api/` import ‚Äî and keeps the route handler thin.
3. **`snap_dir` is the fixed `./data/snaps/snap` default** ‚Äî the extraction target `DEFAULT_DEST / "snap"` in `updates/snaps.py` (note: `refresh-snaps --dest` defaults to the *root* `./data/snaps`; the CLI extracts into the `snap/` subdir, and the source reads that subdir ‚Äî pass `./data/snaps/snap`, not the root, to the factory). No `MediaConfig` field couples the source's read location to the CLI's `--dest`; a user who runs `refresh-snaps --dest /elsewhere` will not be seen by the source. **Known gap ‚Üí deferred to a follow-up ROADMAP item** (a `media.snaps_dir` field binding both, sized as its own small pass ‚Äî out of chunk-7 scope). The `./data/snaps/snap` literal default lives on the `build_registry` `snap_dir` kwarg (a module constant in `media/resolve.py`), **not** on `ProgettoSnapsSource.__init__` ‚Äî the source keeps `snap_dir` a *required* keyword-only arg so the composition root owns the wiring; a test (or the future field) overrides via the factory kwarg.

**Route contract ‚Äî chunk 7 retires the 502 error surface for media (spec-ratified at Step 1, made concrete here).** `resolve_image` *swallows* `MediaFetchError` / `MediaRateLimited` from any one source and falls through to the next (see the orchestrator pseudocode above); it only ever returns a `Path` or `None`. So the route **will no longer map** an upstream 5xx / transport error to `502 media_upstream_error` ‚Äî a single source's failure just advances the chain. If the whole chain misses, the route raises the existing `MediaUpstreamNotFoundError(404)` (envelope code `media_upstream_not_found`, defined on that class in `api/errors.py`), matching this section's "route translates `None` ‚Üí 404 as today". Chunk 7 **rewrites** the three P05-era route tests that asserted 502 (`test_proxy_route_propagates_upstream_500`, `‚Ä¶_500_detail_no_double_wrap`, `‚Ä¶_transport_error_maps_to_502`) to the fall-through-then-404 behaviour, and **drops** the now-unused `MediaUpstreamError` import from `api/routes/media.py`. This is the multi-source design's intended trade-off: resilience (one flaky mirror can't fail the request) at the cost of the 5xx-vs-404 distinction on the media surface.

### `async resolve_image(machine, kind, *, registry, cache_dir, client) -> Path | None`

The new orchestrator. Pseudocode:

```python
async def resolve_image(
    machine: Machine,
    kind: Kind,
    *,
    registry: MediaSourceRegistry,
    cache_dir: Path,
    client: httpx.AsyncClient,
) -> Path | None:
    for source in registry.chain_for(kind):
        try:
            await source.prepare(machine, client=client)
        except MediaRateLimited:
            logger.info("media: %s rate-limited during prepare; falling through", source.name)
            continue
        except MediaFetchError:
            logger.warning(
                "media: %s prepare failed for %s; falling through", source.name, machine.name
            )
            continue
        url = source.url_for(machine, kind)
        if url is None:
            continue  # source has no candidate (lookup miss / kind not covered)
        try:
            path = await fetch_with_cache(url, cache_dir, client=client)
        except MediaFetchError:
            logger.warning(
                "media: %s fetch failed for %s/%s; falling through",
                source.name, machine.name, kind,
            )
            continue
        if path is not None:
            return path
        # path is None ‚Üí upstream 404 ‚Üí continue to next source
    return None
```

**Where `MediaRateLimited` is raised:** only from `source.prepare(...)` (the source's two-step lookup hits the token bucket on its way to the JSON endpoint). `fetch_with_cache` from P05 raises `MediaFetchError` only ‚Äî it never raises `MediaRateLimited`, because the rate-limit semantics belong to the source, not the binary cache. The orchestrator's `try/except` shape reflects this: rate-limit handling sits in the `prepare` block; fetch-time errors sit in the `fetch_with_cache` block.

**`fetch_with_cache` signature** (verbatim from `media/cache.py:56`, P05): `async def fetch_with_cache(url, cache_dir, *, client, max_bytes=DEFAULT_MAX_BYTES) -> Path | None`. The orchestrator uses default `max_bytes`; the 16 MiB cap is appropriate for all four image sources.

Returns the cached image path on first hit, `None` if every configured source 404s or skips. The route translates `None` ‚Üí `MediaUpstreamNotFoundError(404)` as today.

### `async resolve_wikipedia_extract(machine, *, cache_dir, client, limiter) -> WikipediaExtract | None`

Returns a frozen Pydantic model.

**Limiter injection (chunk-8 realisation).** Unlike `resolve_image` ‚Äî where the limiter travels *inside* each `MediaSource` object and the orchestrator signature takes no `limiter=` ‚Äî the Wikipedia extract is a standalone function with no source object to carry a bucket. So it takes an explicit `limiter: TokenBucket` kwarg, injected exactly as the image sources' buckets are (constructor-injection lets the rate-limit test substitute an exhausted bucket without app-state monkey-patching). The route passes **the existing `app.state.wikipedia_limiter`** ‚Äî the same bucket the chunk-5 `WikipediaImageSource` uses, because both hit the same `en.wikipedia.org` REST summary endpoint and should share one courtesy budget (and one cache slot: same URL ‚Üí same `fetch_text_with_cache` key, so an image lookup warms the extract's body cache and vice-versa). `resolve_wikipedia_extract` acquires one token first; an empty bucket raises `MediaRateLimited` before any upstream hit.

```python
class WikipediaExtract(BaseModel):
    model_config = ConfigDict(frozen=True, extra="forbid")
    title: str           # the wiki page title we landed on (post-redirect)
    extract: str         # up to ~600 chars, sentence-aligned
    url: str             # canonical wiki page URL for "read more" link
    license: str = "CC-BY-SA-4.0"   # constant ‚Äî Wikipedia's current text license
```

**Verified at pre-impl prep** (`tests/fixtures/wikipedia_pacman.json`): the Wikipedia REST summary response does **NOT** include a `license` field; the spec sets the constant `"CC-BY-SA-4.0"` client-side (the default value is what we always use). Field-shape fixes from the captured fixture: `title` comes from `data["title"]`; `extract` from `data["extract"]`; `url` from `data["content_urls"]["desktop"]["page"]`. Image URL (for `WikipediaImageSource`) comes from `data["thumbnail"]["source"]`.

The `license` field is a plain `str` (not `Literal[...]`) so a future Wikipedia license bump (CC-BY-SA 5.0 or successor) isn't a Pydantic schema-breaking change.

Stored as JSON in the cache directory (`fetch_text_with_cache` writes the response body verbatim; the parser sits at the route boundary).

### `async fetch_text_with_cache(url, cache_dir, *, client, max_bytes=DEFAULT_TEXT_MAX_BYTES) -> str | None`

Parallel to `fetch_with_cache` but for text / JSON bodies. Same 404 sentinel, same `MediaFetchError`, same atomic-write protocol. Lives in `cache.py`'s sibling `cache_text.py` so the binary-image path stays unchanged.

**Public named constant** (mirrors P05's `DEFAULT_MAX_BYTES` at `media/cache.py:31`; callers may import and pass a custom value):

```python
DEFAULT_TEXT_MAX_BYTES = 256 * 1024  # 256 KiB ‚Äî Wikipedia extracts <2 KiB,
                                     # ArcadeDB responses <50 KiB; 5x headroom
```

**Unhappy paths** (each raises `MediaFetchError`, leaves no `.tmp` behind):

| Upstream condition | Behavior |
|---|---|
| HTTP 404 | Return `None` (sentinel; no negative caching). |
| HTTP 200 + empty body | Raise `MediaFetchError("empty body for {url!r}")` ‚Äî same poison-cache guard as `fetch_with_cache` (P05 cache.py:139). |
| HTTP 200 + body exceeds `max_bytes` | Stream-abort, raise `MediaFetchError("BodyTooLarge: ‚Ä¶")`. |
| HTTP 5xx / non-404 non-200 | Raise `MediaFetchError("upstream {status} for {url!r}")`. |
| Network error (`httpx.HTTPError`) | Raise `MediaFetchError("network error for {url!r}: {exc}")` chained from `__cause__`. |
| Body is not valid UTF-8 | Raise `MediaFetchError("decode failed for {url!r}: {exc}")` chained from the `UnicodeDecodeError`. Falls back to UTF-8 if `response.encoding` is `None`. |
| `Content-Type` mismatch (e.g. upstream returns `text/html` on a soft-404 instead of expected JSON) | NOT enforced at this layer ‚Äî callers parse the body and surface their own typed errors. Keeps `fetch_text_with_cache` content-agnostic. |

Returns the body as `str` so callers can `json.loads` it themselves.

### `class MediaRateLimited(MediaError)`

New typed exception in `media/rate_limit.py`; re-exported from `media/__init__.py`. Raised by source `prepare(...)` implementations that hit their per-source token bucket. The orchestrator catches and continues; the route never sees it.

## Source contracts

### 0. libretro (baseline)

P10 keeps the P05 contract verbatim and re-homes it under the new `MediaSource` Protocol.

- **`name`:** `"libretro"`
- **`license_compatible`:** `True` (CC-BY-SA-4.0 ‚Äî the libretro-thumbnails MAME repo).
- **`kinds`:** `{"boxart", "title", "snap"}`
- **URL pattern:** unchanged from P05 ‚Äî `https://raw.githubusercontent.com/libretro-thumbnails/MAME/master/Named_{Boxarts,Titles,Snaps}/<escape_libretro+quote(description)>.png`. The `escape_libretro` rule and `MediaUrls` model stay in `media/urls.py`; `LibretroSource` imports them and adapts to the Protocol shape.
- **Rate limit:** none. Raw GitHub content has no documented per-IP cap relevant to single-user scale.
- **Auth:** none.
- **`prepare`:** no-op.
- **Failure modes:** 404 / 5xx ‚Üí fall through.
- **Estimate:** ~0.5 days. Pure refactor ‚Äî the existing P05 tests must still pass post-move.

### 1. progettoSnaps ‚Äî local pack model, snap kind only (revised 2026-05-18)

**Architectural pivots from the original spec:**

1. **Per-image direct URLs don't exist (verified 2026-05-17).** `curl -I https://www.progettosnaps.net/{flyers,titles,snap}/pacman.png` returned HTTP 404 on every attempt. ### Orchestrator amendment ‚Äî `file://` URL handling

To support the pack-based source, `resolve_image`'s pseudocode is amended:

```python
async def resolve_image(...):
    for source in registry.chain_for(kind):
        # ... prepare() block as before ...
        url = source.url_for(machine, kind)
        if url is None:
            continue
        if url.startswith("file://"):
            # Local-pack source (P10 ProgettoSnapsSource). Serve direct
            # ‚Äî no cache lookup, no network call. `fetch_with_cache`'s
            # _ALLOWED_URL_SCHEMES intentionally excludes file:// for
            # security (P05 cache.py:33 FP27 B4 guard), so we MUST
            # short-circuit here before reaching it.
            local = Path(urllib.parse.urlparse(url).path)
            if local.exists():
                return local
            continue  # pack file gone between url_for + return; fall through
        # ... fetch_with_cache block as before ...
```

The `file://` short-circuit honors the existing P05 security guard: `fetch_with_cache` rejects non-http(s) schemes with `MediaFetchError` (`cache.py:85-86`), which would 502 the route. The orchestrator's pre-check prevents that.

### `mame-curator refresh-snaps` CLI (new in P10)

```bash
uv run mame-curator refresh-snaps [--dest data/snaps/] [--url <pinned-pack-url>] [--force]
```

- **Discovery step (resolved 2026-05-18 at chunk 3a fixture capture):**
  - **Index URL:** `https://www.progettosnaps.net/snapshots`.
  - **Pack URL pattern:** `https://www.progettosnaps.net/snapshots/packs/full_sets/pS_snap_fullset_<NNN>.zip` where `<NNN>` is the MAME version (e.g. `287` for MAME 0.287). Direct fetch works ‚Äî no `download.php` wrapper, no cookies, no referer. Verified `HEAD https://www.progettosnaps.net/snapshots/packs/full_sets/pS_snap_fullset_287.zip` returns `HTTP/2 200`, `content-type: application/zip`, `content-length: 522800390` (~498 MB).
  - **Discovery method:** GET the index URL; parse `href=".../full_sets/pS_snap_fullset_<NNN>.zip"`; pick highest `<NNN>`.
  - **`--url` override:** if the user passes `--url`, skip discovery and use the given URL directly (escape hatch when upstream restructures or the user wants a pinned older version).
- **Checksum publishing (resolved 2026-05-18):** upstream does **not** publish SHA256s for snap packs (matches the AntoPISA INI refresh pattern from P07 ‚úÖ). The CLI relies on content-length validation (`downloads.py` already caps body size against the declared header) and trusts atomicity for partial-write safety.
- **Download step:** uses the existing `mame_curator.downloads.download(...)` primitive (P07 ‚úÖ) ‚Äî atomic write via `.tmp + os.replace`, streaming with the existing 100 MB default body cap raised to **600 MB for this command** (snap pack is ~500 MB; 600 MB cap leaves headroom for the next MAME version), retries with exponential backoff. No mirrors (upstream is single-origin).
- **Extraction step:** ZIP unzip into `data/snaps/snap/` flat layout. Existing files are overwritten only with `--force`; otherwise the CLI prints a summary of "would-replace" files and exits without writing.
- **Progress reporting:** rich-console progress bar for the download phase (`downloads.py` is streaming; the CLI wraps it with a `Progress` bar driven off content-length); per-file count for the extraction phase.
- **Disk space check:** before download, verify `--dest`'s parent has at least 2√ó the pack's stated content-length free (so post-extract + cached ZIP both fit); abort with a clear error if not.
- **Modes 0644 for files, 0755 for directories** ‚Äî standard, no special permissions.

### 2. ArcadeDB

- **`name`:** `"arcadeDB"`
- **`license_compatible`:** `True` (CC-BY 3.0 per the project's own statement).
- **`kinds`:** `{"boxart", "title", "snap"}`
- **Endpoint:** `http://adb.arcadeitalia.net/service_scraper.php?ajax=query_mame&game_name={name}`. Returns JSON; we extract the `url_image_ingame`, `url_image_title`, `url_image_flyer` fields.
- **Two-step lookup** (implemented inside `ArcadeDBSource.prepare(machine, client)`); shapes verified at pre-impl prep against `tests/fixtures/arcadedb_pacman.json`:
  1. Acquire one token from the per-source `TokenBucket`; on empty bucket ‚Üí `MediaRateLimited`.
  2. Build the scraper URL (`http://adb.arcadeitalia.net/service_scraper.php?ajax=query_mame&game_name={machine.name}`) ‚Äî note: returns **HTTP 301 ‚Üí HTTPS**; the lifespan client's `follow_redirects=True` (FP10 A1) handles this transparently. Hit it once per machine via `fetch_text_with_cache(url, cache_dir, client=client)`. **Same cache function, same SHA-256-of-URL cache key as image fetches** ‚Äî no special prefix or alternate keying needed; URLs differ ‚Üí hashes differ ‚Üí no collision.
  3. **Parse-before-trust** (cache-poisoning guard): `json.loads` the returned text. If parse fails (truncated TLS body, mid-deploy HTML-instead-of-JSON, etc.), `Path(cache_file).unlink(missing_ok=True)` to invalidate the bad slot, then raise `MediaFetchError(...)` chained from the `JSONDecodeError`. The next request re-fetches; transient bad responses don't permanently disable the source for this machine.
  4. On valid JSON: the top-level shape is `{"release": <int count>, "result": [...]}` ‚Äî **the array is `result`, not `release`** (the latter is the integer match count). If `result` is non-empty, stash `_url_cache[machine.name] = {"boxart": ..., "title": ..., "snap": ...}` from `result[0]`'s `url_image_flyer` / `url_image_title` / `url_image_ingame` fields. These URLs are **redirector-form** (`https://adb.arcadeitalia.net/?mame=pacman&type=flyer&resize=0`) ‚Äî `fetch_with_cache` follows the redirect to the actual image. Empty `result` array ‚Üí leave `_url_cache[machine.name]` absent (uniform negative-cache shape ‚Äî see MobyGames below).
  5. `url_for(machine, kind)` reads from `self._url_cache.get(machine.name, {}).get(kind)`; missing entry ‚Üí returns `None`.
- **Rate limit:** the public service has no documented hard cap, but the project is a hobby site. Token bucket: 30 req/min default, configurable via `media.arcadedb_rate_limit_per_min` (new field, optional, defaults to 30).
- **Auth:** none.
- **Failure modes:** all handled by step 3 above (parse-before-trust + cache-file invalidation). Network errors propagate as `MediaFetchError` from `fetch_text_with_cache` (already enumerated in the unhappy-paths table).
- **Estimate:** ~2 days (two-step lookup, JSON parse, rate-limit, integration tests with `respx`).

### 3. Wikipedia (image)

- **`name`:** `"wikipediaImage"`
- **`license_compatible`:** depends per image (Wikipedia hosts mixed licenses ‚Äî public-domain screenshots, CC-BY screenshots, fair-use flyers). P10 sets `license_compatible = False` conservatively; the P11 contribute-back flow will need per-image license inspection if it ever wants to consume this source. P10 itself only displays the image.
- **`kinds`:** `{"boxart"}` only. The infobox image is the only field with a consistent, documented location in the Wikipedia REST summary response (`thumbnail.source`). Title screens and snap shots aren't reliably present and would require HTML scraping or guessing ‚Äî out of scope. Counterexample: a source that returned the infobox image for every `kind` would silently degrade the user's chain (the wrong-shaped image would win over a legit downstream candidate); the `frozenset({"boxart"})` shape prevents this at registry-build time.
- **Endpoint:** [Wikipedia REST API summary](https://en.wikipedia.org/api/rest_v1/page/summary/{title}). The `thumbnail.source` field is the image URL.
- **Title resolution:** `machine.description` passes through a small canonicalization step before hitting the Wikipedia REST URL ‚Äî arcade DAT descriptions typically read `"Pac-Man (Midway)"` or `"Street Fighter II: The World Warrior"`, and the Wikipedia summary endpoint expects the canonical article title.
  1. Strip trailing parenthesised qualifiers: `"Pac-Man (Midway)"` ‚Üí `"Pac-Man"`. Pattern: `re.sub(r"\s*\([^)]*\)\s*$", "", description).strip()`.
  2. Trim leading/trailing whitespace.
  3. URL-quote the result via `urllib.parse.quote`.
  No fuzzy match, no second attempt. If the canonicalised title 404s, fall through. Many arcade entries (regional ports, unofficial conversions, prototypes) won't hit Wikipedia even after canonicalisation ‚Äî accept this as the long-tail. The source's value is the head of the catalog (Pac-Man, Tetris, Donkey Kong, etc.), not full coverage.
- **Rate limit:** Wikipedia tolerates high request rates from clients sending a descriptive `User-Agent` (per the [API:Etiquette](https://www.mediawiki.org/wiki/API:Etiquette) page). Token bucket at 60 req/min as a courtesy cap ‚Äî well under any plausible upstream limit.
- **User-Agent:** set globally on the lifespan client per Wikipedia's API etiquette: `User-Agent: mame-curator/{VERSION} (+https://github.com/milnet01/mame-curator)`. **Implementation note:** the lifespan client is constructed inside the `lifespan(...)` startup block in `src/mame_curator/api/app.py` (anchor by symbol ‚Äî line numbers drift). As of P10 chunk 5 (shipped 2026-05-18) the constructor reads `httpx.AsyncClient(timeout=10.0, follow_redirects=True, headers={"User-Agent": _build_user_agent()})`; `_build_user_agent()` is defined in `media/__init__.py` so the helper lives next to the consumer, not in the API package.
- **Failure modes:** 404, 5xx ‚Üí fall through.
- **Estimate:** ~1 day.

### 4. MobyGames

- **`name`:** `"mobyGames"`
- **`license_compatible`:** `False` (MobyGames API terms restrict redistribution; in-app display is fine, contribute-back is not).
- **`kinds`:** `{"boxart"}` ‚Äî port covers only. MobyGames lists screenshots too, but their per-game pagination is heavy and the catalog overlap with arcade specifically is narrow. P10 keeps the source thin; expanding to title/snap is a future fix-pass if usage warrants it.

- **API key:** required. Loaded in this order:
  1. **`MOBYGAMES_API_KEY` environment variable** ‚Äî primary path; nothing to install, nothing to configure beyond a shell export.
  2. **`data/secrets/mobygames.key`** ‚Äî single-line file under an **injectable `secrets_dir` constructor kwarg** (default the relative `Path("data/secrets")`, which resolves against the process CWD like `MediaConfig.cache_dir`'s `Path("./data/media-cache")`; tests point it at a tmpdir). Fallback for users who'd rather not edit shell config.

The MobyGames source is *enabled* via `media.sources` but only *works* when the key resolves at source construction (per request ‚Äî sources are re-created per request, but the key is read from env / dotfile each time). Missing key ‚Üí source sets its instance `disabled_reason` (a human-readable string; the registry filters disabled sources out of the chain), one-time WARNING at construction, and `prepare` / `url_for` short-circuit to no-op. (Realised as `disabled_reason` + the injected `SourceDisabledFlag`, not a private `_enabled` bool ‚Äî see ¬ß chunk-6 implementation notes.)
  - `keyring` integration is **deferred to P11** (rationale + canonical statement in ¬ß "Out of scope (deferred)" ‚Äî P11 already needs `keyring` for its GitHub PAT path, so folding the MobyGames key under the same umbrella avoids two parallel secret stores). P10 ships env-var + dotfile only.
- **Rate limit:** MobyGames documents 360 req/hr for free accounts; token bucket at 5 req/min (60% margin). Configurable via `media.mobygames_rate_limit_per_min`.
- **Failure modes:**
  - 401 / 403 ‚Üí source sets the injected process-wide `SourceDisabledFlag.reason` (and its own `disabled_reason`) for the rest of the process lifetime (key is wrong; retrying won't help). One-time WARNING. Subsequent `MobyGamesSource` constructions replay the flag, so `prepare` / `url_for` short-circuit immediately.
  - 429 ‚Üí `MediaRateLimited`; orchestrator falls through.
  - 404 ‚Üí `fetch_text_with_cache` returns `None`; `prepare` leaves `_url_cache[machine.name]` absent; `url_for` returns `None`. No synthetic cache body.
  - 200 with empty `games` array (i.e. the title produced zero matches) ‚Üí cached upstream body replays on next `fetch_text_with_cache` call without an upstream hit; `prepare`'s parse step sees an empty list and leaves `_url_cache[machine.name]` absent. Same uniform negative-cache shape as ArcadeDB (cached real body, missing dict entry).
  - JSON-parse failure ‚Üí same parse-before-trust / unlink / re-raise pattern as ArcadeDB step 3.
- **Estimate:** ~2 days.

#### Chunk 6 implementation notes (shipped 2026-07-01 ‚Äî key-handling half)

The user elected **"key-handling now, fetch later"** at the chunk-6 gate (the cover parse needs a real API key to verify the response field path ‚Äî none on this machine, no anonymous API access). Three realisation pins differ from the contract above; all are deliberate, code-and-test-backed:

1. **`SourceDisabledFlag` holder, not a bare `app.state.mobygames_disabled_reason: str`.** The spec places the runtime-disabled reason on a bare `app.state` string, but `media/` is HTTP-agnostic (never imports `api/`) and `prepare` only receives an httpx client ‚Äî it has no app handle to write a string back through. So the reason rides on a small mutable holder (`SourceDisabledFlag`, in `media/mobygames.py`), injected at construction exactly as the token buckets are. The lifespan owns the instance as `app.state.mobygames_disabled`; tests construct their own.
2. **`prepare` makes its own `client.get`, not `fetch_text_with_cache`.** That shared helper collapses every non-200/404 into one generic `MediaFetchError` *embedding the full URL* ‚Äî both fatal here: MobyGames must tell 401 (disable) from 429 (rate-limit) apart, and the API key rides in the query string so the URL must never reach a log line. So `prepare` inspects `resp.status_code` directly and redacts the key (`_redact`) in every error message. JSON-body caching moves into the deferred half (it belongs with the parse).
3. **`MobyGamesSource` lives in `media/mobygames.py`, not `sources.py`.** Adding it inline would have breached the 500-line hard cap (the inline draft measured ~596 lines during the chunk-6 session); it carries the most machinery (the only disable-able source), so it was extracted to its own module. This pre-empts the chunk-7 split the ¬ß "Module layout" note anticipated. `media/__init__.py` re-exports `MobyGamesSource` + `SourceDisabledFlag`.

**Deferred to [`mame-curator-1079`]:** the 200-path cover-URL parse + JSON-body caching. Until it lands a 200 leaves `_url_cache` empty (`url_for` ‚Üí `None`); MobyGames is key-aware and chain-eligible but yields no covers. Locked by `test_mobygames_source_200_does_not_populate_cover_yet` (the delete-point for the follow-up). The "one-time WARNING" for a *missing* key is per-construction here (matches the named test); the per-request registry of chunk 7 owns any cross-request warning dedup.

#### Chunk 9 implementation notes (shipped 2026-07-01 ‚Äî readiness API)

Three realisation pins differ from / extend the contract above; all cold-eyes-reviewed (2 loops) and code-and-test-backed.

1. **`GET /api/media/sources` needs `request` (app.state), not just `world`.** The readiness rows' `enabled` / `disabled_reason` are a source's *constructed* state ‚Äî mobyGames resolves its key, progettoSnaps probes its pack dir ‚Äî so the route must build the sources, which needs the app-state limiters + the `SourceDisabledFlag` (exactly as `build_registry` does for the image chain). The signature becomes `media_sources(request: Request, world: WorldState = Depends(get_world))`. A new `build_all_sources(...)` factory in `media/resolve.py` constructs **all five** known sources (regardless of config ‚Äî readiness must report unconfigured ones too), sharing the source-factory table with `build_registry` (which still builds only the configured subset, to keep the keyless-mobyGames WARNING suppressed when mobyGames isn't in the chain). Constructing mobyGames here fires that WARNING at most once per process (the chunk-7 dedup guard), which is acceptable for a rare surface-only call.

### Frontend hook

```ts
// frontend/src/hooks/useMediaSources.ts ‚Äî uses the real client (apiRequest /
// apiRequestVoid + useApiQuery), NOT an `api.get/api.put` facade (there is
// none ‚Äî every hook validates the 2xx body against a Zod schema).
const MEDIA_SOURCES_KEY = ['media', 'sources'] as const

export function useMediaSources() {
  return useApiQuery(MEDIA_SOURCES_KEY, '/api/media/sources', SourceReadinessSchema, {
    staleTime: 5_000, // readiness can flip mid-process after a PUT / a 401
  })
}

export function useSaveSourceSecret() {
  const qc = useQueryClient()
  return useMutation({
    mutationFn: ({ name, secret }: { name: string; secret: string }) =>
      apiRequestVoid(`/api/media/sources/${encodeURIComponent(name)}/secret`, {
        method: 'PUT',
        body: { secret },
      }),
    onSuccess: () => qc.invalidateQueries({ queryKey: MEDIA_SOURCES_KEY }),
    onError: toastApiError,
  })
}
```

`staleTime: 5_000` (5 seconds) because the readiness state can flip server-side mid-process (after a 401 from MobyGames, or after a successful `PUT /secret` from another tab). Long-cache would lie to the user. The query key is the tuple `['media', 'sources']` (house style, matching `useFs`'s `['fs', ‚Ä¶]`) ‚Äî the mutation invalidates the **same** tuple so the Configure-modal 204 actually triggers the readiness refetch.

### New API route

```python
# response_model=WikipediaExtract | None ‚Äî see Open Verification Items ¬ß
# for the FastAPI version compatibility note.
@router.get("/media/{name}/wiki", response_model=WikipediaExtract | None)
async def media_wiki(
    name: str,
    request: Request,
    world: WorldState = Depends(get_world),
) -> WikipediaExtract | None:
    machine = world.machines.get(name)
    if machine is None:
        raise GameNotFoundError(f"game not found: {name!r}")
    client = request.app.state.media_client
    try:
        extract = await resolve_wikipedia_extract(
            machine,
            cache_dir=world.config.media.cache_dir,
            client=client,
            limiter=request.app.state.wikipedia_limiter,  # shared with WikipediaImageSource
        )
    except MediaError:
        # About text is non-essential flavor ‚Äî degrade to null on ANY
        # media-layer failure (rate-limit, network, parse) rather than 500.
        # MediaError is the base of MediaRateLimited + MediaFetchError.
        return None
    return extract  # may be None (upstream 404) ‚Üí frontend hides the AboutSection
```

**Failure-mode contract (chunk-8 realisation).** `resolve_wikipedia_extract` returns `None` for a genuine upstream 404 (no wiki page) and *raises* `MediaRateLimited` (empty bucket) / `MediaFetchError` (network / parse) ‚Äî mirroring the source `prepare` pattern. The route catches `MediaError` (their common base) and returns `null`, so a rate-limited or flaky Wikipedia never 500s a non-essential "About" paragraph ‚Äî it just vanishes, matching the `AboutSection` UX (`isLoading || isError` ‚Üí render nothing). This is the deliberate contrast with the image route: `resolve_image` swallows per-source errors *internally* (many sources to fall through); the single-source wiki route swallows at the route boundary.

`get_world` is the canonical dependency helper at `api/routes/_deps.py:14`; matches the wiring in every other media-adjacent route. Returns `null` on miss (cleanest contract for the React component above ‚Äî `data === null` is the explicit "no wiki page" signal).

## Architecture notes

- **Protocol over inheritance.** `MediaSource` is a `Protocol` so the five concrete implementations can stay simple ‚Äî three `ClassVar`s and two methods (`prepare`, `url_for`). No base class, no `super()`, no MRO surprises. The orchestrator only depends on the protocol surface.
- **Two-phase per-source dispatch: async `prepare` then sync `url_for`.** Sources that need a lookup (ArcadeDB, MobyGames) populate their internal cache inside `prepare(machine, *, client)`; single-shot sources leave `prepare` as a no-op. `url_for` stays sync and deterministic; `resolve_image` awaits `prepare` once per `(source, machine)` before iterating kinds.
- **No async lock around the per-source lookup cache.** Concurrent `prepare(machine)` calls from two parallel requests for the same machine are safe at the cache layer (`fetch_text_with_cache` uses the same `.tmp + os.replace` discipline as P05's binary path; both writers race-write to a unique `.tmp` and the second `os.replace` wins atomically). At single-user scale, the wasted second JSON parse is acceptable; pre-warming an `asyncio.Lock` per source √ó machine doesn't pay off ‚Äî same trade-off the inline concurrency-note paragraph in P05 spec (P05.md:105) already pinned for the binary cache.
- **Per-source rate-limiter as a constructor injection.** `ArcadeDBSource(limiter=TokenBucket(rate=30/60))`. App-level `state.arcadedb_limiter` is constructed once at lifespan startup and shared across all per-request source instances ‚Äî the limiter holds the token-bucket state that *must* persist process-wide, even as sources are recreated per request. Construction-time injection (vs. e.g. `request.app.state.arcadedb_limiter` lookups inside `prepare`) lets tests substitute a fake limiter without monkey-patching app state. Same model for Wikipedia (`state.wikipedia_limiter`) and MobyGames (`state.mobygames_limiter`).
- **The source registry is per-request, not per-process.** It's built from `world.config.media.sources` on each `resolve_image` call. This means a `PATCH /api/config` that changes the source order takes effect on the next request without a server restart ‚Äî matches P05's "read config on every render" pattern. **Two consequences worth pinning:** (1) each source instance's `_url_cache` is dropped at request end ‚Äî `prepare` re-runs on the next request, but the on-disk `fetch_text_with_cache` body cache catches it without an upstream hit; the in-memory dict is just a per-request parsed view. (2) Process-wide state (token buckets, the disabled-flag on MobyGames after a 401) lives on the *limiter* / *app state*, not on the per-request source ‚Äî same separation as Django's "request lifecycle" pattern.
- **No async lock in the binary image cache.** Inherited from P05.
- **The Wikipedia extract is text, not an image, so it lives next to image fetching but doesn't share the binary cache path.** `fetch_text_with_cache` is the sibling. Same `.tmp + os.replace` discipline, different cache-key namespace (different URLs), different `max_bytes` default (`DEFAULT_TEXT_MAX_BYTES = 256 KiB`).
