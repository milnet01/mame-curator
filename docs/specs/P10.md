## Contract

Given a parsed DAT and the existing P05 libretro-thumbnails baseline, this phase adds four fallback art / metadata sources so games the upstream MAME repo doesn't cover still surface a face — and so the Alternatives drawer can show a sentence of flavor text. Specifically:

1. **Add four image sources** (`progettoSnaps`, `ArcadeDB`, `Wikipedia` *image*, `MobyGames`) that produce candidate URLs for `boxart` / `title` / `snap` kinds. Each source is opt-in/opt-out via a new `media.sources` config array.
2. **Walk a fallback chain** — for one `(machine, kind)` request, try sources in user-configured order; first source that returns a real image wins. 404 from one source falls through to the next; rate-limit / network errors fall through too, with structured logging.
3. **Add a Wikipedia flavor-text endpoint** (`GET /media/{name}/wiki`) returning ~1–2 sentences (the page's `extract` field) for the Alternatives drawer. Separate route because it returns JSON, not an image.
4. **Cache like P05** — every source's URL feeds the same `fetch_with_cache` (image) or a parallel `fetch_text_with_cache` (Wikipedia extract). SHA-256-keyed flat directory; `.tmp + os.replace` atomic write; 404 returns `None`; non-200 raises `MediaFetchError`.
5. **License gate** — each source carries a `license_compatible: bool` flag. Sources whose images aren't CC-BY-compatible (today: MobyGames) are flagged in the source metadata so a future P11 "contribute back to libretro-thumbnails" flow can short-circuit on them. P10 itself only reads images for display; the gate is data that P11 will consume.
6. **Rate-limit handling** — per-source token-bucket limiter inside the process; on exhaustion the source signals "soft-skip" so the chain moves on. No retries. No persistent state (token bucket lives in-memory; resets on `mame-curator serve` restart).

The module depends on `parser/` (for `Machine`) and `_atomic.atomic_write_bytes` only — same shape as P05. It MUST NOT import from `api/`, `filter/`, `copy/`, or `updates/` (the four sibling top-level packages under `src/mame_curator/`). The route wiring lives in `api/routes/media.py` (caller side); `media/` itself stays HTTP-agnostic.

EmuMovies is explicitly **out of scope** (paid account required — `ROADMAP.md` § `mame-curator-1005`).

## Public API

### `Kind` (new typed alias)

```python
Kind = Literal["boxart", "title", "snap"]
```

P10 **introduces** this typed alias in `media/sources.py` (it does not pre-exist — `api/routes/media.py:27` currently uses `_VALID_KINDS = {"boxart", "title", "snap", "video"}` as an untyped set, with `video` short-circuited inside the handler per P05). The new alias *excludes* `video` deliberately: `MediaUrls` has no `video` field (P05 spec § "class MediaUrls") so no `MediaSource` can ever cover that kind. The route's `_VALID_KINDS` set stays unchanged (it gates user input including video, which still 404s); the new `Kind` literal is the *source-chain* vocabulary.

Wikipedia *flavor text* is not a `Kind`; it has its own endpoint (§ "Wikipedia flavor-text surface — new").

### `class MediaSource(Protocol)`

Every source implements this interface. The `@runtime_checkable` decorator lets the registry call `isinstance(x, MediaSource)` at registration time — per PEP 544 this verifies *attribute presence* only (the four names below exist on the object), not method signatures or `ClassVar` types. Signature correctness is enforced by `mypy` at build time, not by the runtime check; the runtime check is a defence-in-depth gate against truly-malformed extensions, not a typed contract.

```python
@runtime_checkable
class MediaSource(Protocol):
    name: ClassVar[str]                    # config key — "libretro", "progettoSnaps", ...
    license_compatible: ClassVar[bool]     # future contribute-back gate
    kinds: ClassVar[frozenset[Kind]]       # which kinds this source covers

    # NEW (readiness surface). Sources that decide they can't serve any
    # request set this to a human-readable reason. The registry filters
    # them out of the fallback chain; the readiness endpoint surfaces the
    # string to the UI. ``None`` means "ready". Instance-level (not
    # ``ClassVar``) because the same Source class can be ready on one
    # process and disabled on another (different env, missing dotfile).
    disabled_reason: str | None

    async def prepare(self, machine: Machine, *, client: httpx.AsyncClient) -> None:
        """Populate any per-machine lookup state this source needs.

        Two-step sources (ArcadeDBSource, MobyGamesSource) hit their JSON
        endpoint here, cache the result via ``fetch_text_with_cache``, parse
        the body, and populate ``self._url_cache``. Single-shot sources
        (LibretroSource, ProgettoSnapsSource, WikipediaImageSource) still
        implement ``prepare`` explicitly — see "Single-shot vs two-step
        sources" below — but the body is just ``return``. The Protocol's
        ``...`` is a typing stub, not an inheritable default; every
        implementer must provide the method.

        ``MediaRateLimited`` may be raised here — that's the canonical signal
        the orchestrator catches to fall through to the next source. Network
        / parse errors raise ``MediaFetchError``.
        """
        ...

    def url_for(self, machine: Machine, kind: Kind) -> str | None:
        """Return URL for (machine, kind), or None if this source has no candidate.

        ``None`` is the deterministic "skip this source for this machine" signal —
        e.g. a source that only has flyers returns None for ``kind="snap"``,
        or ArcadeDBSource returns None when ``prepare`` hasn't been awaited yet
        (the lookup-cache miss). Distinct from a 404 at fetch time (network
        outcome, not lookup outcome).

        SYNC by design — the protocol's deterministic shape lets the orchestrator
        iterate sources in a tight inner loop without per-source ``await``.
        """
        ...
```

**Single-shot vs two-step sources.** Single-shot sources implement `prepare` as `async def prepare(self, machine, *, client): return` — explicit, one-line body. Two-step sources do the lookup in `prepare` and stash the URL on `self._url_cache[machine.name][kind]`; their `url_for` reads from that dict. The orchestrator awaits `prepare` once per `(source, machine)` before iterating kinds.

**Disabled-source mechanism.** A source that can't serve any request sets its instance-level `disabled_reason: str | None` to a human-readable string at construction (or mid-process for runtime-disabled flows). The registry's `chain_for(kind)` filters out sources with non-None `disabled_reason` *before* the orchestrator iterates, so disabled sources never get a `prepare` call or a `url_for` call. The reason string is what the readiness endpoint (§ "Readiness surface — UI + API for config-bearing sources") returns to the UI.

For MobyGames specifically (the only P10 source that can become disabled — others have no config to be missing):

- **Construction-time disable.** `MobyGamesSource.__init__` checks the API key sources in order: `os.environ["MOBYGAMES_API_KEY"]`, then `CWD/data/secrets/mobygames.key` (mode-0600 enforced). On missing / mode-wrong / empty key → emits one WARNING line and sets:
  ```python
  self.disabled_reason = (
      "No API key configured. Set MOBYGAMES_API_KEY env var, or paste a "
      "key into Settings → Media (writes data/secrets/mobygames.key, mode 0600)."
  )
  ```
- **Runtime disable** (after 401/403 from a *configured* key). The disabled state must persist across the per-request source instances — process-wide, not per-request. The reason lives on **`app.state.mobygames_disabled_reason: str | None`** (lifespan-initialized to `None`). On the first 401/403, `prepare` writes the failure reason to that field and emits one WARNING. Subsequent `MobyGamesSource.__init__` calls read the app-state field; if non-None, `self.disabled_reason = app.state.mobygames_disabled_reason` regardless of whether the key now resolves (a process restart is required to clear the flag — matches the "key is wrong; retrying won't help" semantics).
- **Generalises naturally.** Any future source that needs config follows the same pattern: set `disabled_reason` at construction; one-time WARNING; registry filters; readiness endpoint surfaces the string. No special-cases.

### `class MediaSourceRegistry`

```python
class MediaSourceRegistry:
    """Resolves a list of source names + a kind → an ordered list of MediaSource."""

    def __init__(self, configured: tuple[str, ...]) -> None: ...
    def chain_for(self, kind: Kind) -> tuple[MediaSource, ...]:
        """Sources that cover ``kind`` AND are ready, in the user's configured order.

        Filtering rules, in order:

        1. Unknown source names are silently dropped with a one-time WARNING.
        2. "libretro" is appended if absent (it's the baseline).
        3. Sources that don't cover ``kind`` are filtered out (kind mismatch).
        4. Sources with non-None ``disabled_reason`` are filtered out (config
           missing / runtime-disabled). The readiness endpoint surfaces the
           reason to the UI separately — the orchestrator just skips them.
        """
```

### `async resolve_image(machine, kind, *, registry, cache_dir, client) -> Path | None`

The new orchestrator. Pseudocode:

```python
async def resolve_image(
    machine: Machine,
    kind: Kind,
    *,
    registry: MediaSourceRegistry,
    cache_dir: Path,
    client: httpx.AsyncClient,
) -> Path | None:
    for source in registry.chain_for(kind):
        try:
            await source.prepare(machine, client=client)
        except MediaRateLimited:
            logger.info("media: %s rate-limited during prepare; falling through", source.name)
            continue
        except MediaFetchError:
            logger.warning(
                "media: %s prepare failed for %s; falling through", source.name, machine.name
            )
            continue
        url = source.url_for(machine, kind)
        if url is None:
            continue  # source has no candidate (lookup miss / kind not covered)
        try:
            path = await fetch_with_cache(url, cache_dir, client=client)
        except MediaFetchError:
            logger.warning(
                "media: %s fetch failed for %s/%s; falling through",
                source.name, machine.name, kind,
            )
            continue
        if path is not None:
            return path
        # path is None → upstream 404 → continue to next source
    return None
```

**Where `MediaRateLimited` is raised:** only from `source.prepare(...)` (the source's two-step lookup hits the token bucket on its way to the JSON endpoint). `fetch_with_cache` from P05 raises `MediaFetchError` only — it never raises `MediaRateLimited`, because the rate-limit semantics belong to the source, not the binary cache. The orchestrator's `try/except` shape reflects this: rate-limit handling sits in the `prepare` block; fetch-time errors sit in the `fetch_with_cache` block.

**`fetch_with_cache` signature** (verbatim from `media/cache.py:56`, P05): `async def fetch_with_cache(url, cache_dir, *, client, max_bytes=DEFAULT_MAX_BYTES) -> Path | None`. The orchestrator uses default `max_bytes`; the 16 MiB cap is appropriate for all four image sources.

Returns the cached image path on first hit, `None` if every configured source 404s or skips. The route translates `None` → `MediaUpstreamNotFoundError(404)` as today.

### `async resolve_wikipedia_extract(machine, *, cache_dir, client) -> WikipediaExtract | None`

Returns a frozen Pydantic model. Limiter injection is consistent with the image-chain model: the source (here, the Wikipedia-extract fetcher constructed at lifespan-startup) owns its own `TokenBucket`. The orchestrator-level signature deliberately **does not** take a `limiter=` kwarg — the rate-limit policy travels with the source.

```python
class WikipediaExtract(BaseModel):
    model_config = ConfigDict(frozen=True, extra="forbid")
    title: str           # the wiki page title we landed on (post-redirect)
    extract: str         # up to ~600 chars, sentence-aligned
    url: str             # canonical wiki page URL for "read more" link
    license: str = "CC-BY-SA-4.0"   # constant — Wikipedia's current text license
```

**Verified at pre-impl prep** (`tests/fixtures/wikipedia_pacman.json`): the Wikipedia REST summary response does **NOT** include a `license` field; the spec sets the constant `"CC-BY-SA-4.0"` client-side (the default value is what we always use). Field-shape fixes from the captured fixture: `title` comes from `data["title"]`; `extract` from `data["extract"]`; `url` from `data["content_urls"]["desktop"]["page"]`. Image URL (for `WikipediaImageSource`) comes from `data["thumbnail"]["source"]`.

The `license` field is a plain `str` (not `Literal[...]`) so a future Wikipedia license bump (CC-BY-SA 5.0 or successor) isn't a Pydantic schema-breaking change.

Stored as JSON in the cache directory (`fetch_text_with_cache` writes the response body verbatim; the parser sits at the route boundary).

### `async fetch_text_with_cache(url, cache_dir, *, client, max_bytes=DEFAULT_TEXT_MAX_BYTES) -> str | None`

Parallel to `fetch_with_cache` but for text / JSON bodies. Same 404 sentinel, same `MediaFetchError`, same atomic-write protocol. Lives in `cache.py`'s sibling `cache_text.py` so the binary-image path stays unchanged.

**Public named constant** (mirrors P05's `DEFAULT_MAX_BYTES` at `media/cache.py:31`; callers may import and pass a custom value):

```python
DEFAULT_TEXT_MAX_BYTES = 256 * 1024  # 256 KiB — Wikipedia extracts <2 KiB,
                                     # ArcadeDB responses <50 KiB; 5x headroom
```

**Unhappy paths** (each raises `MediaFetchError`, leaves no `.tmp` behind):

| Upstream condition | Behavior |
|---|---|
| HTTP 404 | Return `None` (sentinel; no negative caching). |
| HTTP 200 + empty body | Raise `MediaFetchError("empty body for {url!r}")` — same poison-cache guard as `fetch_with_cache` (P05 cache.py:139). |
| HTTP 200 + body exceeds `max_bytes` | Stream-abort, raise `MediaFetchError("BodyTooLarge: …")`. |
| HTTP 5xx / non-404 non-200 | Raise `MediaFetchError("upstream {status} for {url!r}")`. |
| Network error (`httpx.HTTPError`) | Raise `MediaFetchError("network error for {url!r}: {exc}")` chained from `__cause__`. |
| Body is not valid UTF-8 | Raise `MediaFetchError("decode failed for {url!r}: {exc}")` chained from the `UnicodeDecodeError`. Falls back to UTF-8 if `response.encoding` is `None`. |
| `Content-Type` mismatch (e.g. upstream returns `text/html` on a soft-404 instead of expected JSON) | NOT enforced at this layer — callers parse the body and surface their own typed errors. Keeps `fetch_text_with_cache` content-agnostic. |

Returns the body as `str` so callers can `json.loads` it themselves.

### `class MediaRateLimited(MediaError)`

New typed exception in `media/rate_limit.py`; re-exported from `media/__init__.py`. Raised by source `prepare(...)` implementations that hit their per-source token bucket. The orchestrator catches and continues; the route never sees it.

### 1. progettoSnaps — local pack model (revised 2026-05-17)

**Architectural pivot from the original spec:** progettoSnaps does **not** serve per-image direct HTTP URLs. Verified at the pre-impl prep step: `curl -I https://www.progettosnaps.net/{flyers,titles,snap}/pacman.png` returned HTTP 404 on every attempt. Their distribution model is **downloadable pack archives** (~hundreds of MB each, sometimes GB-scale), refreshed periodically. P10 adopts a **download-once, serve-from-disk** model.

- **`name`:** `"progettoSnaps"`
- **`license_compatible`:** `True` (CC-BY-NC-SA 4.0 — compatible with libretro-thumbnails for in-app display; P11 contribute-back will need a license-conversion check).
- **`kinds`:** `{"boxart", "title", "snap"}`
- **Local layout** (created by `mame-curator refresh-snaps`):
  ```
  data/snaps/
  ├── flyers/   <name>.png  (boxart kind)
  ├── titles/   <name>.png  (title kind)
  └── snap/     <name>.png  (snap kind)
  ```
  Filename is the short name (e.g. `pacman.png`), matching upstream's pack-internal layout.
- **`prepare(machine, *, client)`:** no-op (this source never hits the network).
- **`url_for(machine, kind)`:** returns `f"file://{cwd}/data/snaps/{folder_for[kind]}/{machine.name}.png"` *if* the file exists on disk; else `None`. The `folder_for` map is `{"boxart": "flyers", "title": "titles", "snap": "snap"}`. The source caches the existence check across `url_for` calls within one request (a `set[str]` of resolved-present basenames) so repeated calls don't `stat()` repeatedly. Cache lifetime is per-request (matches the per-request source instance model in § Architecture notes).
- **`disabled_reason`:** set to `"Pack not downloaded. Run `mame-curator refresh-snaps` to fetch the latest progettoSnaps art packs."` on `__init__` if `data/snaps/` doesn't exist or is empty. The readiness endpoint surfaces this to the UI; Settings → Media gets a `[Download pack…]` button (the equivalent of MobyGames' `Configure key`).
- **Auth:** none.
- **Rate limit:** none (no network calls from the source). The downloader (`refresh-snaps` CLI) does its own rate-limit handling against progettoSnaps's CDN.
- **Failure modes:** file missing → `url_for` returns `None`. No upstream errors.
- **Estimate:** ~2 days total — ~1 day for the CLI + downloader + pack discovery (uses existing `downloads.py` infrastructure from P07 ✅); ~1 day for the source + disk-cache wiring + UI surface for the download trigger.

### Orchestrator amendment — `file://` URL handling

To support the pack-based source, `resolve_image`'s pseudocode is amended:

```python
async def resolve_image(...):
    for source in registry.chain_for(kind):
        # ... prepare() block as before ...
        url = source.url_for(machine, kind)
        if url is None:
            continue
        if url.startswith("file://"):
            # Local-pack source (P10 ProgettoSnapsSource). Serve direct
            # — no cache lookup, no network call. `fetch_with_cache`'s
            # _ALLOWED_URL_SCHEMES intentionally excludes file:// for
            # security (P05 cache.py:33 FP27 B4 guard), so we MUST
            # short-circuit here before reaching it.
            local = Path(urllib.parse.urlparse(url).path)
            if local.exists():
                return local
            continue  # pack file gone between url_for + return; fall through
        # ... fetch_with_cache block as before ...
```

The `file://` short-circuit honors the existing P05 security guard: `fetch_with_cache` rejects non-http(s) schemes with `MediaFetchError` (`cache.py:85-86`), which would 502 the route. The orchestrator's pre-check prevents that.

### `mame-curator refresh-snaps` CLI (new in P10)

```bash
uv run mame-curator refresh-snaps [--dest data/snaps/] [--kinds flyers,titles,snap] [--force]
```

- **Discovery step:** scrape `https://www.progettosnaps.net/` (or hit a known pack-index URL) to find current pack download URLs per kind. **Open Verification Item:** pin the exact discovery method during chunk 3 fixture capture (see § Open verification items).
- **Download step:** uses the existing `mame_curator.updates.downloads` infrastructure (P07 ✅) — atomic write, checksum verification if upstream publishes one, retries via mirror list if available.
- **Extraction step:** ZIP unzip into `data/snaps/{flyers,titles,snap}/` flat layout. Existing files are overwritten only with `--force`; otherwise the CLI prints a summary of "would-replace" files and exits.
- **Progress reporting:** rich-console progress bar for the download phase; per-file count for the extraction phase.
- **Disk space check:** before download, verify `data/` has at least 2× the pack's stated content-length free; abort with a clear error if not.
- **Modes 0644 for files, 0755 for directories** — standard, no special permissions.

### 2. ArcadeDB

- **`name`:** `"arcadeDB"`
- **`license_compatible`:** `True` (CC-BY 3.0 per the project's own statement).
- **`kinds`:** `{"boxart", "title", "snap"}`
- **Endpoint:** `http://adb.arcadeitalia.net/service_scraper.php?ajax=query_mame&game_name={name}`. Returns JSON; we extract the `url_image_ingame`, `url_image_title`, `url_image_flyer` fields.
- **Two-step lookup** (implemented inside `ArcadeDBSource.prepare(machine, client)`); shapes verified at pre-impl prep against `tests/fixtures/arcadedb_pacman.json`:
  1. Acquire one token from the per-source `TokenBucket`; on empty bucket → `MediaRateLimited`.
  2. Build the scraper URL (`http://adb.arcadeitalia.net/service_scraper.php?ajax=query_mame&game_name={machine.name}`) — note: returns **HTTP 301 → HTTPS**; the lifespan client's `follow_redirects=True` (FP10 A1) handles this transparently. Hit it once per machine via `fetch_text_with_cache(url, cache_dir, client=client)`. **Same cache function, same SHA-256-of-URL cache key as image fetches** — no special prefix or alternate keying needed; URLs differ → hashes differ → no collision.
  3. **Parse-before-trust** (cache-poisoning guard): `json.loads` the returned text. If parse fails (truncated TLS body, mid-deploy HTML-instead-of-JSON, etc.), `Path(cache_file).unlink(missing_ok=True)` to invalidate the bad slot, then raise `MediaFetchError(...)` chained from the `JSONDecodeError`. The next request re-fetches; transient bad responses don't permanently disable the source for this machine.
  4. On valid JSON: the top-level shape is `{"release": <int count>, "result": [...]}` — **the array is `result`, not `release`** (the latter is the integer match count). If `result` is non-empty, stash `_url_cache[machine.name] = {"boxart": ..., "title": ..., "snap": ...}` from `result[0]`'s `url_image_flyer` / `url_image_title` / `url_image_ingame` fields. These URLs are **redirector-form** (`https://adb.arcadeitalia.net/?mame=pacman&type=flyer&resize=0`) — `fetch_with_cache` follows the redirect to the actual image. Empty `result` array → leave `_url_cache[machine.name]` absent (uniform negative-cache shape — see MobyGames below).
  5. `url_for(machine, kind)` reads from `self._url_cache.get(machine.name, {}).get(kind)`; missing entry → returns `None`.
- **Rate limit:** the public service has no documented hard cap, but the project is a hobby site. Token bucket: 30 req/min default, configurable via `media.arcadedb_rate_limit_per_min` (new field, optional, defaults to 30).
- **Auth:** none.
- **Failure modes:** all handled by step 3 above (parse-before-trust + cache-file invalidation). Network errors propagate as `MediaFetchError` from `fetch_text_with_cache` (already enumerated in the unhappy-paths table).
- **Estimate:** ~2 days (two-step lookup, JSON parse, rate-limit, integration tests with `respx`).

### 3. Wikipedia (image)

- **`name`:** `"wikipediaImage"`
- **`license_compatible`:** depends per image (Wikipedia hosts mixed licenses — public-domain screenshots, CC-BY screenshots, fair-use flyers). P10 sets `license_compatible = False` conservatively; the P11 contribute-back flow will need per-image license inspection if it ever wants to consume this source. P10 itself only displays the image.
- **`kinds`:** `{"boxart"}` only. The infobox image is the only field with a consistent, documented location in the Wikipedia REST summary response (`thumbnail.source`). Title screens and snap shots aren't reliably present and would require HTML scraping or guessing — out of scope. Counterexample: a source that returned the infobox image for every `kind` would silently degrade the user's chain (the wrong-shaped image would win over a legit downstream candidate); the `frozenset({"boxart"})` shape prevents this at registry-build time.
- **Endpoint:** [Wikipedia REST API summary](https://en.wikipedia.org/api/rest_v1/page/summary/{title}). The `thumbnail.source` field is the image URL.
- **Title resolution:** `machine.description` passes through a small canonicalization step before hitting the Wikipedia REST URL — arcade DAT descriptions typically read `"Pac-Man (Midway)"` or `"Street Fighter II: The World Warrior"`, and the Wikipedia summary endpoint expects the canonical article title.
  1. Strip trailing parenthesised qualifiers: `"Pac-Man (Midway)"` → `"Pac-Man"`. Pattern: `re.sub(r"\s*\([^)]*\)\s*$", "", description).strip()`.
  2. Trim leading/trailing whitespace.
  3. URL-quote the result via `urllib.parse.quote`.
  No fuzzy match, no second attempt. If the canonicalised title 404s, fall through. Many arcade entries (regional ports, unofficial conversions, prototypes) won't hit Wikipedia even after canonicalisation — accept this as the long-tail. The source's value is the head of the catalog (Pac-Man, Tetris, Donkey Kong, etc.), not full coverage.
- **Rate limit:** Wikipedia tolerates high request rates from clients sending a descriptive `User-Agent` (per the [API:Etiquette](https://www.mediawiki.org/wiki/API:Etiquette) page). Token bucket at 60 req/min as a courtesy cap — well under any plausible upstream limit.
- **User-Agent:** set globally on the lifespan client per Wikipedia's API etiquette: `User-Agent: mame-curator/{VERSION} (+https://github.com/milnet01/mame-curator)`. **Implementation note:** P05's lifespan client lives at `src/mame_curator/api/app.py:130` (look for the FP09-B4 comment block at lines 118–129) — `httpx.AsyncClient(timeout=10.0, follow_redirects=True)`. P10 adds `headers={"User-Agent": _build_user_agent()}` to the constructor; `_build_user_agent()` is defined in `media/__init__.py` so the helper lives next to the consumer, not in the API package.
- **Failure modes:** 404, 5xx → fall through.
- **Estimate:** ~1 day.

### 4. MobyGames

- **`name`:** `"mobyGames"`
- **`license_compatible`:** `False` (MobyGames API terms restrict redistribution; in-app display is fine, contribute-back is not).
- **`kinds`:** `{"boxart"}` — port covers only. MobyGames lists screenshots too, but their per-game pagination is heavy and the catalog overlap with arcade specifically is narrow. P10 keeps the source thin; expanding to title/snap is a future fix-pass if usage warrants it.
- **Endpoint:** `https://api.mobygames.com/v1/games?title={url-quoted machine.description}&api_key={key}`. Picks the first game's `cover_url`. Two-step lookup like ArcadeDB; JSON-cached.
- **API key:** required. Loaded in this order:
  1. **`MOBYGAMES_API_KEY` environment variable** — primary path; nothing to install, nothing to configure beyond a shell export.
  2. **`data/secrets/mobygames.key`** — single-line file at the path resolved as **`CWD / "data" / "secrets" / "mobygames.key"`**, matching `MediaConfig.cache_dir`'s relative-to-CWD convention (`api/schemas.py:81` ships `cache_dir: Path = Path("./data/media-cache")`). Mode 0600 enforced at load via `stat().st_mode & 0o077 == 0` (refuses if any group or other bit is set). `data/secrets/` is a new directory P10 creates on first run; this is the first user, no pre-existing convention. Fallback for users who'd rather not edit shell config.
  - **Not in `config.yaml`.** API keys never enter the YAML config, never round-trip through the API surface, never appear in snapshots / exports. The MobyGames source is *enabled* via `media.sources` but only *works* when the key resolves at startup. Missing key → source's private `_enabled` flag stays `False`, one-time WARNING at construction, both `prepare` and `url_for` short-circuit to no-op.
  - `keyring` integration is **deferred to P11** (rationale + canonical statement in § "Out of scope (deferred)" — P11 already needs `keyring` for its GitHub PAT path, so folding the MobyGames key under the same umbrella avoids two parallel secret stores). P10 ships env-var + dotfile only.
- **Rate limit:** MobyGames documents 360 req/hr for free accounts; token bucket at 5 req/min (60% margin). Configurable via `media.mobygames_rate_limit_per_min`.
- **Failure modes:**
  - 401 / 403 → source flips `_enabled = False` for the rest of the process lifetime (key is wrong; retrying won't help). One-time WARNING. Subsequent `prepare` / `url_for` calls short-circuit immediately.
  - 429 → `MediaRateLimited`; orchestrator falls through.
  - 404 → `fetch_text_with_cache` returns `None`; `prepare` leaves `_url_cache[machine.name]` absent; `url_for` returns `None`. No synthetic cache body.
  - 200 with empty `games` array (i.e. the title produced zero matches) → cached upstream body replays on next `fetch_text_with_cache` call without an upstream hit; `prepare`'s parse step sees an empty list and leaves `_url_cache[machine.name]` absent. Same uniform negative-cache shape as ArcadeDB (cached real body, missing dict entry).
  - JSON-parse failure → same parse-before-trust / unlink / re-raise pattern as ArcadeDB step 3.
- **Estimate:** ~2 days.

**Ordering semantics:** the `sources` tuple is the **fallback order**. Sources not in the tuple are *disabled* (no chain entry); unknown names log a one-time WARNING and are skipped.

**Defaults rationale:**
- libretro first — fastest (raw GitHub, no rate limit, the project's existing baseline).
- progettoSnaps second — closes ~60–70% of libretro's gaps with no auth.
- ArcadeDB third — high-quality but rate-limited; only hit on miss.
- wikipediaImage fourth — mediocre quality but covers oddball/non-arcade entries.
- mobyGames last — needs a key; most users won't have one; placing it last means it costs nothing when disabled.

**Backwards compatibility:** P10 is pre-v1.0.0; per CLAUDE.md "No backwards-compat shims pre-v1.0.0", config files written before P10 will lack `media.sources` and get the default tuple. No migration needed.

- Snapshots and `/api/config/export` already exclude `data/secrets/` (already true; no new exclusion logic needed — paths under `data/` other than the explicit allow-list aren't included).
- Whether to require an additional auth gate (e.g. a token printed at server startup) — flagged in § Open Verification Items as a question for the lifespan of P10 vs deferral.

### Frontend hook

```ts
// frontend/src/hooks/useMediaSources.ts
export function useMediaSources() {
  return useQuery({
    queryKey: ['mediaSources'],
    queryFn: () => api.get<SourceReadiness>('/api/media/sources'),
    staleTime: 5_000,   // brief — readiness can flip mid-process after PUT
  })
}

export function useSaveSourceSecret() {
  const qc = useQueryClient()
  return useMutation({
    mutationFn: ({ name, secret }: { name: string; secret: string }) =>
      api.put(`/api/media/sources/${name}/secret`, { secret }),
    onSuccess: () => qc.invalidateQueries({ queryKey: ['mediaSources'] }),
  })
}
```

`staleTime: 5_000` (5 seconds) because the readiness state can flip server-side mid-process (after a 401 from MobyGames, or after a successful `PUT /secret` from another tab). Long-cache would lie to the user.

## Architecture notes

- **Protocol over inheritance.** `MediaSource` is a `Protocol` so the five concrete implementations can stay simple — three `ClassVar`s and two methods (`prepare`, `url_for`). No base class, no `super()`, no MRO surprises. The orchestrator only depends on the protocol surface.
- **Two-phase per-source dispatch: async `prepare` then sync `url_for`.** Sources that need a lookup (ArcadeDB, MobyGames) populate their internal cache inside `prepare(machine, *, client)`; single-shot sources leave `prepare` as a no-op. `url_for` stays sync and deterministic; `resolve_image` awaits `prepare` once per `(source, machine)` before iterating kinds.
- **No async lock around the per-source lookup cache.** Concurrent `prepare(machine)` calls from two parallel requests for the same machine are safe at the cache layer (`fetch_text_with_cache` uses the same `.tmp + os.replace` discipline as P05's binary path; both writers race-write to a unique `.tmp` and the second `os.replace` wins atomically). At single-user scale, the wasted second JSON parse is acceptable; pre-warming an `asyncio.Lock` per source × machine doesn't pay off — same trade-off the inline concurrency-note paragraph in P05 spec (P05.md:105) already pinned for the binary cache.
- **Per-source rate-limiter as a constructor injection.** `ArcadeDBSource(limiter=TokenBucket(rate=30/60))`. App-level `state.arcadedb_limiter` is constructed once at lifespan startup and shared across all per-request source instances — the limiter holds the token-bucket state that *must* persist process-wide, even as sources are recreated per request. Construction-time injection (vs. e.g. `request.app.state.arcadedb_limiter` lookups inside `prepare`) lets tests substitute a fake limiter without monkey-patching app state. Same model for Wikipedia (`state.wikipedia_limiter`) and MobyGames (`state.mobygames_limiter`).
- **The source registry is per-request, not per-process.** It's built from `world.config.media.sources` on each `resolve_image` call. This means a `PATCH /api/config` that changes the source order takes effect on the next request without a server restart — matches P05's "read config on every render" pattern. **Two consequences worth pinning:** (1) each source instance's `_url_cache` is dropped at request end — `prepare` re-runs on the next request, but the on-disk `fetch_text_with_cache` body cache catches it without an upstream hit; the in-memory dict is just a per-request parsed view. (2) Process-wide state (token buckets, the disabled-flag on MobyGames after a 401) lives on the *limiter* / *app state*, not on the per-request source — same separation as Django's "request lifecycle" pattern.
- **No async lock in the binary image cache.** Inherited from P05.
- **The Wikipedia extract is text, not an image, so it lives next to image fetching but doesn't share the binary cache path.** `fetch_text_with_cache` is the sibling. Same `.tmp + os.replace` discipline, different cache-key namespace (different URLs), different `max_bytes` default (`DEFAULT_TEXT_MAX_BYTES = 256 KiB`).

## Anti-jump compliance

- **No cross-package imports** from `api/`, `filter/`, `copy/`, or `updates/` (the four sibling top-level packages under `src/mame_curator/`). Intra-`media/` imports (e.g. `from mame_curator.media.cache import fetch_with_cache`) are unconstrained — that's the same package. `parser.Machine` and `_atomic.atomic_write_bytes` are the only out-of-package symbols `media/` reaches for, matching P05's shape.
