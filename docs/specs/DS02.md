## Why a single fix-pass and not seven

Each cluster is small in isolation but they all share one risk:
**they touch the same files multiple times.** App.tsx is touched by Cluster B
(strings), Cluster C (skip-to-main + main-landmark aria-label), and Cluster E
(ErrorBoundary nesting). LibraryPage.tsx is touched by Cluster C (sidebar/cart
aria-labels) and Cluster E (CopyModal + AlternativesDrawer). Cluster D
rewires SettingsPage.tsx. Cluster G touches `filter/heuristics.py` and
`api/state.py`. Seven separate fix-passes would mean seven rounds of
import-graph churn and seven closing audits each re-discovering the next
cluster's drift. One pass with internal cluster discipline keeps each
cluster's diff focused while letting the closing review check the whole
re-layout in a single pass.

### Cluster G — Perf micro-fixes (2 sub-items)

Two perf opportunities from the source review that the first verification
pass wrongly dismissed:

- **G1** — Memoize `revision_key_of` (`src/mame_curator/filter/heuristics.py:60`).
  Called twice per pair in the `_cmp_revision` comparator
  (`src/mame_curator/filter/picker.py:119`) — for a candidate group of N
  machines, the sort drives O(N log N) comparisons, each redoing the
  regex-and-tuple work on the same `description` strings. Wrap with
  `@functools.lru_cache(maxsize=8192)`. Description strings are short and
  hashable; max cardinality is bounded by the DAT (~43k machines), so an
  8 K cache covers any realistic candidate-group sort without unbounded
  growth. **Verify the wrapper does not introduce a per-process global**
  that breaks test isolation — if it does, drop to a per-`_compose_cmp`
  dict instead. (Step 4 picks based on whether tests stay green.)
- **G2** — Precompute `bytes_by_machine: Mapping[str, int]` on `WorldState`
  (`src/mame_curator/api/state.py:48`). Currently every
  `GET /api/games` and `GET /api/games/cart` recomputes
  `sum(sum(r.size or 0 for r in world.machines[s].roms) for s in filtered)`
  at `api/routes/games.py:144` and again at `:374,384` (two distinct call
  sites). The work is **O(M × R)** (M machines × R ROMs/machine) per
  request and is fully deterministic given the parsed DAT. Move it to a
  one-time computation at WorldState construction
  (`api/state.py:137,204`): walk each `Machine`'s ROMs once, store the sum
  per machine name. Per-request work drops to **O(|filtered|)**: just
  `sum(world.bytes_by_machine.get(s, 0) for s in filtered)`. No behaviour
  change — same byte total surfaces to the API. Treat `bytes_by_machine`
  as part of WorldState's immutable post-construction state (matching the
  `frozen=True` invariant already used for `Machine`).

## Architecture notes

DS02 does **not** change any public API surface:

- **`api/schemas.py` split (A5)** is a pure file move — the `__init__.py`
  re-export facade preserves `from mame_curator.api.schemas import Foo`
  call shape. No caller updates required.
- **`cli/__init__.py` split (A2)** is a pure file move — `set_defaults(func=...)`
  dispatch shape preserved; `mame-curator <subcommand>` CLI surface
  unchanged.
- **`copy/runner.py` extraction (A4)** introduces one module-private helper
  (`_resolve_conflicts`); `run_copy()` signature unchanged.
- **Frontend `types.ts` / `strings.ts` splits (A1, A3)** preserve re-export
  shape from the original module path. Existing imports survive without
  edit.

The Settings-tab `useSearchParams` wiring (D1) is the only behavioural
change beyond a11y: a deep-linked URL now restores the active tab on
reload. The change preserves the default-tab behaviour when the param is
absent.

The CHANGELOG / package-version drift fixes (F1, F2) are documentation +
metadata only. No code surface touched.

The ErrorBoundary wraps (E1, E2) widen the existing LibraryPage error
boundary's domain to cover two more modal subtrees. A render error in
those modals previously crashed the page; now it renders the boundary's
fallback UI. Net user-visible change: a degraded but functional library
page instead of a white screen, only on the rare render-error path.

Cluster G is the only cluster with measurable runtime impact:

- **G1 (`revision_key_of` memoization)** is invisible at the API boundary
  (same return values) — only the per-comparison work drops from
  `regex.match + parse + tuple` to a dict lookup. Largest gain shows on
  large multi-revision candidate groups during `_apply_picker`.
- **G2 (`bytes_by_machine` precompute)** changes the cost of
  `GET /api/games` (and the cart endpoint) from
  ~O(43k machines × ~10 ROMs each) per request to ~O(|filtered|) per
  request. The one-time WorldState construction grows by the same amount,
  but WorldState is built once per parse — a single up-front O(M × R)
  walk amortised across every subsequent API call.

Neither G1 nor G2 changes any public type, response shape, or error code.
Both are pure local-perf refactors with regression-lock tests.
