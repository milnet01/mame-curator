## Deliberately not in scope

- **Major-version breaking bumps that touch public API.** Any bump where the
  upstream changelog flags a breaking change to an API surface the project
  uses (e.g. pydantic v3, fastapi 1.0, react 20, vite 9) opens a separate
  per-dep phase. DS03 ships only bumps that pass `uv sync` + `npm ci` + the
  full test matrix without code edits beyond version-string changes.
  Deferred breakers are logged as conditional `P##` follow-ups at Step 9.

  **R1 amendment (2026-05-16):** mypy 2.x was originally enumerated above as
  a deferred breaker. Cluster F's `uv lock --upgrade` pulled the upgrade
  forward (the `>=1.13` constraint had no upper bound), and `uv run mypy`
  returned clean on all 175 source files with `strict = true` unchanged.
  The opportunistic adoption was accepted under the spec's own non-breaking
  test ("passes the full test matrix without code edits"). The mypy 2.0
  default-shifts (`--strict-bytes`, `--local-partial-types`) are documented
  inline in `pyproject.toml [tool.mypy]` so future contributors know which
  defaults the strict config inherits. Other breakers (pydantic v3, fastapi
  1.0, react 20, vite 9, mypy 3.x, etc.) still defer when they ship.
- **Promoting optional → runtime deps.** Coding-standards § 8 "Phase-staged
  dependency declaration" forbids promoting `[project.optional-dependencies]`
  entries into `[project.dependencies]` as a side-effect of a version bump.
  Promotion happens in the commit that adds the first `import`, never here.
- **Python language-version bump.** `requires-python = ">=3.12"` stays as-is;
  the CI matrix runs 3.12 + 3.13 (already current). A Python 3.14 add-on is
  a future post-v1 item, not DS03 scope.
- **Adopting new tool families.** Switching mypy → Astral's Ty (per § 8
  comment), swapping npm → pnpm, or moving off Vite stays out. DS03 is a
  *freshness* sweep, not a tooling swap.
- **Coding-standards § 8 amendments.** The "Pin to most recent stable" rule
  is what DS03 enforces, not amends. If the sweep reveals the rule needs
  refinement (e.g. an explicit cadence or an allowlist for known-pinned
  deps), surface it for a future § 16 amendment — not a same-phase edit.

## Why a single bundled sweep

Each bump in isolation is small (~1 LOC per pin). Together they share three
costs the close-phase reviewer would re-discover N times if shipped as N
separate phases:

1. **CI matrix cost.** The CI matrix is `[ubuntu, macos, windows] ×
   [py3.12, py3.13]` = 6 runs per push, plus the `gitleaks` lane = 7. Six
   separate bump commits cost ~42 CI runs; one coordinated cluster sequence
   re-uses cached `uv` venvs and amortises the matrix across 5–6 commits in
   one push.
2. **Cross-cluster ruff / mypy / bandit / gitleaks couplings.** Each of those
   tools appears in BOTH `pyproject.toml` (dev-dep constraint) AND
   `.pre-commit-config.yaml` (hook rev). Independent bumps drift; a
   coordinated bump keeps the pair in lockstep, and the Step 3 docs-test
   makes the invariant CI-enforceable thereafter.
3. **Lockfile churn.** Splitting bumps across commits forces N
   `uv.lock` + `package-lock.json` regens, each touching ~50–800 lines and
   each prone to merge conflicts. One coordinated bump = one lockfile churn
   per ecosystem (Python + frontend).

## Scope

### Cluster A — Python runtime + dev deps (`pyproject.toml`)

**Methodology (run at Step 4):**

1. From a fresh `uv sync --extra dev`, run `uv pip list --outdated --format
   json` and parse the `name` / `version` / `latest_version` fields.
2. Filter out pre-releases (`info.version` strings matching
   `(?i)(a|b|rc|dev|preview)\d`).
3. For each candidate bump, read the upstream CHANGELOG (PyPI `info.project_urls`
   → Source / Homepage → `CHANGELOG.md` or `releases/`) for the
   source → target span. If the diff flags a breaking change to a project-used
   API, skip and log to the conditional-follow-ups list at Step 9.
4. Bump the constraint in `pyproject.toml` to `>=<latest stable>`.
5. Lock with `uv lock --upgrade-package <name>` per bump (NOT a wholesale
   `uv lock --upgrade` — keeps transitive bumps inspectable in the diff).

**A1 — runtime deps** (8 entries: `lxml`, `pydantic`, `pyyaml`, `rich`,
`fastapi`, `uvicorn`, `httpx`, `sse-starlette`).

**A2 — dev deps** (10 entries: `pytest`, `pytest-cov`, `pytest-asyncio`,
`hypothesis`, `respx`, `ruff`, `mypy`, `bandit`, `pre-commit`,
`types-PyYAML`).

A1 + A2 ship in **one commit** on `pyproject.toml` + `uv.lock` unless Step 4
discovers a breaking change in one tier that needs separate triage — in
which case the unaffected tier ships first and the breaker is deferred per
the methodology step 3.

**Reuse before rewriting (rule 3):** Cluster A doesn't introduce new
helper files. The methodology's `uv pip list --outdated` call is shell-only;
the per-dep changelog read is manual review at the commit-message level.

### Cluster B — Frontend runtime + dev deps (`frontend/package.json`)

**Methodology (run at Step 4):**

1. From a fresh `npm ci`, run `npm outdated --json` and parse each entry's
   `current` / `latest` fields.
2. `npm outdated` returns the `latest` tag from the registry, which excludes
   pre-releases tagged `next` / `beta` / `alpha` (npm dist-tag semantics).
3. For each candidate bump, read the upstream CHANGELOG (registry
   `repository.url` → GitHub releases). Breaking changes defer.
4. Bump the constraint range in `frontend/package.json` (preserve `^` / `~`
   prefix; the `~` on `typescript` is intentional per the dev-tooling pinning
   convention).
5. Regenerate `package-lock.json` via `npm install` (no `--save`, manifest
   is authoritative).

**B1 — React + Tailwind v4 + Vite core** (10 entries): `react`, `react-dom`,
`@types/react`, `@types/react-dom`, `@vitejs/plugin-react`, `tailwindcss`,
`@tailwindcss/vite`, `vite`, `vitest`, `@vitest/coverage-v8`.

**B2 — TanStack + UI primitives + state** (14 entries):
`@tanstack/react-query`, `@tanstack/react-virtual`, `@radix-ui/react-slot`,
`radix-ui`, `cmdk`, `framer-motion`, `lucide-react`, `sonner`,
`tailwind-merge`, `class-variance-authority`, `clsx`, `dompurify`, `zod`,
`react-router`.

**B3 — Test + lint tooling** (13 entries): `@playwright/test`,
`@testing-library/{jest-dom, react, user-event}`, `eslint`,
`eslint-plugin-react-hooks`, `eslint-plugin-react-refresh`,
`typescript-eslint`, `@eslint/js`, `msw`, `jsdom`, `prettier`, `typescript`,
`@types/node`, `globals`.

B1/B2/B3 ship as **one commit** on `package.json` + `package-lock.json`
unless a tier breaks (same rule as Cluster A).

### Cluster C — GitHub Actions pins (`.github/workflows/{ci,release}.yml`)

**Methodology (run at Step 4):**

1. For each `uses: <owner>/<repo>@<version>` line, fetch latest via
   `gh api /repos/<owner>/<repo>/releases/latest --jq .tag_name`.
2. Prefer the major-version floating tag (`@v6`) over a pinned minor
   (`@v6.1.2`) — matches the existing `checkout@v6` / `upload-artifact@v4`
   pattern. The exception is `astral-sh/setup-uv@v8.1.0`, pinned to a
   specific minor; check whether the project intentionally pinned (search
   commit history + comments) before floating to `@v8` or bumping to the
   latest minor.
3. Bump major-floating tags to the latest major; bump minor-pinned entries
   to the latest minor of the same major. Only cross major versions when
   the upstream major-bump notes flag NO breaking change to the action's
   input/output schema (rare for actions/* but common for setup-* actions).

Entries (6): `actions/checkout`, `astral-sh/setup-uv`,
`actions/upload-artifact`, `actions/download-artifact`,
`softprops/action-gh-release`, and the embedded `GITLEAKS_VERSION` env var
in both `ci.yml` and `release.yml` (must be bumped together — see Cluster D).

### Cluster D — Pre-commit hook revs (`.pre-commit-config.yaml`)

**Methodology (run at Step 4):**

1. For each `repo: <url>`, fetch the latest tag via
   `git ls-remote --tags --refs <url> | grep -v -E '(a|b|rc|dev)' | tail -1`.
2. Bump `rev:` to that tag.
3. **Cross-pin coupling check** — for each repo whose tool also appears in
   `pyproject.toml`, the pre-commit rev MUST resolve to the same locked
   version `uv.lock` records:
   - `astral-sh/ruff-pre-commit` rev ↔ `uv.lock` `ruff` version
   - `pre-commit/mirrors-mypy` rev ↔ `uv.lock` `mypy` version
   - `PyCQA/bandit` rev ↔ `uv.lock` `bandit` version
   The Step 3 docs-test (below) makes this invariant CI-enforceable.
4. **gitleaks-binary coupling** — `gitleaks/gitleaks` rev MUST equal the
   `GITLEAKS_VERSION` env var in `ci.yml` + `release.yml` (Cluster C
   touches the env var; Cluster D's commit lands second to close the loop).

Entries (5): `pre-commit-hooks`, `ruff-pre-commit`, `mirrors-mypy`,
`bandit`, `gitleaks`.

### Cluster F — Lockfile refresh (transitive bumps + final verification)

**Two responsibilities:**

(i) **Transitive bumps not surfaced by Clusters A-B.** Items where the
manifest constraint already permits the latest stable, but `uv.lock` /
`package-lock.json` lag transitively. Run:

- `uv lock --upgrade` (Python — refreshes every transitive entry in the
  lock to the newest version satisfying the manifest constraints).
- `npm update` (frontend — same effect on `package-lock.json`).

This catches deps the project doesn't import directly but ships in the
resolved tree (e.g. `starlette` under `fastapi`, `tomli` under `uv`).
Per coding-standards § 8 "Pin to the most recent stable" applies
transitively, not just to top-level constraints.

(ii) **Verification stage after Clusters A-E land.** Re-run the same two
commands; the expected diff is empty (Clusters A-B already absorbed
every direct bump). If a non-empty diff surfaces, fold it into the
relevant cluster's commit (amend before push) and re-run the full gate.

**Cluster F commits separately from A/B** so the diff between
"direct-pin bumps" (A/B) and "transitive-only refresh" (F) stays
inspectable in `git log -p`. Reviewers can scan F's lockfile churn
without cross-referencing the manifest changes.

### Cluster G — Frontend CI gate (`.github/workflows/ci.yml`)

**Why this is folded into DS03 (not punted):** Cluster B bumps every
frontend dep to latest stable, but `ci.yml` has no `actions/setup-node`,
no `npm ci`, and no frontend test invocation — the dep bumps would land
with zero CI gating, and any regression would surface only on the next
developer's local pre-commit. Adding the frontend lane makes the
freshness sweep actually complete for the frontend stack.

**Methodology (run at Step 4):**

1. Add a new job `frontend-lint-types-test` to `ci.yml`, parallel to
   the existing `lint-types-test` (backend) and `gitleaks` jobs:
   - `runs-on: ubuntu-latest` (no OS matrix — frontend tests don't
     need it, and the backend matrix already exercises Windows/macOS
     for cross-platform shell quirks; the frontend pipeline is
     OS-agnostic and Linux-only is faster + cheaper).
   - `actions/setup-node@v4` with `node-version` reading from
     `frontend/package.json`'s `engines.node` (use the `node-version-
     file` input pointed at `frontend/package.json` — single source of
     truth, no drift risk).
   - `working-directory: frontend` block-scoped via the job's
     `defaults.run.working-directory`.
   - Steps: `npm ci` → `npm run lint` → `npx tsc --noEmit` →
     `npm test`. NOT `npm run test:coverage` (the local gate runs that;
     CI just needs pass/fail).
2. Add the same job to `release.yml` (the release pipeline already
   re-runs the full CI gate at the tagged commit — the frontend lane
   joins that re-run).
3. Verify the existing `concurrency` block in `ci.yml` covers the new
   job (it does — `group: ${{ github.workflow }}-${{ github.ref }}`
   applies to every job in the workflow).

**Out of scope inside Cluster G:**

- **Frontend coverage upload.** The existing `Upload coverage` step in
  `lint-types-test` uploads Python coverage only. A matching frontend
  coverage upload is a separate add (it'd need the `vitest --coverage`
  invocation + an `actions/upload-artifact` step), defer to a follow-up.
- **e2e Playwright lane.** Playwright tests live in
  `frontend/playwright.config.ts` and run via `npm run e2e`. They need
  browser binaries (`npx playwright install --with-deps`), which is a
  multi-minute CI step. Defer to a follow-up phase — Cluster G is
  unit + lint + types only.

## Tests to write first (Step 3)

Dep bumps are not TDD-driver-shaped — the failing test isn't "current pin
is wrong." What CAN be RED-then-GREEN is the **cross-pin drift detector**,
so the DS02-R2-shape "CI catches what local missed" lesson doesn't recur
for dep pins. Two new docs-tests:

### `tests/docs/test_dep_pin_coupling.py`

1. Parse `uv.lock` for the resolved versions of `ruff`, `mypy`, `bandit`.
2. Parse `.pre-commit-config.yaml` for the `astral-sh/ruff-pre-commit`,
   `pre-commit/mirrors-mypy`, and `PyCQA/bandit` hook revs (strip leading
   `v` from each).
3. Assert each pair is byte-equal (e.g. `uv.lock` ruff `0.15.12` ↔
   pre-commit rev `v0.15.12`).
4. Parse `ci.yml` and `release.yml` for the `GITLEAKS_VERSION` env var
   (regex `GITLEAKS_VERSION:\s*"?([0-9.]+)"?`).
5. Parse `.pre-commit-config.yaml` for the `gitleaks/gitleaks` rev (strip
   leading `v`).
6. Assert all three values (ci.yml, release.yml, pre-commit) are byte-equal.

RED at HEAD (pre-commit `v8.21.0` ≠ CI `8.24.3`); GREEN after Cluster C
+ D's coupling fix lands.

### `tests/docs/test_no_pre_release_pins.py`

1. Walk every pin string in `pyproject.toml` (both `[project.dependencies]`
   and `[project.optional-dependencies].dev`), `frontend/package.json` (both
   `dependencies` and `devDependencies`), `.pre-commit-config.yaml`
   (`rev:` lines), and the `GITLEAKS_VERSION` env vars.
2. Assert no string matches `(?i)(alpha|beta|rc|preview|next|dev)(\d|$)`
   (PEP 440 + npm pre-release semantics).
3. Anchors `coding-standards.md § 8` "No lock-pinning to bleeding-edge
   zero-x releases unless essential" — makes the rule
   CI-enforceable instead of aspirational.

GREEN at HEAD (no current pin uses a pre-release suffix); stays GREEN
through DS03 if the methodology's "filter pre-releases" step holds.
Documents-as-test pattern matches `tests/docs/test_file_size_caps.py` from
DS05 — same `tests/docs/` directory, same pytest collection scope.

## Architecture notes

DS03 changes nothing user-visible:

- **No production code changes.** Only manifest files (`pyproject.toml`,
  `frontend/package.json`), lockfiles (`uv.lock`, `package-lock.json`), CI
  workflow YAML, pre-commit config, and the two new docs-tests.
- **No public API surface change.** Pydantic model shapes, FastAPI route
  signatures, React component props, CLI surface, and the spec contracts
  all stay byte-identical. The DS02 `check-api-types-sync` pre-commit gate
  (DS05 Cluster D) fires unchanged — its failure mode is real API drift,
  not dep bumps.
- **No coding-standards changes.** § 8 "Pin to most recent stable" is the
  rule being honored, not amended.

**Commit cadence** (one cluster per commit; final tag at end):

1. Cluster A — Python pins + `uv.lock` regen.
2. Cluster B — frontend pins + `package-lock.json` regen.
3. Cluster E — `engines.node` bump (lands BEFORE Cluster G so the new
   CI job reads the bumped LTS, not the old `20.x`).
4. Cluster C — GitHub Actions pins + `GITLEAKS_VERSION` bump in both
   `ci.yml` and `release.yml`.
5. Cluster G — new `frontend-lint-types-test` CI job (depends on
   Cluster C's actions/setup-node@v4 reference resolving to its latest
   stable; lands after C so the new job uses the freshest action pin).
6. Cluster D — pre-commit revs + cross-pin coupling fix (lands AFTER
   C/G so the `GITLEAKS_VERSION` ↔ pre-commit gitleaks coupling closes
   monotonically — never gets worse mid-sweep). Step 3 docs-tests
   (`test_dep_pin_coupling.py` + `test_no_pre_release_pins.py`)
   committed alongside Cluster D (RED before D's coupling-fix lands,
   GREEN after).
7. Cluster F — `uv lock --upgrade` + `npm update` transitive refresh.
   Usually small; one commit unless the transitive bumps cluster into
   identifiable groups.
8. Cluster H — `pnpm` → `npm` spec-text correction (DS02.md +
   DS05.md). Independent of every other cluster; commits last so it
   doesn't entangle with the gate-touching changes.

Matches DS05's "one commit per cluster" cadence and lets the closing audit
see each stage discretely. Steps 3 docs-tests land at #6, not earlier,
because their RED-at-HEAD state is the gitleaks coupling drift that
Cluster D fixes — committing them earlier would block intermediate
commits on a failing test.
