## Scope

### Concurrency (A1–A3, 3)

- **A1 — `JobManager._emit` lock gap: choose narrow + document.** The class is documented loop-thread-only at `api/jobs.py:283`. The hardening choice is between (i) installing a `threading.Lock` / `asyncio.Lock` around the mutation block and (ii) tightening the invariant so non-loop-thread entries are impossible. Option (ii) is cheaper and aligns with FastAPI's single-event-loop model. **Current-call-site sanity check:** the worker-completion path at `api/jobs.py:182-191` marshals back via `self._loop.call_soon_threadsafe(...)` (both the error-path call near L187 and the success-path call near L191) — so today's call sites all enter `_emit` on the loop thread. The guard hardens against a future regression rather than a current bug. **Fix:** keep `_emit` lock-free but add a `_check_loop_thread()` guard at the top — `running_loop = asyncio.get_running_loop()` then `if running_loop is not self._loop: raise RuntimeError("JobManager._emit must run on the manager's loop thread")`. Use `raise RuntimeError(...)`, not `assert`, so `python -O` (optimised builds) cannot strip the guard. **`self._loop` is already an attribute** at `api/jobs.py:154` (initialised to `None` in `__init__`) and assigned inside `start()` at `api/jobs.py:166` (under `self._lock`). The fix is to **move the `asyncio.get_running_loop()` assignment up from `start()` to `__init__`** so the comparator is non-None before any `_emit` call. If `JobManager()` is ever constructed from outside a running loop (sync test code), `get_running_loop()` raises — fall back to **accepting the loop as a constructor parameter** (`def __init__(self, loop: AbstractEventLoop | None = None)`, defaulting to `asyncio.get_running_loop()` when omitted). The guard turns the latent race into a deterministic crash if the invariant is ever broken; the test (Test 1 below) pins the contract by calling `_emit` from a thread executor and asserting `RuntimeError`.
- **A2 — `recyclebin` counter race: choose serialize.** The counter walk + parent-dir mkdir is non-atomic. Two parallel `recycle_file` calls in the same session_id with the same basename can both pass `target_dir.exists()` at the same counter value. **Fix:** wrap the counter walk + `target_dir.mkdir` + manifest write + final `shutil.move` in a `filelock`-style file-system mutex keyed on `recycle_root / session_id / ".lock"`. The project doesn't currently depend on the `filelock` library (grep `pyproject.toml`); a tighter dependency-free fix uses `os.O_EXCL` for the lock file: `lock_path = recycle_root / f"{session_id}.lock"; fd = os.open(lock_path, os.O_CREAT|os.O_EXCL|os.O_WRONLY); try: ...critical section...; finally: os.close(fd); lock_path.unlink(missing_ok=True)`. With this lock, A3 below is solved by construction (only one session is in the critical section at a time, so the rollback's `target_dir.exists()` snapshot can't be invalidated). **Perf budget:** the recycle critical section is bounded by (a) a manifest atomic-write (~1 ms on tmpfs / SSD) plus (b) one `shutil.move`. **Same-filesystem rename** (the common case — source and recycle root on the same disk): `os.rename` is ~µs regardless of file size. Target **p99 < 50 ms for same-fs operations on 1 KB – 100 MB files**. **Cross-filesystem move** (rare — only when `data/recycle` is on a different mount than the source): falls back to byte-bound copy at the disk/network speed; a 100 MB cross-fs move at ~500 MB/s SATA is ~200 ms, at slower disks or network FS may reach seconds. Cross-fs case is bytes-bound, not lock-bound; the 50 ms p99 does not apply. **Orphan lockfile recovery:** the 60 s wall-clock staleness threshold targets the same-fs p99 (1200× headroom), which dominates real-world usage; cross-fs jobs exceeding 60 s would also exceed any sensible same-process lock-hold window, suggesting a crashed process. **Test:** spawn two threads each calling `recycle_file` on the same session_id + same basename; assert no `shutil.move` race + both files end up in serialized counter-walked targets.
- **A3 — `target_dir_existed` rollback race: solved by A2.** Two rollback paths share the race window: `recyclebin.py:86-88` (manifest-write failure) and `recyclebin.py:100-105` (post-`shutil.move` failure) — both `rmdir` `target_dir` if `target_dir_existed` was False. Once A2's lock serializes the critical section, both rollbacks become safe. **Fix:** no separate code change beyond A2's; add a comment at `recyclebin.py:63` crediting the lock for the invariant and naming both rollback sites. **Test:** the A2 test exercises both — assert that after the parallel run, no spurious `rmdir` of a parallel session's directory occurred. Scope the counter to invocations whose path argument equals the parallel session's `target_dir` (use a wrapper that records `target_dir.rmdir` calls only, not all `Path.rmdir`) so other in-test `rmdir` calls don't pollute the count.

### Correctness (B1–B5, 5)

- **B1 — `_LICENSE_RE` nested parens: choose rewrite.** Current pattern `r"^\s*(?P<publisher>.+?)\s*\((?P<developer>.+?)\s+license\)\s*$"` mis-captures `"Atari (JSA III) (Williams license)"` as publisher=`"Atari"`, developer=`"JSA III) (Williams"`. **Fix:** anchor the license parenthetical at the **last** open-paren rather than the first. Rewritten pattern: `r"^\s*(?P<publisher>.+?)\s*\((?P<developer>[^()]+?)\s+license\)\s*$"` — `[^()]+?` forbids nested parens inside the developer-capture, so the regex falls through to the "no match → return (cleaned, cleaned)" branch on inputs that nest. Verified post-rewrite: `"Atari (JSA III) (Williams license)"` matches publisher=`"Atari (JSA III)"`, developer=`"Williams"` — correct. **Test:** parameterized inputs (`"Capcom (Sega license)"` → `("Capcom", "Sega")`, `"Atari (JSA III) (Williams license)"` → `("Atari (JSA III)", "Williams")`, `"Capcom"` → `("Capcom", "Capcom")`, `"(no opening name) (Sega license)"` → preserves current "no-match → return cleaned twice" semantic verified by test).
- **B2 — `REGION_RE` parenthetical-title false-positive: choose tighten.** The pattern matches `\(\s*(World|USA|...)\b` — so any parenthetical text starting with a region word matches. **Fix:** the region must be followed by one of two patterns: (a) optional whitespace then a comma / close-paren / EOL (e.g. `(World)`, `(USA, Set 2)`), OR (b) required whitespace then a non-uppercase character — typically a digit (e.g. `(World 910411)`, `(Germany 1998)`) or a lowercase letter (e.g. `(Europe v2.1)`). The two-branch form preserves the real MAME parenthetical patterns enumerated in `tests/filter/test_heuristics.py` (which the initial single-branch rewrite broke); the lookahead branch was added during cluster-B implementation after the single-branch form regressed `(World 910411)`-style descriptions. Final pattern: `r"\(\s*(?P<region>World|USA|...)(?:\s*(?:,|\)|$)|\s+(?=[^A-Z]))"`. Rejects `(World Heroes 2)` because `Heroes` starts with an uppercase H — real region parentheticals never have Title-cased continuations, only modifiers introduced by a comma. **Test:** parameterized — `"World Heroes (Japan)"` → `Region.JAPAN`, `"World Heroes (set 1)"` → `Region.UNKNOWN`, `"World Heroes 2 (USA, Europe)"` → `Region.USA`, `"(World Heroes 2)"` → `Region.UNKNOWN`. The fourth is the regression-lock for the cited bug; the existing `tests/filter/test_heuristics.py::test_region_of` cases pin the preserved patterns. (`region_of` returns `Region.UNKNOWN` rather than `None` — sentinel value at `filter/heuristics.py:42`.)
- **B3 — `Machine.description` mixed-content: choose `itertext`.** Switch `description_elem.text` to `"".join(description_elem.itertext()).strip()` at `parser/dat.py:171` (presence check) and `parser/dat.py:179` (read). `itertext()` walks the whole subtree including any child element text. The fix is defensive (MAME DATs don't currently ship mixed-content `<description>`), but loss-free vs the current truncation. **Perf impact:** `itertext()` is a generator; per-call cost on a pure-text element is one extra function call + one `"".join` over a single-item iterator (~µs per machine). On the full Pleasuredome DAT (~43k machines) this adds < 50 ms to parse time vs the current `.text` path — well within the existing CI parse-budget. **Test:** construct a synthetic `<machine>` XML with `<description>World Heroes <i>2</i> Jet</description>` (using `lxml.etree.fromstring`), pass through `_machine_from_element`; assert `machine.description == "World Heroes 2 Jet"`. Pre-fix this returns `"World Heroes "`.
- **B4 — `filter.runner` logger unused: choose wire.** Three override-warning sites at `filter/runner.py:66 / 69 / 73-77` (the third is a multi-line append spanning four lines) `warnings.append(...)` without a `logger.warning(...)` companion. **Spec-text ambiguity (noted, not blocking):** `filter/spec.md:71` § Phase C says "log a warning, skip the override, and append the message to `FilterResult.warnings`". The phrase "log a warning" admits two readings — (i) call `logger.warning(...)` AND append to `FilterResult.warnings` (dual-channel; the stricter contract), (ii) "produce a warning entry" where the warning is the `FilterResult.warnings.append` itself (single-channel). FP28 chooses (i) — the dual-channel reading — because `logger = logging.getLogger(__name__)` was declared at L23 with no other use, which the file's history strongly implies was meant for this path. The implementer should tighten `filter/spec.md:71` to spell out "call `logger.warning(message)` AND append the same message to `FilterResult.warnings`" so the contract is unambiguous after the fix lands. **Fix:** at each of the three sites, add a matching `logger.warning(<same-message>)` call alongside `warnings.append(<same-message>)`. **Test:** call `run_filter(...)` with an `Overrides` entry pointing at a non-existent parent — **seed one override only** so the test seeds exactly one warning path; assert `caplog` captured exactly one `WARNING` record at `mame_curator.filter.runner` matching `r"override key '.+' is not a known parent; ignored"`. Mirror tests (one override each) for the "target not in DAT" and "target wrong group" paths.
- **B5 — `_apply_session` raw KeyError: choose typed error.** `filter/runner.py:100` `sessions.sessions[sessions.active]` raises bare `KeyError` if `active` references a stale key. **Reachability note:** `Sessions` has a `model_validator(mode="after")` at `filter/sessions.py:66-89` that raises `ValueError` when `active not in sessions` — so direct construction with stale state is blocked. The runtime bug is reachable via `Sessions.model_copy(update={"sessions": {}})`: Pydantic v2 `model_copy` does NOT re-run validators by default, so a previously-valid instance can be mutated into a stale state that survives all the way to `_apply_session`. **Fix:** wrap the subscript in `try`/`except KeyError` at `filter/runner.py:100` and re-raise as `SessionsError(f"sessions.active = {sessions.active!r} but session not in sessions.sessions")` — `SessionsError(FilterError)` already exists at `src/mame_curator/filter/errors.py:14`. **Test (Test 7):** construct valid `Sessions(active="real", sessions={"real": Session(...)})`, then `stale = valid.model_copy(update={"sessions": {}})` (skips validation; produces the stale state); call `run_filter(..., sessions=stale)`; assert raises `SessionsError`; assert `isinstance(exc, FilterError)`. Pin the message via substring assertion (`"sessions.active = 'real'"`).

### Boundary hardening (C1–C2, 2)

- **C1 — `_validate_paths` missing retroarch/retroarch_core: choose extend.** Security-relevant: PATCH-config → launch chain. **Fix:** in `api/routes/config.py:_validate_paths`, after the existing `retroarch_playlist.parent` check, add:
  (a) **POSIX path** (`if sys.platform != "win32":`) — if `p.retroarch is not None`: check `p.retroarch.exists()` and `os.access(p.retroarch, os.X_OK)`; on failure append `FieldError(loc="paths.retroarch", msg="path is not an executable file", type="path_invalid")`.
  (b) **Windows path** (`if sys.platform == "win32":`) — `os.X_OK` semantics differ (Windows treats `.exe`/`.bat`/`.cmd` as executable via file-extension, not POSIX mode bits). Use `shutil.which(str(p.retroarch))` which performs the Windows-correct extension check; on failure (returns `None`), append the same `FieldError`. Add an inline `import shutil` if not already imported.
  (c) Same shape for `p.retroarch_core` but **check `.exists()` only on both platforms** — cores are `.so` (Linux) / `.dll` (Windows) / `.dylib` (Mac) libraries loaded by RetroArch via `dlopen`/`LoadLibrary`, not directly executable.
  **Threat-model comment** at the new check site: `# FP28 C1: gates the launch-RetroArch path at games.py:275 — any PATCH that lands a non-executable path here would otherwise hit subprocess.run with a bad arg.` Promote the comment to its own line above the check (not appended inline) for scannability.
  **Test:** `POST /api/config` with body setting `paths.retroarch=<tmp_path that doesn't exist>`; assert response is 422 with the new `FieldError`. Mirror for `retroarch_core`. POSIX happy-path: write a file via `(tmp_path / "ra").touch(); (tmp_path / "ra").chmod(0o755)`; PATCH with that path; expect 200. Skip the executable assertion (not the validator itself) on Windows via `pytest.mark.skipif(sys.platform == "win32", reason="executable-bit semantics differ; Windows uses shutil.which extension check")` — the **Windows-specific positive-path test** is its own follow-up (out of scope for FP28 unless a Windows user reports a regression).
- **C2 — `media_proxy` headers: choose sniff + cache.** **Fix:** at `api/routes/media.py:61`, replace `FileResponse(path, media_type="image/png")` with `FileResponse(path, media_type=mimetypes.guess_type(str(path))[0] or "image/png", headers={"Cache-Control": "public, max-age=2592000, immutable"})` — 30-day cache (matches **design § 6.3** "Cache is permanent by default"; `immutable` tells the browser never to revalidate). Import `mimetypes` at module top. **Test:** GET `/media/foo/boxart` against a fixture where the cached file is `.jpg`; assert response's `Content-Type` starts with `image/jpeg` and `Cache-Control` is `public, max-age=2592000, immutable`.

### CLI exit-code / error-surface drift (D1–D3, 3)

- **D1 — `serve` Ctrl-C exit code: choose explicit catch.** **Fix:** wrap the `uvicorn.run(...)` call at `cli/__init__.py:492` with `try: uvicorn.run(...) except KeyboardInterrupt: return 130`. Uvicorn's documentation states that it catches `KeyboardInterrupt` internally and returns normally, so the wrapping catch is defense-in-depth: if a future uvicorn version re-raises, our catch handles it; if it keeps catching, the function falls through to a deliberate `return 130` at the end of the function (not the current implicit `return 0`). The explicit `return 130` honors POSIX convention. **Test:** `subprocess.run(["mame-curator", "serve", "--port", <free port>], timeout=2)` in a thread; send SIGINT; assert exit code 130. (Use `signal.CTRL_C_EVENT` on Windows; skip on Windows if uvicorn's signal handling differs — `pytest.mark.skipif(sys.platform == "win32", reason="...")` per the existing FP09 fixes test precedent.)
- **D2 — bare `except Exception`: choose narrow.** `cli/__init__.py:485` `try: app = create_app(args.config) except Exception as exc:`. **Fix:** narrow to the typed errors `create_app` actually raises — `ConfigError` / `ParserError` / `FilterError` (typed) on bad config inputs, locked in § Pre-spec verification. An unexpected `Exception` would be a programmer error and **should propagate** so the user gets the traceback (not a one-line "failed to create app: ..." that hides the cause). Narrow to `except (ConfigError, ParserError, FilterError) as exc:`. **Decision (single-path, no catch+log alternative):** the project's coding standards § 9 ("Typed exceptions per module") favour clean propagation over defense-in-depth wrapping; if a programmer error reaches the CLI, the traceback is the actionable signal. **Test:** monkeypatch `create_app` to raise `RuntimeError`; call `_cmd_serve`; assert the `RuntimeError` propagates (post-fix) rather than being squashed into "failed to create app: ...".
- **D3 — `refresh-inis` raw `ImportError`: choose mirror `serve`'s guard.** **Reachability note:** `httpx>=0.28` is a **top-level dep** at `pyproject.toml:35` (not under `[project.optional-dependencies]`; verified at HEAD), so on a normally-installed wheel `import httpx` cannot fail. The fix is therefore **defense-in-depth pattern consistency** with `_cmd_serve` (which has the same import-guard shape at L473-481), not a guard against a reachable bug — it covers exotic install states (`pip install --no-deps`, broken wheel, partial editable install). **Fix:** lift the `import httpx` + `from mame_curator.updates import refresh_inis` lines at `cli/__init__.py:573-577` (three inline imports at L573 `asyncio`, L575 `httpx`, L577 `mame_curator.updates`) into a `try: ... except ImportError as exc:` block matching `_cmd_serve`'s shape at L473-481. On ImportError print the user-facing message:

  ```
  error: failed to import dependencies ({exc}); reinstall the project (uv sync, or pip install -e .)
  ```

  Use `rich.Console`'s markup (`[red]error:[/red]`) for the prefix; render the command names as plain text in the message (no inline backticks inside the f-string to avoid backtick-rendering issues in this spec and in the runtime output). No extras-key in the error message — the project has no relevant `[updates]` extras. Return 1. **Test:** monkeypatch `sys.modules["httpx"] = None`; call `_cmd_refresh_inis`; assert returns 1 and stderr contains `"failed to import dependencies"`.

## Architecture notes

- **No new modules.** All fixes land in existing files: `api/jobs.py`, `copy/recyclebin.py`, `parser/manufacturer.py`, `filter/heuristics.py`, `parser/dat.py`, `filter/runner.py`, `api/routes/config.py`, `api/routes/media.py`, `cli/__init__.py`, plus the new ADR file at `docs/decisions/0004-ini-refresh-trust-model.md` and a one-line addition to `docs/superpowers/specs/2026-04-27-mame-curator-design.md § 6.7`.
- **`os.O_EXCL` lockfile for A2.** Avoids a new dependency (`filelock` would otherwise pull in `portalocker` on Windows). The pattern: `os.open(path, os.O_CREAT|os.O_EXCL|os.O_WRONLY)` raises `FileExistsError` if the file already exists, giving us a portable atomic-create primitive. The cleanup path uses `try/finally` to guarantee the lockfile is deleted on the happy path; orphaned lockfiles (process killed mid-section) are cleaned by an opportunistic check at the start of every `recycle_file` call: if the lockfile is older than 60 s (~1200× the targeted p99 of 50 ms — see § Scope A2 perf budget), unlink it before re-acquiring.
- **`mimetypes` for C2.** Stdlib, no new dep. Note that `mimetypes.guess_type` returns `None` for unknown extensions; the post-fix call uses `or "image/png"` as the fallback (preserves current behavior for un-suffixed cache paths). Test 9's `.jpg` happy-path relies on the cache filename ending in `.jpg` for the suffix-based sniff — implementation-coupling between cache layout (`fetch_with_cache` writes `sha256(url).<ext>` per design § 6.3) and the test fixture; acceptable.
- **`SessionsError(FilterError)` for B5.** Verified at `src/mame_curator/filter/errors.py:14` (subclass of `FilterError` at L6). The typed-error hierarchy is named in `coding-standards.md` § 9 (Errors, logging, observability) — "Typed exceptions per module". No new class — use the existing `SessionsError`.
- **No backwards-compat shims for D2's narrowing.** Per `CLAUDE.md:82` "No backwards-compat shims pre-v1.0.0" — project is at 1.2.0, but the rule of thumb still applies: narrow the catch directly, no deprecation cycle. The narrowed `except` matches the actual raise sites in `create_app`; any caller relying on "all exceptions become 'failed to create app:'" was relying on broken behavior.
- **Implementation order.** A2's lockfile lands before A3's comment-only commit (A3 has no separate code). D1 + D2 share lines `cli/__init__.py:483-496` (D2's `except` at L485 + D1's `KeyboardInterrupt` wrap at L492 + the explicit `return 130` near L496); bundle into one commit with the test pair.
- **No `setup/` work in scope.** The `setup/` package's absence (FP27 § C2 verification) means E1's deferred work doesn't have a "wizard" package to retrofit. The ADR explicitly documents this — the wizard's mirrors+sha256 promise lives with the wizard implementation, whenever it ships.
