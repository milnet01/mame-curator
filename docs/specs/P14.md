## Contract

Each MAME machine carries one of four review states: `pending` (the default), `reviewed`, `skipped`, `needs-decision`. Users mark games with the **R / S / ?** keys; arrow keys move a focus ring between cards in the library grid and into the alternatives drawer. The `/library` header shows an `X / Y handled` progress chip scoped to the current filter set, plus a `Currently: <letter|decade>` sub-chip when walkthrough auto-advance is on. Auto-advance is on by default and can be toggled off in the header.

State persists across sessions on disk; it is **not** scoped to the active session — a game marked Reviewed from the catalogue view is also Reviewed when seen through an "80s shooters" session. State changes append events to the existing `data/activity.jsonl` log.

## Out of scope (YAGNI guardrails)

The following are deliberately not addressed by P14:

- Cross-device sync / cloud backup of state.
- Per-session state (state is global).
- Sound effects or motion beyond Radix defaults.
- Bulk operations such as "mark all currently-visible as reviewed".
- Multi-step undo (the toggle gives single-step revert).
- Pruning state entries for games no longer in the DAT — entries are tolerated on load and filtered out at use time.
- Help-overlay shortcut. `?` is currently free and P14 claims it; a future help overlay must pick a different key.

## Data model

### On-disk format — `data/state.yaml`

Sparse store; pending = absence:

```yaml
state:
  sf2: reviewed
  pacman: skipped
  galaga: needs-decision
```

Two-space indent matches `overrides.yaml`. Missing file = empty state (no error). Schema violations raise `ReviewStateError`.

### Pydantic model — `src/mame_curator/filter/review_state.py`

```python
from enum import StrEnum
from pathlib import Path

from pydantic import BaseModel, ConfigDict, Field


class ReviewStateValue(StrEnum):
    """Storage values — what `state.yaml` ever holds.

    `pending` is the sparse default and is intentionally absent from this
    enum: a game in pending state has NO entry in `state.yaml`.
    """

    REVIEWED = "reviewed"
    SKIPPED = "skipped"
    NEEDS_DECISION = "needs-decision"


class ReviewStateFilter(StrEnum):
    """Query-parameter values for the filter pipeline.

    Distinct from `ReviewStateValue` because the filter accepts two
    sentinel values (`all`, `pending`) that never appear on disk.
    """

    ALL = "all"
    PENDING = "pending"
    REVIEWED = "reviewed"
    SKIPPED = "skipped"
    NEEDS_DECISION = "needs-decision"


class ReviewState(BaseModel):
    """Per-game review state. Frozen; mutations create a new instance.

    Mirrors `Overrides` (`alias="overrides"`): the dict field is named
    `entries` in code but serialises under the top-level YAML key `state`,
    so `state.yaml` reads `state: {sf2: reviewed, ...}` while in-Python
    we keep the name decoupled from the file format.
    """

    model_config = ConfigDict(frozen=True, extra="forbid", populate_by_name=True)

    entries: dict[str, ReviewStateValue] = Field(default_factory=dict, alias="state")


class ReviewStateError(Exception):
    """Raised on malformed state.yaml."""


def load_review_state(path: Path) -> ReviewState:
    """Load state from YAML; missing file → empty ReviewState."""
```

`load_review_state` mirrors `load_overrides` exactly (`src/mame_curator/filter/overrides.py:27`): missing file returns empty model, empty / `null` top-level returns empty model, non-mapping raises `ReviewStateError`, schema violations re-raised as `ReviewStateError`.

### `WorldState` integration

`src/mame_curator/api/state.py` — the `WorldState` Pydantic class gains one field, placed beside `overrides` / `sessions` / `notes`:

```python
class WorldState(BaseModel):
    model_config = ConfigDict(frozen=True, extra="forbid", arbitrary_types_allowed=True)
    ...
    overrides: Overrides
    sessions: Sessions
    review_state: ReviewState        # NEW
    ...
```

`replace_world()` (same module) is **extended** in this phase — its current signature is `(*, base, config=None, overrides=None, sessions=None, notes=None, rerun_filter=False)` and does not yet accept review-state. P14 adds:

```python
def replace_world(
    *,
    base: WorldState,
    config: AppConfig | None = None,
    overrides: Overrides | None = None,
    sessions: Sessions | None = None,
    notes: Mapping[str, str] | None = None,
    review_state: ReviewState | None = None,   # NEW — passive field swap; does NOT re-run filter pipeline
    rerun_filter: bool = False,
) -> WorldState: ...
```

**The filter-recompute trigger is NOT broadened to cover `review_state`.** This is the key architectural fact: review state does not gate machine eligibility — every machine in the DAT is eligible regardless of whether it has been reviewed. `world.filter_result` (the materialised eligible set, produced by `run_filter()`) is independent of `world.review_state`. A `replace_world(review_state=...)`-only swap therefore leaves `filter_result` untouched. The new field is passed through to the returned `WorldState` and nothing else is recomputed. (The allowlist short-circuit at `compose_allowlist`, load-bearing per FP09 Cluster R H2, is likewise untouched.)

The `?review_state=` filter that DOES narrow the visible set is applied per-request in `api/routes/games.py` against `world.review_state.entries` and the already-computed `world.filter_result` — see "Filter (per-request)" below.

## API contract

Three new routes, registered on the `curate.py` router. All three mirror the overrides/sessions pattern of the same file.

### `POST /api/state` — set a non-pending state

Request body (`StatePostRequest`):

```json
{ "short_name": "sf2", "state": "reviewed" }
```

`state` ∈ `{reviewed, skipped, needs-decision}`. Setting `pending` is rejected — use `DELETE` to clear.

```python
@router.post("/api/state", response_model=StateView)
async def post_state(body: StatePostRequest, request: Request) -> StateView:
    async with request.app.state.world_lock:
        world: WorldState = request.app.state.world

        # Inline the existing `if name not in world.machines: raise ...`
        # pattern (games.py:312-313 et al.) — no new helper.
        if body.short_name not in world.machines:
            raise GameNotFoundError(f"game not found: {body.short_name!r}")

        previous = world.review_state.entries.get(body.short_name)

        # INV-13: skip the write + activity event when the request is a
        # no-op (same state already recorded). Return current state, 200.
        if previous == body.state:
            return StateView(entries=dict(world.review_state.entries))

        entries = dict(world.review_state.entries)
        entries[body.short_name] = body.state
        # Validate by field name (matches `routes/curate.py:70` calling
        # `Overrides.model_validate({"entries": entries})`). The `alias="state"`
        # is the YAML key on disk; in code, the field name `entries` is what
        # the existing Overrides pattern passes.
        new_state = ReviewState.model_validate({"entries": entries})

        # Disk-write order under partial failure: persist before activity
        # before world swap. If yaml write raises, no activity event is
        # appended and no world swap happens — caller sees 500 with state
        # unchanged. If activity append raises after persist succeeds, the
        # on-disk state is ahead of the activity log; recovery walks the
        # YAML (the source of truth) on next startup.
        _persist_review_state(world, new_state)
        append_activity(_build_review_event(body.short_name, body.state, previous),
                        log_path=world.data_dir / "activity.jsonl")

        new_world = replace_world(base=world, review_state=new_state)
        set_world(request, new_world)
        return StateView(entries=dict(new_state.entries))
```

The two private helpers introduced in this phase:

- `_persist_review_state(world, new_state)` — atomic YAML write via the existing `write_yaml_atomic` (`api/persist.py:155`); see "Snapshot policy" below for why this skips `snapshot_files()`.
- `_build_review_event(short_name, state, previous)` — constructs the `ActivityEvent` with the new tagged-union details payload (see "Activity log integration").

The append uses the existing helper `append_activity()` from `src/mame_curator/copy/activity.py:50`.

### `DELETE /api/state/{short_name}` — clear to pending

Removes the entry from the sparse store. Idempotent — `DELETE` on a game already at pending returns 200 with current state and writes no activity event. `short_name` must resolve to a known machine; unknown → 404 (`GameNotFoundError` / `code: "game_not_found"`).

A no-op DELETE (entry was already absent) also skips `_persist_review_state` since the state dict is unchanged — avoids touching the file's mtime on a keypress that did nothing.

### `GET /api/state` — full state map

Returns `StateView { entries: {[shortname]: ReviewStateValue} }`. Used by frontend cache hydration on page load.

### Errors

- Unknown `short_name` → 404 `{code: "game_not_found", detail: "..."}` (reuses the existing `GameNotFoundError` from `src/mame_curator/api/errors.py:137-140` — no new code).
- Invalid `state` enum on POST → 422. `pending` is rejected by Pydantic enum validation **before the handler runs** (the `state` field is typed `ReviewStateValue`, which excludes `pending` by design) — the 422 envelope is FastAPI's standard validation error, not a hand-rolled rejection.
- Body shape mismatch → 422.
- `?review_state=` query param with an unknown value → 422 (Pydantic enum validation on `ReviewStateFilter`).
- Disk-write failures → 500 (yaml write raises bare `OSError`; the `append_activity` call raises typed `ActivityLogError` — a `CopyError` subclass — on its own write failures; both reach FastAPI's default 500 handler).

P14 introduces **no new error codes**. Frontend `strings.ts` does not need additions on the error-code side, only for the new user-facing copy (see "Strings to add" below).

## Filter (per-request, in `api/routes/games.py`)

`GET /api/games` gains one query parameter:

```
?review_state=all|pending|reviewed|skipped|needs-decision    (default: all)
```

Typed as `ReviewStateFilter` (the 5-value enum; see "Pydantic model" above). Applied **per-request inside the route handler**, NOT inside `run_filter()`:

```python
# In src/mame_curator/api/routes/games.py, after the existing filter slice:
if review_state != ReviewStateFilter.ALL:
    entries = world.review_state.entries
    if review_state == ReviewStateFilter.PENDING:
        # short_name not in the sparse map = pending
        machines = [m for m in machines if m.short_name not in entries]
    else:
        # entry must match the requested value
        target = ReviewStateValue(review_state.value)
        machines = [m for m in machines if entries.get(m.short_name) == target]
```

This sits on top of the slice already returned by the world's `filter_result` — every game whose state-filter passes still has to be in `world.filter_result`. The two filters compose (eligibility from `run_filter`, then visibility from review-state).

Why per-request and not in `run_filter()`: `run_filter` produces the materialised eligible set used by every page load until `replace_world` runs — caching at world-build cost, not per-request cost. Review state is a fast-mutating per-request slice (the same world can serve `?review_state=pending` to one request and `?review_state=reviewed` to the next without rebuilding `filter_result`). Folding it into `run_filter` would force a full re-filter per keypress.

`src/mame_curator/filter/spec.md`'s filter-stage table is **not** modified — there is no new run_filter stage. The per-request layer is documented in `api/routes/games.py`'s docstring.

**Frontend ↔ backend pending parity (INV-12).** The set of pending candidates the walkthrough cycles through must equal the set returned by `GET /api/games?review_state=pending&<current other filter params>`. The frontend computes the next-pending index by walking the locally cached `games[]` array (which already reflects the active filter set) and skipping any short-name present in the cached `reviewState.entries` map — identical predicate to the backend route.

### Badge — `GameCard.tsx`

| State value | Icon (Lucide) | Tint |
|---|---|---|
| `reviewed` | `CheckCircle` | `text-emerald-500` |
| `skipped` | `Ban` | `text-rose-500` |
| `needs-decision` | `HelpCircle` | `text-amber-500` |

The three Tailwind utilities (`text-emerald-500`, `text-rose-500`, `text-amber-500`) are part of Tailwind v4's default palette and resolve against the project's `@theme inline` block in `frontend/src/index.css`, which only declares *semantic* tokens additively — the colour palette is not overridden. **Rendering path is frontend-local, not backend-emitted.** Unlike the existing five badges (`contested` / `overridden` / `chd_missing` / `bios_missing` / `has_notes`) — which the backend `_badges()` function emits onto each `GameCard` row in `GET /api/games`, and which the bidirectional type-sync gate keeps in lockstep with the TS `Badge` enum — the three review-state badges are computed entirely on the frontend from the locally cached review-state map. The reason: optimistic UI — when the user presses `R`, the badge must appear in the same frame as the keypress, before the backend round-trip completes. Reading from the local cache (which React Query's `onMutate` updates synchronously) achieves this; waiting for the next games-list refetch would not.

**The review-state badges are NOT added to the `Badge` enum.** Adding them there without a backend emitter would fail the bidirectional type-sync gate (`tools/check_api_types_sync.py` — Pydantic and TS must each contain every field the other contains; an enum value present on the TS side but absent on the backend is a CI failure). Instead the spec introduces a frontend-only type:

```ts
// frontend/src/api/types.ts
export type ReviewBadgeKind = 'reviewed' | 'skipped' | 'needs-decision'
```

The `BADGE_LABELS` / `BADGE_ICONS` maps in `GameCard.tsx` gain a parallel `REVIEW_BADGE_LABELS` / `REVIEW_BADGE_ICONS` pair — **not** merged into one map keyed by `Badge | ReviewBadgeKind`. Reason: the existing maps are `Record<Badge, …>`, which gives TS exhaustiveness checks across the five backend-emitted values; merging would weaken that guarantee for the existing five.

Concretely: `GameCard` accepts a new `reviewState?: ReviewStateValue` prop. `LibraryPage` derives it via `reviewState.entries[card.short_name] ?? undefined`, where `reviewState` is the hydrated value from `useReviewState()` (cache key `['reviewState']`; initial fetch via `GET /api/state`). The component renders backend `badges[]` first (existing five) then conditionally appends a single review-state badge if `reviewState` is set. The badge slot reuses the existing 6×6 top-right `<li>` pattern; the per-state tint is applied via Tailwind class on the icon.

### Keyboard bindings — split by surface

Two surfaces, two registration styles:

**Grid (`LibraryGrid.tsx`'s existing `onKeyDown` prop on `<div role="grid">`).** Extend the existing handler — same React event already wired for arrows / Home / End / Enter. New cases:

| `event.key` | Action |
|---|---|
| `r` | toggle Reviewed on `cards[activeIndex]` (via mutation callback prop) |
| `s` | toggle Skipped |
| `?` | toggle Needs-decision |

Arrow keys keep their existing behaviour. The grid's onKeyDown only fires when grid focus is held — when the search input has focus, the keydown stays on the input and never reaches the grid. No `isModifiable` check needed (React onKeyDown is naturally scoped to the focused element's bubble path).

**Drawer (`AlternativesDrawer.tsx`).** Uses `useKeyboard` (the drawer has no existing keyboard surface). Registered on drawer-mount, unmounts on close. Bindings:

| Combo | Action |
|---|---|
| `r` / `s` / `?` | toggle state on the highlighted drawer row |
| `ArrowUp` / `ArrowDown` | move `highlightedRowIndex` within drawer |

The drawer is a shadcn `Sheet` (Radix Dialog focus-trap), so when open, document-level useKeyboard handlers fire only with focus inside the drawer. There is no race with the grid's React onKeyDown.

```ts
  if (current === next) {
    onClearReviewState(card.short_name)                // callback prop
    // No walkthrough advance on toggle-to-pending; the just-cleared
    // card is itself pending now.
  } else {
    onSetReviewState(card.short_name, next)            // callback prop
    if (walkthrough) {
      const nextIdx = focusNextPending(startedAtIndex + 1)
      if (nextIdx === null) toast.success(strings.library.walkthroughCaughtUp)
      else setActive(nextIdx)
    }
  }
  return
}
```

`onSetReviewState` / `onClearReviewState` are mutation callbacks supplied by `LibraryPage` (which holds the React Query `useReviewState()` hook). `walkthrough` is also a prop from `LibraryPage` (sourced from `localStorage`). The `focusNextPending(startIndex)` signature is load-bearing: walking from `activeIndex + 1` guarantees the just-marked card is skipped regardless of when its updated state lands in the `cards` array on the next render — closes the race the naive `focusNextPending()` (start-from-current) would have.

**Drawer behaviour.** `AlternativesDrawer` is implemented with the shadcn `Sheet` primitive (`@/components/ui/sheet`), which wraps Radix `Dialog` internally and inherits the focus-trap. Inside the drawer:

- `↑` / `↓` move a `highlightedRowIndex` between the alternatives (parent + clones presented as rows). Highlight uses an existing-style outline (mirror the `focused?` prop visual on `GameCard`).
- `r` / `s` / `?` from inside the drawer call the same mutations on the highlighted row's `short_name` — drawer registers its own `useKeyboard` bindings scoped to its mount lifetime, with the `if (editing) continue` guard inside `useKeyboard.ts` (line 78) automatically suppressing bindings while focus sits in a text input.
- `Esc` (Sheet default) closes the drawer; the grid's roving-tabindex restores focus to the originating card automatically.

The drawer's `r / s / ?` handlers do **not** auto-advance — the user typically opens the drawer to compare clones individually, not to walk through them. Per-clone state is the intended model (each clone is its own state entry; reviewing `sf2ce` does not mark `sf2t`).

**No listener-overlap concern.** Grid R/S/? is a React `onKeyDown` prop on the `<div role="grid">` root; it fires only when grid focus is held. Drawer R/S/? is a document-level `useKeyboard` listener; while the drawer is open, Sheet's focus-trap keeps DOM focus inside the drawer so the grid's onKeyDown never fires for those keypresses. The two surfaces are physically scoped by focus — no guard variable needed.

**Keyboard-layout assumption.** Both the grid's React `onKeyDown` and the drawer's `useKeyboard` match by `event.key`, which is layout-dependent: on US QWERTY, `Shift+/` yields `event.key === '?'` and the `?` binding fires. On AZERTY / German / other layouts, `?` may sit behind a different physical chord or no single chord at all. P14 v1 assumes US layout; non-US layouts get R / S working (single letter keys, layout-independent) but `?` may be unreachable. A follow-up could match `event.code === 'Slash' && event.shiftKey` for layout independence; out of scope for v1.

### Optimistic UI

The mutations use React Query's `onMutate` / `onError` rollback pattern. Cache key is `['reviewState']` (not the overloaded `['state']`) to align with the existing `['games']` / `['sessions']` / `['facets']` keys in the project:

```ts
useMutation({
  mutationFn: postReviewState,
  onMutate: async (body) => {
    await queryClient.cancelQueries({ queryKey: ['reviewState'] })
    const prev = queryClient.getQueryData<StateView>(['reviewState'])
    queryClient.setQueryData<StateView>(['reviewState'], (old) =>
      updateLocally(old, body),
    )
    return { prev }
  },
  onError: (_err, _body, ctx) => {
    if (ctx?.prev) queryClient.setQueryData(['reviewState'], ctx.prev)
    toast.error(strings.library.stateUpdateFailed)
  },
  onSettled: () => queryClient.invalidateQueries({ queryKey: ['reviewState'] }),
})
```

Local state updates immediately so the badge appears and focus advances within the same frame as the keypress; rollback on backend rejection is rare and surfaces as a toast plus state revert.

Both `postReviewState` and `deleteReviewState` hooks live in `frontend/src/hooks/useReviewState.ts` (alongside the existing `useOverride` / `useSessions` hooks in `frontend/src/hooks/`). The hook also exposes the hydrated cache for downstream consumers (`GameCard` reads the per-game state from `useReviewState()` and renders the badge from local state, per the "Badge" section above).

## Activity log integration

The project already ships an `ActivityEvent` tagged-union model at `src/mame_curator/copy/types.py:318-326`:

```python
class ActivityEvent(BaseModel):
    timestamp: datetime
    event_type: ActivityEventType           # closed StrEnum (copy/types.py:210-221)
    summary: str
    session_id: str
    details: ActivityDetails                # Annotated discriminated union
```

P14 extends this in **three** small places (all in `src/mame_curator/copy/types.py`):

1. `ActivityEventType` gains one enum value: `REVIEW_STATE = "review_state"`.
2. A new `ReviewStateDetails` payload class is added, mirroring `OverrideSetDetails`:

   ```python
   class ReviewStateDetails(BaseModel):
       """`review_state` event payload (P14 emitter)."""

       model_config = ConfigDict(frozen=True, extra="forbid")
       event_type: Literal[ActivityEventType.REVIEW_STATE] = ActivityEventType.REVIEW_STATE
       short_name: str
       state: str             # post-change value: one of {reviewed, skipped, needs-decision, pending}
       previous: str          # pre-change value, same domain
   ```

   `state` and `previous` are `str` (not `ReviewStateValue`) because the activity log records the literal transition including the sentinel `"pending"` — keeping the log's domain decoupled from the storage enum (which excludes pending) avoids forcing the log readers to care about the sparse-store invariant.

3. The `ActivityDetails` `Annotated` union picks up `ReviewStateDetails`. (One line added to the union expression at `copy/types.py:304-315`.)

The serialised JSONL line is therefore:

```json
{
  "timestamp": "2026-05-16T17:46:00Z",
  "event_type": "review_state",
  "summary": "marked sf2 as reviewed",
  "session_id": "",
  "details": {
    "event_type": "review_state",
    "short_name": "sf2",
    "state": "reviewed",
    "previous": "pending"
  }
}
```

`session_id` is `""` (empty string) for review-state events. Justification on its own merits: per-game state is not scoped to any user-defined session or copy job — it is a global per-game annotation that survives session activation/deactivation. The empty string communicates "no enclosing scope" without inventing a placeholder. (No existing call site of `OverrideSetDetails` / `SessionActivatedDetails` is currently in production, so there is no "convention" to anchor against; P14 sets the precedent for non-job-scoped events.) `summary` is a short human-readable line composed at emit time (`"marked {short_name} as {state}"` / `"cleared {short_name}"`).

P14 emits one event per mutation that **changes** the in-memory map (INV-5). A no-op DELETE does not emit. The route in `src/mame_curator/api/routes/activity.py` is unchanged — the JSONL stream is already serialised per-event without route-side filtering.

Frontend rendering: `frontend/src/pages/ActivityPage.tsx` (existing) needs one new case in its event-renderer switch to format `review_state` rows (icon + caption from `strings.ts`).

## Snapshot policy

State writes do **not** call `snapshot_files()`. Justification:

- State changes are high-frequency (one per keypress; a walkthrough session can mark 100+ games in a few minutes).
- The snapshot directory `data/snapshots/` is **shared** across every writer that snapshots — overrides, sessions, config (`api/persist.py:31` — `MAX_SNAPSHOTS = 200`, LRU eviction across the whole pool, not per-file). State writes at keypress frequency would churn the pool, evicting every overrides / sessions / config snapshot within minutes.
- Recovery is covered by `activity.jsonl`: every state transition is one append-only line, so the state file can be rebuilt from activity if ever lost. A small `tools/rebuild_state_from_activity.py` script (out of scope for v1; not implemented unless needed) would walk the log.

`overrides.yaml` and `sessions.yaml` keep their existing per-write snapshot policy — those are low-frequency user-editorial mutations and snapshotting them is cheap.

**Snapshot UI caveat.** `GET /api/snapshots` lists files in `data/snapshots/`; since `state.yaml` is not snapshotted, the snapshot-listing UI will not show review-state changes and the user cannot roll back state via the snapshots view. Recovery is `activity.jsonl` replay only. The Settings → Snapshots page copy should note this (one-line caption, added to `strings.ts` as `settings.snapshots.stateExclusionNote`).

**Activity-log path invariant.** P14's emitter passes `log_path=world.data_dir / "activity.jsonl"` explicitly — it does not rely on `append_activity`'s default `Path("data/activity.jsonl")` (relative-to-CWD). Other emitters in `copy/runner.py` use the default; the two paths resolve to the same file only because the API process's CWD == config-parent (the `build_world` startup default at `api/state.py:136`). If a future caller invokes the activity helpers from a different CWD, divergence is a silent loss. The invariant: **all emitters must funnel through `world.data_dir`** going forward; existing default-path emitters should migrate in a follow-up phase.

## Invariants (audit surface)

Each invariant below has at least one test enforcing it.

- **INV-1** `state.yaml` is sparse — only games with state ∈ `{reviewed, skipped, needs-decision}` appear. Pending entries are never written.
- **INV-2** Missing `state.yaml` returns an empty `ReviewState` with no error.
- **INV-3** All state mutations serialise via `request.app.state.world_lock`.
- **INV-4** `WorldState` is frozen; mutations rebuild via `replace_world(review_state=...)`. The swap is a **passive field replacement** — it does NOT trigger `run_filter()` recomputation, because the review state does not gate machine eligibility (every machine is eligible regardless of state). `world.filter_result` is unchanged across review-state-only swaps. The `?review_state=` per-request filter (in `api/routes/games.py`) is what narrows the visible set; it reads `world.review_state.entries` on demand.
- **INV-5** Every mutation that changes the in-memory `review_state.entries` map appends exactly one `kind: "review_state"` line to `data/activity.jsonl`; the `previous` field reflects the prior in-memory value. A no-op `DELETE` against an already-pending game writes nothing.
- **INV-6** R / S / ? on a focused card mutates that card's state. When walkthrough is on AND the keypress results in a non-pending state, focus advances to the next pending card in current filter scope; the mutation and the advance are independent observable events.
- **INV-7** When the alternatives drawer is open, R / S / ? mutates the highlighted drawer row, not the card the drawer was launched from. Drawer R / S / ? does **not** auto-advance regardless of walkthrough setting.
- **INV-8** `POST /api/state` with unknown `short_name` returns 404 `game_not_found`; with invalid `state` value returns 422; with state = `pending` returns 422. The `pending` rejection happens at Pydantic-validation layer (the request-body `state` field is typed `ReviewStateValue`, which excludes `pending`) **before the handler runs** — the 422 is FastAPI's standard validation envelope.
- **INV-9** On-disk format is `{state: {short_name: enum_value}}` with two-space indent. Round-trip stability rests on `write_yaml_atomic`'s `yaml.safe_dump(..., sort_keys=True, default_flow_style=False)` contract (`src/mame_curator/api/persist.py:155-158`): given the same logical map, two writes produce **byte-identical** files. If either flag ever flips (`sort_keys` defaulting to `True` already; `default_flow_style=None` would inline small dicts), INV-9 falls — but those are the persist module's contracts, not P14's. (P14's contract: it never relies on insertion order and never reads the file's byte-form for any equality check.)
- **INV-10** `?review_state=` accepts `{all, pending, reviewed, skipped, needs-decision}`; any other value returns 422.
- **INV-11** State entries for unknown short names (game removed from DAT between sessions) are tolerated on load — they sit in `world.review_state.entries` without raising. They are filtered out wherever the filter runner consults state, since the game itself is no longer in `world.machines`.
- **INV-12 (pending parity).** The set of pending candidates the walkthrough cycles through equals the set returned by `GET /api/games?review_state=pending&<current other filter params>`. Frontend implements this by walking the locally cached `games[]` (already shaped by the active filters) and skipping any `short_name` present in the cached `reviewState.entries` map — identical predicate to the backend stage.
- **INV-13 (no-op write skip).** Mutations that leave the in-memory `review_state.entries` map byte-identical skip both `write_yaml_atomic` and `append_activity`. Covers two cases: (a) `DELETE /api/state/{short_name}` against a game already at pending (entry absent); (b) `POST /api/state` with `{short_name, state}` where `entries[short_name] === state` already (same value re-posted). The POST handler reads `previous = world.review_state.entries.get(body.short_name)` and short-circuits when `previous == body.state`, matching INV-5's "changes the in-memory map" wording.

## Open design calls

These are the choices baked into this spec; flag any to revise before sign-off.

1. **Sparse-store with pending = absence.** Deviates from any "store every machine" reading. Justified by the badge-when-non-default phrasing in ROADMAP.
2. **Single file `data/state.yaml`.** Deviates from ROADMAP wording *"persisted under `data/state/`"* (literal-read = directory). Picked single-file in brainstorming because per-game files would mean up to 43k tiny files on a fully-marked catalogue; `notes.json` is the existing single-file precedent for per-game user data.
3. **Flat entry shape `{shortname: state}`** (deferring `{shortname: {state, marked_at, ...}}`). `marked_at` deferred until a "resume" or "recently marked" UX needs it. Migration is one helper function if the time ever comes.
4. **Single-select segmented picker for filter** (over four independent `Only X` switches; states are mutually exclusive). Implemented with the existing `RadioGroup` primitive — `ToggleGroup` is not in the project.
5. **Toggle on same-key press** (R on reviewed → pending; over an explicit clear key).
6. **Auto-advance on by default, header toggle to disable** (over always-on or always-off).
7. **No "Currently: A / 1980s" sub-chip in v1.** Deferred to a phase that adds a sort selector to `LibraryPage` — there is no sort state to read against today.
8. **Optimistic UI on mutations** (over pessimistic wait; rollback on backend rejection is the rare path). Cache key `['reviewState']` matches the project's existing key-namespace convention.
9. **No per-change snapshot of `state.yaml`** (recovery via `activity.jsonl` replay; explicit deviation from the overrides/sessions snapshot policy because state writes are keypress-frequency).
10. **Per-clone state in the alternatives drawer** (over "drawer R/S/? marks the parent"). Each clone is its own state entry; reviewing `sf2ce` does not mark `sf2t`. INV-7 codifies this.
11. **`?` claimed for needs-decision** (per ROADMAP). No help-overlay collision today; FP27 removed the only prior advertisement of `?` (the chord-shortcut list was unwired). A future help overlay must pick a different key.
12. **Walkthrough does not auto-fetch the next API page.** `useGames` returns one paged slice (page size in `LibraryPage`); `focusNextPending(startIndex)` walks the in-memory slice and returns `null` when the slice is exhausted, surfacing the "All caught up" toast. The user advances pages manually. A future enhancement could chain auto-fetch + auto-advance, but it is not in v1 scope.
