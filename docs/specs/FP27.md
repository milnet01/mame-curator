### Tier 2 — data integrity (5)

- **B1 — `executor.copy_one` fsync gap.** `copy_one` has **two** write paths in `src/mame_curator/copy/executor.py`: (i) `_chunked_copy` at `executor.py:26-33` (taken when `progress is not None`), and (ii) `shutil.copy2(src, tmp)` at `executor.py:86` (taken when `progress is None`). **Neither path** calls `os.fsync(fout.fileno())` before `os.replace(tmp, dst)` at `executor.py:87`. After `os.replace` returns, the dirent has been swapped but the underlying tmp's bytes may still live in dirty page cache; a power cut before the next FS sync leaves `dst` zero-byte or truncated on reboot. Failure-mode framing: any power cut within the kernel's writeback window (~30s default `dirty_expire_centisecs`) after a long copy leaves `dst` bytes potentially un-journaled. **Fix:** (1) in `_chunked_copy`, add `fout.flush(); os.fsync(fout.fileno())` as the last two statements inside the `with fout` block (so the close-on-`__exit__` is the third action). (2) For the `shutil.copy2` branch in `copy_one`, replace with the chunked path unconditionally (call `_chunked_copy(src, tmp, total, progress or (lambda _d, _t: None))`) — eliminates the second code path AND the unprotected branch in one move. (3) `_atomic.py:74` exports `fsync_parent_dir(path)` (per the FP09 work landed at `_atomic.py:63-66`); add `fsync_parent_dir(dst)` immediately after the `os.replace(tmp, dst)` at `executor.py:87`. The executor does not currently import `_atomic` (grep-confirmed); the new import line lands as part of the fix.
- **B2 — `persist.restore_snapshot` transaction atomicity.** Per-file `atomic_write_bytes` is atomic *per file*; the loop is not transaction-atomic. A crash mid-loop with N files in `targets` leaves k files restored (k < N) and (M-k) files unlinked (M = files-absent-from-snapshot). The user's settings end up in a mixed state — half old, half new — that the snapshot system was supposed to prevent. **Fix (option 1, simple, chosen):** stage-then-promote. Compute the full set of `(dst_path, action, payload)` tuples first (action ∈ {`write`, `unlink`}). For every `write` action, copy the snapshot file to a sibling staging path (`snap_dir / "_restore_staging" / dst.name`). Once all staging files are in place, walk the action list and perform `os.replace(stage_path, dst)` for each `write` plus `dst.unlink(missing_ok=True)` for each `unlink`. **Honest trade:** doubles disk I/O during the staging copy (snapshot bytes are read once to write into staging, then `os.replace` moves the dirent rather than copying bytes) and uses extra disk space ≈ size-of-snapshot during the operation; in exchange the "finish all writes before any unlink" guarantee eliminates the half-restored / half-unlinked window. **Fix (option 2, deferred):** introduce a per-snapshot transaction marker. Write a `_in_progress` file before any restore step; the snapshot API on next startup reads `_in_progress` markers and surfaces a `SnapshotRestoreInterruptedError`. Out of scope for FP27 (more invasive); see "Deliberately not in scope" for the post-v1 entry — and add the corresponding ROADMAP `📋` line if not present.
- **B3 — `downloads.download` chunk-list buffering.** `src/mame_curator/downloads.py:80-170`. The function signature already takes a required `dest: Path` and writes via `atomic_write_bytes(dest, body)` at line 169; **no new parameter is needed**. The defect is at `downloads.py:135-148`: the streaming `aiter_bytes(chunk_size=64*1024)` loop accumulates each chunk into `chunks: list[bytes]`, then joins via `body = b"".join(chunks)`, then `atomic_write_bytes(dest, body)`. Peak in-RAM working set at the `join` call is ~2× the body size (chunks list still live while `b"".join` builds the joined buffer). For a 50 MB DAT, that's ~100 MB resident. **Fix:** rewrite the streaming loop to write chunks straight into a `.tmp` sibling of `dest`. Sketch: replace `chunks: list[bytes] = []` + `chunks.append(chunk)` + `body = b"".join(chunks)` + `atomic_write_bytes(dest, body)` with: open `tmp = dest.parent / (dest.name + ".tmp")` for binary write at request entry (use `tmp_handle = open(tmp, "wb")`, not the `with` form, so close-order is explicit and survives the Windows unlink-while-open hazard); in the loop, `tmp_handle.write(chunk)` and `hasher.update(chunk)` (incremental sha256) and `total += len(chunk)`; on cap abort or sha256 mismatch, **close the handle first, then unlink** (`tmp_handle.close(); tmp.unlink(missing_ok=True)`) — the close-before-unlink ordering mirrors `_atomic.py:60-71` and is mandatory on Windows where an open handle blocks unlink; on size-cap pre-check via `Content-Length`, never open `.tmp`; on success, `tmp_handle.flush(); os.fsync(tmp_handle.fileno()); tmp_handle.close(); os.replace(tmp, dest); fsync_parent_dir(dest)`. **Import additions:** replace the existing import line at `downloads.py:31` (`from mame_curator._atomic import atomic_write_bytes`) with `from mame_curator._atomic import fsync_parent_dir` (grep-confirmed `fsync_parent_dir` exported at `_atomic.py:74`). The legacy `atomic_write_bytes` call disappears post-fix — drop the import to keep `ruff check` clean (no F401 silencing). Post-fix peak RAM ≈ `chunk_size` + httpx async-machinery overhead, independent of body size. **Sha256 verification** (currently `actual = hashlib.sha256(body).hexdigest()` at line 162): move to the incremental `hasher` running through the chunks; compare `hasher.hexdigest()` to `sha256` parameter post-loop, before the `os.replace` — on mismatch, unlink the `.tmp` (close-first) and fall through to the next mirror (same shape as today, but without the full-body buffer). **No public-API change** — `dest` is still required; return type remains `Path | ManualFallback`; callers (just `updates/ini.py:77`) need no edits.
- **B4 — `media/cache.py:fetch_with_cache` size cap + scheme check + streaming.** `client.get(url)` buffers the full response; `resp.content` puts the body in RAM. Three gaps: (a) no max-bytes cap (a malicious or misconfigured upstream can OOM the server); (b) no scheme check (a `file://` URL would be processed by httpx and might leak FS-path semantics — note `downloads.py` already has `_ALLOWED_URL_SCHEMES` at line 40, but `media/cache.py` does not); (c) full-RAM resident even though the eventual sink is on-disk. **Fix:** add a `max_bytes: int = 16 * 1024 * 1024` parameter (default 16 MiB — chosen as ~30× the typical observed body size of ≈ 500 KiB for boxart/screenshots, leaving headroom for fanart up to a few MiB while still capping the OOM ceiling). Add a scheme check at the top: `if urlparse(url).scheme not in {"http", "https"}: raise MediaFetchError(f"unsupported scheme: {urlparse(url).scheme!r}")`. Switch to `client.stream("GET", url)` + `aiter_bytes(64 * 1024)` and write directly to the `.tmp` sibling; abort with `MediaFetchError` if the accumulated total exceeds `max_bytes` (drop the `.tmp` on abort). **Mirror the streaming-cap pattern shape** from `downloads.py:138-148` for the log-line prefix only (`f"BodyTooLarge: {url}: streamed {total} bytes exceeds cap {max_bytes}"`); control flow differs (`downloads.py` sets `last_error` and `break`s to the next mirror — `media/cache.py` has no mirror list, so the size-cap path here `raise`s `MediaFetchError(...)` directly after closing + unlinking the `.tmp`). Keeping the log prefix consistent helps future log-grep audits without forcing the same control flow.
- **B5 — `GET /api/activity` streaming.** `log_path.read_text(encoding="utf-8").splitlines()` reads the whole JSONL into memory. The activity log is append-only and grows: at the design's "30-day retention with ~50 events/day" rate, 30 days is ~1500 events ≈ 200 KB *as a file*; at "user with heavy copy traffic over a year", the file can reach tens of MB. The route's `page` + `page_size` parameters apply *after* the full file is parsed (see `api/routes/activity.py:32-58`), so the per-request RAM cost scales with the whole file regardless of pagination. The current route also computes `total = len(items)` (line 56) and reverses to newest-first. **Fix:** stream the file line-by-line via `open(log_path, "r", encoding="utf-8")` (sync open OK for a route that already does sync I/O via Pydantic + sync FastAPI deps). Maintain two pieces of streaming state: (a) a running `total` integer incremented on every matching line (preserves the response's `total` field semantic); (b) `deque(maxlen=page * page_size)` of matching dicts appended in file order (chronological — oldest to newest). After iteration, the deque holds the newest `page * page_size` matches in chronological order. **Slice formula:** build the newest-first list once with `newest_first = list(reversed(deque))`, then take `items = newest_first[(page-1) * page_size : page * page_size]`. This is the explicit form of "page N of the newest-first ordering" — page 1 takes `newest_first[0:page_size]`, page 2 takes `[page_size:2*page_size]`, etc. Post-fix per-request RAM: ≈ `page * page_size × event_size` for the deque (e.g. 5 × 20 × ~130 B ≈ 13 KB for page 5), independent of total log size. Iteration cost: O(file size) per request — one full pass; memory grows linearly with `page × page_size`, never with total log size. **Counter-cases:** if the route ever moves to async, switch to `aiofiles` then; not in scope here. If file-size grows past 100 MB regularly, a SQLite-backed activity log is post-v1 work.
