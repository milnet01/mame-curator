## How to read this roadmap

This roadmap is **strictly sequential**. Each phase has explicit pre-conditions, an ordered step list, acceptance criteria, and an explicit "do not do" list. The phases were chosen so that **every phase produces something runnable and testable end-to-end** — never a phase that's only useful once the next phase lands.

### Anti-jump rules

These rules exist because Claude Code sessions (and developers in general) tend to skip ahead, leave half-finished work, or pull future-phase concerns into the current phase. They are non-negotiable:

1. **Do not start phase N+1 until every acceptance-criteria checkbox in phase N is ticked and verified.**
2. **Do not write code in the current phase that imports or references modules belonging to a later phase.** If you need it, you mis-phased — stop and revise the roadmap.
3. **Do not "stub out" a later phase's logic to make the current phase appear complete.** If the current phase's acceptance requires real behavior from something that doesn't exist yet, the phasing is wrong.
4. **Do not skip the spec.md / tests-first sequence within a phase.** Even if the implementation feels obvious. The spec is the audit surface — without it, the feature isn't done.
5. **Every phase ends with a commit message stating which phase completed and which acceptance items were verified.** No silent phase boundaries.
6. **If you discover a missing piece mid-phase, finish the phase as scoped and add the missing piece to the appropriate phase explicitly.** Do not enlarge the current phase's scope.

### How a phase is structured

Every phase below contains:

- **Goal** — one-sentence outcome.
- **Pre-conditions** — what must be true before starting (verifiable).
- **Spec to write first** — path to the markdown spec for this feature, written before any implementation.
- **Tests to write first (TDD)** — concrete test names that should fail before the implementation exists.
- **Implementation steps** — strictly ordered. Each step ends with "tests for step N pass."
- **Acceptance criteria** — binary checkboxes. Phase isn't done until all are ticked.
- **Out of scope for this phase** — explicit list of things that belong to later phases.
- **Commit & gate** — exact commit message format and what to ask the user before moving on.

## Phase 2 — Filter rule chain (`filter/`)

### Goal
Given parsed data from Phase 1, produce a deterministic curated set: a list of winner short-names, with each non-winner annotated by reason for rejection.

### Pre-conditions
- Phase 1 acceptance criteria all ticked.
- `Machine`, the parsers, and the CLI exist.

### Important Phase-1 finding (must read before designing Phase 2)

The Pleasuredome DATs strip `cloneof` and `romof` attributes (verified: `grep -c cloneof= ...` returns `0` on both 0.284 DATs). The `Machine` records produced by Phase 1 therefore have `cloneof=None` for **every** machine. Phase 2's parent/clone grouping cannot rely on `Machine.cloneof` alone.

**Solution:** Phase 2 also loads the official MAME `-listxml` (already needed for CHD detection in Phase 1's `parse_listxml_disks`). The `-listxml` is the source of truth for parent/clone relationships. Phase 2 builds a `cloneof_map: dict[str, str]` from the `-listxml` (one helper inside `filter/`, ~30 lines) and joins it onto the Pleasuredome machines by short name.

This is a small adjustment to the original design and does not change the spec contract for Phase 2 — it just clarifies where the input data comes from. The `filter/spec.md` for Phase 2 must call this out explicitly so reviewers know.

### Spec to write first
`src/mame_curator/filter/spec.md` — full Phase A drop list (including genre/publisher/developer/year filters), Phase B tiebreaker chain (including preferred boosts), Phase C overrides, **Phase D session focus** for continuation mode. All numbered exactly as in the design spec §6.2. Includes exact category match patterns, region detection regex, revision-letter detection regex.

The spec must explicitly document that **parent/clone relationships are sourced from the official MAME `-listxml`, not from the Pleasuredome DAT**, and how the join is performed.

The spec must also pin the **YAML schemas** for `overrides.yaml` and `sessions.yaml` (today they exist only as examples in the design spec). Required deliverables:

- `overrides.yaml`: a Pydantic model + JSON Schema for the file. State explicitly whether values are constrained to `(parent → single winner short name)` or whether richer expressions (multi-winner, "drop entire group") are supported. The simplest correct answer is acceptable; document the choice.
- `sessions.yaml`: a Pydantic model + JSON Schema covering `active`, `sessions[name].include_genres`, `include_publishers`, `include_developers`, `include_year_range`, plus any `exclude_*` counterparts the implementation needs. Glob/prefix syntax (e.g. `Shooter*`) must be defined precisely (fnmatch? regex? prefix only?).

Without these schemas pinned, Phase 4's API can't validate POST bodies and the wizard's filter customization step has no contract.

### Tests to write first (TDD)
- **Phase A drops:** one test per drop rule (BIOS, device, mechanical, each category prefix, language-Japanese-only, preliminary driver, CHD-required, genre allow/deny, publisher allow/deny, developer allow/deny, year-before, year-after).
- **Phase B picks:** synthetic groups exercising each tiebreaker including the preferred-genre / preferred-publisher / preferred-developer boost.
- **Phase C overrides:** override beats rule chain; unknown override warns but doesn't crash.
- **Phase D session focus:** active session correctly slices visible set; null session returns full library; session with empty includes is invalid.
- **Determinism property test (`hypothesis`):** same input → same output across 100 random runs.
- **Idempotency:** applying the filter twice yields the same set.
- **Region detection unit tests:** `"(World)"`, `"(USA, Set 2)"`, `"(Europe v2.1)"`, `"(Japan, prototype)"` — each parses to the expected region tag.
- **Revision detection unit tests:** `"Foo (rev A)"`, `"Foo (Set 2)"`, `"Foo v1.5"` — each parses to a comparable version key.
- **Snapshot test:** hand-picked 30-machine fixture DAT → expected curated JSON checked into `tests/snapshots/filter_smoke.json`.
- **Explanation test:** `explain_pick(winner)` returns the chain of tiebreakers that selected this winner.

### Implementation steps (strictly ordered)

1. Write `filter/spec.md`. Explicitly document the `-listxml`-as-cloneof-source decision.
2. Write all listed tests; confirm they fail.
3. Implement `parse_listxml_cloneof(path: Path) -> dict[str, str]` in `src/mame_curator/parser/listxml.py` (extends the existing module). Returns `{clone_short_name: parent_short_name}`. Mirror the lxml.iterparse + `# nosec B410` pattern from `parse_listxml_disks`. Add tests using a fixture that has both parent and clone entries.
4. Implement `region_of(description) -> Region` (enum) and `revision_key_of(description) -> tuple` as pure functions in `filter/heuristics.py`.
5. Implement Phase A as a list of small predicates, each `(Machine, Config) -> bool` returning True if the machine should be dropped. Order matches the spec.
6. Implement Phase B as a list of comparators, each `(Machine, Machine, Config) -> int` returning <0, 0, >0. Compose with `functools.cmp_to_key`. Phase B's input includes the cloneof map — `parent of x = cloneof_map.get(x, x)`.
7. Implement Phase C as an internal phase of `run_filter`: iterate `overrides.entries`, replace per-group winners in-place, append non-fatal advisories (unknown parent / target outside its parent's group / target machine not in DAT) to `FilterResult.warnings`. **Not** a public callable; the per-module `filter/spec.md` is the audit surface.
8. Implement `run_filter(machines, ctx, cfg, overrides, sessions) -> FilterResult` orchestrator. `ctx: FilterContext` carries the cloneof map, catver, languages, bestgames, mature, and chd_required reference data; `sessions: Sessions` carries the session slice. `FilterResult` is a frozen Pydantic model with `winners: tuple[str, ...]` (post-session-slice, alphabetically sorted), `dropped: dict[str, DroppedReason]`, `contested_groups: tuple[ContestedGroup, ...]`, `warnings: tuple[str, ...]`.
9. Add `mame-curator filter` CLI subcommand. Outputs `report.json` and prints summary. Takes `--listxml <path>` flag for the cloneof source.
10. Run against user's real DAT. Verify ~2,000-3,000 winners, sane drops summary.

### Acceptance criteria
- [x] All listed tests pass. *(Tier 1 fixes from indie-review pass 3 closed: 158 tests pass, 96%+ coverage.)*
- [x] Coverage on `filter/` ≥ 95%.

- [x] No public function exceeds 50 lines.

## Phase 3 — Copy, BIOS resolution, and `.lpl` writer (`copy/`)

### Goal
Given a winner list from Phase 2, copy every winner's `.zip` plus all transitively-required BIOS `.zip`s to the destination, write a RetroArch `mame.lpl` playlist, and emit a copy report.

### Pre-conditions
- Phase 2 acceptance criteria all ticked.

### Spec to write first
`src/mame_curator/copy/spec.md` — exact behavior of BIOS resolution, dedup, atomic copy, `.lpl` format, **playlist conflict resolution (append vs overwrite, per-game version replace, recycle-bin)**, **pause/resume/cancel semantics**, **activity-log append**, report contents, error handling.

The spec must pin two file schemas that the design spec mentions only by example:

- `data/activity.jsonl` — one JSON object per line. Required fields: `timestamp` (ISO-8601 UTC), `event_type` (enum: `copy_started`, `copy_finished`, `copy_aborted`, `override_set`, `session_activated`, `ini_refreshed`, `app_updated`, `file_recycled`, `recycle_purged`), `summary` (one-line string), and an event-type-specific `details` object. The full enum + per-type `details` shape is the spec's responsibility.
- `CopyReport` (returned from `run_copy`; in-memory only — persistence to `data/copy-history/<id>/report.json` is Phase 4 scope, the API owns session-history endpoints) — Pydantic model fields, error shapes, and the relationship between `succeeded` / `skipped` / `failed` / `overwritten` / `recycled` lists.

Phase 4's `/api/activity` and `/api/copy/history/{id}/report` cannot be validated without these schemas pinned here.

### Tests to write first (TDD)
- `test_resolve_bios_chain_simple` — one parent → one bios-set; resolves to one bios file.
- `test_resolve_bios_chain_transitive` — A romof B romof C; resolves to {B, C}.
- `test_bios_dedup_across_winners` — 100 Neo-Geo winners → `neogeo.zip` copied once.
- `test_copy_dry_run_no_writes` — dry-run leaves dest dir untouched.
- `test_copy_handles_missing_source` — game listed as winner but `.zip` absent on disk → recorded in report, doesn't crash.
- `test_copy_atomic` — interrupted copy leaves no partial files (writes to `.tmp` first).
- `test_copy_idempotent` — re-running with no changes is a no-op (already-copied with matching size/mtime are skipped).
- `test_lpl_format_matches_retroarch_spec` — output `.lpl` parses as JSON, has expected fields per entry.
- `test_lpl_special_chars_in_description` — descriptions with `&`, `:`, `'` survive correctly into the playlist label.
- `test_copy_report_completeness` — every winner appears in `succeeded`, `skipped`, or `failed`.
- `test_progress_callback_emits_per_file` — N winners → N progress events.
- **`test_existing_playlist_append_no_conflict`** — append mode adds new games without disturbing existing entries.
- **`test_existing_playlist_append_with_same_version`** — same-version conflict is reported but doesn't prompt the user.
- **`test_existing_playlist_append_with_different_version`** — different-version conflict surfaces a per-game decision; "keep existing" leaves files untouched; "replace" copies new + replaces playlist entry; "replace + delete" moves old to recycle.
- **`test_existing_playlist_overwrite`** — overwrite discards entire playlist; with delete-files flag, all destination zips moved to recycle.
- **`test_recycle_retention`** — files moved to `data/recycle/` carry their timestamp; cleanup function removes entries older than 30 days.
- **`test_pause_holds_at_file_boundary`** — pause issued mid-copy waits for current file to finish, then halts; resume continues from the next file.
- **`test_cancel_with_keep_partial`** — cancel keeps already-copied files; cancel with `remove=True` moves them to recycle.
- **`test_activity_log_append`** — every copy session appends one line to `data/activity.jsonl` with action counts.

### Implementation steps (strictly ordered)

1. Write `copy/spec.md`.
2. Write all listed tests; confirm they fail.
3. Implement `resolve_bios_dependencies(winners, machines) -> set[str]` — walks `romof` and `<biosset>` chains.
4. Implement `preflight(plan, source_dir, dest_dir) -> PreflightResult` — checks existence, writability, free space.
5. Implement `copy_one(src, dst) -> CopyOutcome` with `.tmp` + `os.replace` atomicity, mtime preserved via `shutil.copy2`.
6. Implement `write_lpl(playlist_path, entries)` — RetroArch v6+ JSON format.
7. Implement `run_copy(plan, callbacks=...) -> CopyReport` orchestrator. Emits progress via callback per file.
8. Add `mame-curator copy --dry-run` and `mame-curator copy --apply` CLI subcommands.
9. Run dry-run against user's real data. Verify dest layout, BIOS list, total bytes.
10. Run apply against a small subset (10 hand-picked games) into a temp dir. Verify in RetroArch that they show pretty names and launch.

- [ ] Small-subset apply works; RetroArch shows pretty names and launches games.

## Phase 4 — HTTP API (`api/`)

### Goal
A FastAPI server that exposes Phase 1-3 functionality over HTTP, including SSE for live copy progress.

### Pre-conditions
- Phase 3 acceptance criteria all ticked.

### Spec to write first
`src/mame_curator/api/spec.md` — every route, every request/response Pydantic model, every error shape.

### Tests to write first (TDD)
- One TestClient test per route shape (status + payload) for every route in design §6.5.
- `test_sse_copy_progress_streams_events` — `httpx`-driven SSE consumer reads N events for N files.
- `test_overrides_post_persists_to_yaml` — POST then re-read shows the override.
- `test_sessions_crud` — create / read / activate / delete a session round-trips.
- `test_notes_round_trip` — PUT + GET notes for a game.
- `test_explanation_returns_chain` — `/api/games/{name}/explanation` lists the tiebreaker rules that selected this winner.
- `test_stats_aggregations` — `/api/stats` returns sane counts on a fixture library.
- `test_config_patch_validates` — invalid filter config returns 422.
- `test_config_snapshots_round_trip` — patch creates snapshot; restore returns previous state.
- `test_config_export_import_round_trip` — exported JSON imports cleanly into a fresh install.
- `test_pause_resume_abort_copy` — pause holds, resume continues, abort halts; each transitions correctly.
- `test_activity_log_paginated` — `/api/activity` is paginated and filterable.
- **`test_fs_list_sandboxed`** — `/api/fs/list` rejects `../../etc/passwd` and other escape attempts; allows configured paths.
- **`test_fs_roots_per_platform`** — Linux returns `/`, Windows returns drive letters.
- **`test_help_index_renders`** — `/api/help/index` returns table of contents from `docs/help/`.
- `test_paths_in_responses_are_safe` — no path-traversal leaks.

### Implementation steps (strictly ordered)

1. Write `api/spec.md`.
2. Write tests; confirm they fail.
3. Define Pydantic request/response models in `api/schemas.py`.
4. Implement `app = FastAPI(...)` with lifespan handler that loads parsed data once on startup.
5. Implement read-only routes first: `GET /api/games`, `GET /api/games/{name}`, `GET /api/games/{name}/alternatives`, `GET /api/games/{name}/explanation`, `GET /api/config`, `GET /api/stats`.
6. Implement notes routes: `GET/PUT /api/games/{name}/notes`.
